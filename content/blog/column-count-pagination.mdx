## 한 줄이면 끝날 줄 알았다

문제지 편집기를 만들고 있었다. 문제들을 A4 용지에 2단으로 깔아야 했다. `column-count: 2` 한 줄이면 되겠지 싶었다.

```css
.sheet {
  column-count: 2;
  column-gap: 24px;
}
```

브라우저가 알아서 두 컬럼에 나눠 담아줬다. 잠깐은 좋았다. 그런데 긴 지문 하나가 1열 맨 아래에서 시작해 2열 맨 위로 **반토막 나서 이어졌다.** 문장이 컬럼 경계에서 그냥 잘렸다.

웹페이지였다면 자연스러운 동작이다. 신문도 그렇게 흐른다. 하지만 이건 인쇄돼서 시험을 보는 학생한테 가는 **문제지**다. 1번 문제가 1페이지 1열과 2열에 쪼개져 있으면, 그건 불량품이다.

## CSS로 할 수 있는 건 다 해봤다

가장 먼저 떠오른 건 `break-inside`다.

```css
.problem { break-inside: avoid; }
```

문제 카드가 작을 땐 잘 됐다. 컬럼을 안 넘어가게 잡아준다. 문제는 **한 문제가 컬럼 높이보다 길 때**다. 지문 + 보기 5개가 한 컬럼을 꽉 채우고도 남으면, `break-inside: avoid`는 "그럼 이 블록 통째로 다음 컬럼/페이지로 밀어"라고 동작한다. 그 결과 앞 컬럼이 텅 비고, 뒤 컬럼에 거대한 블록이 혼자 들어간다. 조판이 망가진다.

`orphans`, `widows`도 만져봤다. 이것들은 텍스트 줄 단위 제어라, 통짜 블록 분할에는 무력했다.

핵심을 정리하면 이렇다.

> CSS 다단에는 "이 블록을 컬럼 높이에 맞춰 **여기서 끊고, 다음 컬럼에 이어 붙여라**"를 지시하는 표준이 없다.

브라우저가 알아서 자르긴 하는데, 그 "알아서"를 내가 통제할 수가 없었다. 출판물에선 이게 치명적이다.

## 통제권을 가져오기로 했다

그래서 방향을 틀었다. 브라우저의 자동 분배를 포기하고, **내가 직접 측정해서 어디에 무엇을 놓을지 정하자.**

흐름은 이렇다.

1. 화면 밖(`position: absolute; left: -9999px` 같은 오프스크린)에 콘텐츠를 **1열로 전부 렌더**한다.
2. 각 아이템의 높이를 `getBoundingClientRect()`로 잰다.
3. 컬럼 높이를 누적하다가, 다음 아이템을 넣으면 페이지 높이를 넘는 경우 → 다음 컬럼으로. 컬럼도 꽉 찼으면 → 다음 페이지로.
4. 한 아이템이 **컬럼 하나보다 클 때**가 진짜 문제다. 이건 노드를 복제해 두 조각으로 나눈다.

분할이 까다로웠다. 두 번째 조각은 원본을 그대로 둔 채 `overflow: hidden` 컨테이너에 넣고, `transform: translateY(-앞부분높이)`로 윗부분을 끌어올려 가린다. 그러면 "잘린 게 아니라 다음 컬럼으로 이어지는" 것처럼 보인다.

```tsx
// 개념적 의사코드
const overflow = itemHeight - remainingColumnHeight;
if (overflow > 0) {
  // 1번째 조각: 남은 높이만큼만 보이게
  // 2번째 조각: 복제 후 translateY로 끌어올려 이어 붙이기
}
```

## 진짜 시간은 엣지케이스가 먹었다

알고리즘 뼈대는 며칠이면 섰다. 그 다음부터가 길었다.

**첫 페이지만 헤더가 컸다.** 문제지 첫 장에는 제목/안내가 들어가서 콘텐츠 영역 높이가 다르다. 페이지별로 기준 높이를 다르게 잡는 보정이 필요했다.

**잘린 노드의 가로 위치가 어긋났다.** 2단 레이아웃에서 두 번째 조각이 2열로 넘어가면, 세로(`translateY`)는 맞췄는데 가로가 미묘하게 틀어졌다. 원인 찾는 데 한참 걸렸고, 결국 `translateX`로 보정했다.

그리고 제일 기억에 남는 버그.

**가끔 마지막 컬럼이 통째로 사라졌다.** 재현이 들쭉날쭉해서 한참 헤맸다. 컬럼 개수를 계산하는 코드를 따라가 보니, 분할 측정에서 이런 값이 나오고 있었다.

```
innerWidth / columnWidth = 2090 / 697 = 2.998...
Math.floor(2.998) → 2   // 컬럼 하나 증발
```

서브픽셀 렌더링 때문에 정수 3이어야 할 값이 2.998로 떨어졌고, `Math.floor`가 그걸 2로 깎아내면서 마지막 컬럼을 통째로 누락시키고 있었다. 고친 건 한 줄이다.

```diff
- Math.floor(innerWidth / columnWidth)
+ Math.round(innerWidth / columnWidth)
```

한 줄짜리 수정인데, 거기 도달하는 데 오래 걸렸다. 측정 기반 레이아웃에서 서브픽셀은 늘 이렇게 뒤통수를 친다.

이런 버그는 눈으로 못 잡는다. 그래서 측정·페이지네이션·노드 분할·크롭을 검증하는 **브라우저 모드 테스트를 45개** 직접 작성했다. 이후 리팩토링할 때마다 이 테스트들이 안전망이 됐다.

## 빠르게 만드는 라운드

처음엔 페이지가 늘어날 때마다 측정을 다시 호출했다. 수백 번씩 레이아웃을 읽으니 느렸다(읽기/쓰기가 섞이면 layout thrashing도 난다). 측정을 한 번에 모아 **batch로 읽고 결과를 캐시**하는 구조로 바꾸니 확 빨라졌다. [확인: 실제 절감 수치]

이 성능 라운드는 솔직히 AI를 페어로 끼고 했다. 처음 발상과 v1 구현은 내가 했고, 반복적인 측정·튜닝은 도구를 적극적으로 썼다. 문제를 정의하고 결과를 검증하는 일과, 그 안에서 도구를 쓰는 일은 다르다고 생각한다.

## 사내 컴포넌트에서 npm 패키지로

처음엔 한 프로젝트 안의 컴포넌트였다. v1을 만들고, 다른 프로젝트에서 쓰며 다듬고, 결국 **처음부터 다시 설계해 독립 패키지로 분리했다.**

패키지로 만들면서 API를 고민했다.

```tsx
<ColumnPager columnCount={2} pageHeight={1123} showDividers
  onPagesGenerated={(pages, html) => {/* 서버로 보내 PDF로 */}}>
  <ColumnPager.PageBreak />
  {items.map((it) => <Problem key={it.id} {...it} />)}
</ColumnPager>
```

흐름 제어는 컴파운드 컨트롤(`PageBreak` / `ColumnBreak` / `SectionMark`)로 children 사이에 끼우게 했고, 렌더 결과 HTML은 `onPagesGenerated`로 방출해 소비자가 PDF로 변환하든 뭘 하든 알아서 하게 열어뒀다.
