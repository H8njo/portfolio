처음엔 블랙홀이면 빛을 진짜 휘게 만들어야 하는 줄 알았다.

null geodesic을 ray-marching으로 적분하는 글을 몇 개 읽다가 멈췄다. 이건 논문 재현이 목적이 아니라, GLSL 셰이더랑 WebGL 파이프라인을 내 손으로 한 번 처음부터 끝까지 짜보려고 시작한 사이드 프로젝트다. 물리적으로 정확한 측지선이 아니라, **화면 좌표에서 그럴듯하게 빨려드는 룩(look)**이면 충분했다. 그래서 방향을 틀었다. 거리에 따라 배경 UV를 블랙홀 중심 주위로 회전시키면, 빛이 휘는 것처럼 보인다.

## 트레이드오프 결정

별 배경을 어떻게 그릴지가 첫 갈림길이었다.

정석대로면 별도 WebGL로 그려서 한 패스에 합성하거나 FBO를 쓰는 게 맞다. 그런데 나는 별 수천 개의 크기 분포, 색 온도, 큰 별만 후광(radial gradient) 입히기 같은 걸 빠르게 만지고 싶었고, 그건 Canvas 2D의 API가 압도적으로 편했다. 그래서 **별판은 Canvas 2D로 그리고, 그 캔버스를 매 프레임 통째로 WebGL 텍스처로 업로드**하는 2-레이어 구조를 택했다.

`texImage2D(..., HTMLCanvasElement)`는 매 프레임 CPU→GPU로 픽셀을 통째로 밀어넣는 무거운 경로다. 이걸 모르고 고른 게 아니라, 알고도 골랐다. 데모 해상도에서 60fps가 무난하게 나오는 선까지는 편한 개발 속도를 사는 게 맞다고 봤다. 진짜 프로덕션이라면 별판도 GPU로 올려야 한다 — 그건 문서에 트레이드오프로 적어뒀다.

## 접근

셰이더 한 장(`blackholeShaders.ts`) 안에서 전체 파이프라인을 직접 짰다.

- **종횡비 보정** — `gl_FragCoord`를 UV로 바꿀 때 화면비와 이미지비가 다르면 별이 늘어난다. 중심 기준으로 한쪽 축을 스케일해 보정하고, 왜곡 중심(마우스) UV에도 똑같은 보정을 먹였다. 안 그러면 왜곡 중심이 배경과 따로 논다.
- **거리 → 당김(pull)** — `pull = u_mass / (dist² + 0.001)` 역제곱. `+0.001` epsilon이 없으면 중심(dist=0)에서 0으로 나눠 무한대로 폭발한다.
- **회전 왜곡** — `pull`을 회전각으로 써서 UV를 중심 주위로 돌린다. 여기서 의도적으로 "틀린" 행렬을 썼다. 표준 회전이면 `sin`/`cos`가 같은 각을 써야 하는데, `cos(angle*PI)`와 `sin(angle*0.5)`로 다른 각을 먹였다. 순수 회전이 아니라 전단(shear)이 섞이면서 더 "빨려드는" 느낌이 났다. 물리 정합성보다 룩을 택한 결정이다.
- **radius falloff** — 초기엔 화면 어디서나 미세하게 휘었다. `u_radius`를 화면 폭으로 정규화하고, 그 밖이면 왜곡 0, 안에서는 `1.0 - smoothstep(0, R, dist)`로 감쇠시켰다. 단순히 `dist > radius`로 자르면 경계에 딱딱한 원형 자국이 남는데, smoothstep의 Hermite 보간이 경계를 녹여준다.

여기에 질량 ease-in(목표 질량으로 지수 보간해 등장 시 서서히 커짐)과 별판 가로 스크롤을 얹었다. 스크롤은 CPU(별판)와 GPU(셰이더 UV)가 각자 도니까, 속도가 어긋나면 왜곡 영역과 배경이 미끄러진다. `u_cameraSpeed` 유니폼 하나로 셰이더 쪽 속도를 별판에 묶어 맞췄다.

## 결과

데모 해상도에서 `requestAnimationFrame` 60fps로 도는, 클릭하면 블랙홀이 그쪽으로 천천히(lerp 2%) 따라오는 인터랙티브 씬이 나왔다. 라이브러리 없이 raw WebGL 1.0으로 shader compile → program link → buffer → uniform → texture를 전부 손으로 다뤘고, React 19 커스텀 훅(`useBlackHole`)으로 WebGL 생명주기를 캡슐화했다. 저장소는 `H8njo/webgl-black-hole`.

가장 오래 잡아먹은 건 셰이더 수식이 아니라 **두 캔버스의 해상도가 안 맞는 버그**였다.

레티나(`devicePixelRatio > 1`) 디스플레이에서 블랙홀 왜곡 중심과 별판이 어긋나고 스케일이 안 맞았다. 한참 들여다보니 두 레이어가 해상도 정의를 다르게 잡고 있었다. 별판(Galaxy)은 `canvas.width = rect.width * dpr` + `ctx.scale(dpr, dpr)`로 물리 픽셀을 썼고, WebGL 쪽은 viewport를 `gl.drawingBufferWidth`로 잡았다. 한 캔버스를 다른 캔버스의 텍스처로 쓰는데 두 좌표계가 따로 노니, 텍스처 매핑·마우스 좌표·왜곡 중심이 전부 미세하게 틀어진 거였다.

해결은 의외로 단순했다. **양쪽 다 `getBoundingClientRect()` 기반 논리 해상도로 통일하고 DPR 스케일링을 제거**했다. 선명함은 텍스처 필터를 `NEAREST`로 두는 걸로 챙겼다(보간을 끄니 별이 안 뭉갰다). 교훈은 명확했다 — 여러 캔버스를 합성할 땐 DPR/스케일 규칙을 하나 골라서 모든 레이어에 똑같이 적용해야 한다. 한쪽만 "선명하게 하려고" DPR을 곱하는 순간 정렬이 깨진다.

## 회고

배운 건 두 가지다.

하나, 셰이더는 "정확"보다 "그럴듯"이 먼저인 영역이 있다. 일부러 틀린 회전 행렬이 더 좋은 결과를 냈고, 그걸 코드 옆에 "왜 틀리게 짰는지" 주석으로 남기는 게 중요했다. 안 그러면 나중에 내가 "버그"인 줄 알고 고친다.

둘, 합성에서 좌표계 통일은 협상 대상이 아니다. DPR 불일치 버그를 잡는 데 든 시간이 셰이더 전체 작성 시간만큼 됐다.

다시 만든다면 매 프레임 `texImage2D` 재업로드는 손볼 거다. 별판을 GPU로 옮기거나 FBO로 합성하면 그 무거운 CPU→GPU 전송이 사라진다. 지금은 데모라 60fps가 나오지만, 별을 더 늘리거나 고해상도로 가면 거기가 먼저 막힐 걸 안다. [확인: 실제 프레임 타임 측정값]
