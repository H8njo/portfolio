인터스텔라 가르강튀아에서 빛이 왜곡되는 효과를 보고 "저거 어떻게 만들지" 하다가 시작했다.

처음엔 블랙홀이면 빛을 진짜 휘게 만들어야 하는 줄 알았다. 빛이 어떻게 휘는지를 물리적으로 정확히 계산하는 글을 몇 개 읽었는데, 솔직히 몇 문단 못 가서 멈췄다. 어차피 논문 재현이 목적은 아니었으니까. 그냥 GLSL 셰이더랑 WebGL 파이프라인을 내 손으로 처음부터 끝까지 한 번 짜보고 싶어서 시작한 사이드 프로젝트다. 물리적으로 정확한 빛의 경로가 아니라 **화면 좌표에서 그럴듯하게 빨려드는 룩(look)이면 됐다.** 그래서 방향을 틀었다. 거리에 따라 배경 좌표를 블랙홀 중심 주위로 회전시키면 빛이 휘는 것처럼 보인다.

## 트레이드오프 결정

별 배경을 어떻게 그릴지가 첫 갈림길이었다.

처음엔 그냥 별 사진을 찾아다녔다. 별이 빽빽해서 중력렌즈로 휘었을 때 티가 확 나는, 잘 만든 영상 한 장면 같은 우주 배경이 필요했으니까. 근데 막상 뒤져 보니 칠흑 같은 어둠에 별만 박힌 그림이 거의 없더라. 배경이 환하거나 성운 색이 진해서, 왜곡을 먹이면 별이 휘는 게 아니라 배경 전체가 출렁대 어색했다. 그나마 쓸 만한 걸 하나 골라 붙였는데, 이번엔 그게 정지 사진이라 블랙홀만 혼자 도는 꼴이 더 어색했다. (데모에서 '배경 이미지'를 켜면 그때 그 사진이 나온다.)

그래서 별을 직접 그리기로 했다. 흰 별을 기본으로 파랑·빨강을 섞고, 크기 분포를 손으로 만져 그럴듯한 별판을 만들었다. 그래도 정지면 아쉬워서, 결국 별판을 천천히 흐르게 해 움직임까지 줬다.

정석대로면 별도 WebGL로 그려서 한 패스에 합성하거나 GPU 버퍼를 쓰는 게 맞다. 그런데 별 수천 개의 크기 분포, 색 온도, 큰 별만 후광(radial gradient) 입히기 같은 걸 빠르게 만지고 싶었다. 그건 Canvas 2D API가 압도적으로 편하다. 그래서 **별판은 Canvas 2D로 그리고, 그 캔버스를 매 프레임 통째로 WebGL 텍스처로 업로드**하는 2-레이어 구조로 갔다.

`texImage2D(..., HTMLCanvasElement)`는 매 프레임 CPU→GPU로 픽셀을 통째로 밀어넣는 무거운 경로다. 모르고 고른 건 아니고 알고도 골랐다. 데모 해상도에서 60fps가 무난히 나오는 선까지는 개발 속도를 사는 게 맞다고 봤다. 진짜 프로덕션이면 별판도 GPU로 올려야겠지만, 그건 문서에 트레이드오프로 적어뒀다.

## 접근

셰이더 한 장(`blackholeShaders.ts`) 안에서 전체 파이프라인을 직접 짰다. 핵심은 거리에서 당김을 뽑고, 그 당김을 회전각으로 써서 배경 좌표를 중심 주위로 돌리는 이 몇 줄이다.

```glsl
// 역제곱 당김 — +0.001 epsilon이 없으면 중심(dist=0)에서 폭발한다.
// falloffFactor로 u_radius 밖에서는 왜곡을 0으로 죽인다.
float pull = (u_mass / (dist * dist + 0.001)) * falloffFactor;

// 회전 왜곡 — pull을 각도로 넘긴다. 의도적으로 "틀린" 행렬:
// 순수 회전이면 sin/cos가 같은 각을 써야 하지만,
// cos(angle*PI)와 sin(angle*0.5)로 다른 각을 먹여 전단(shear)을 섞었다.
vec2 r = rotate(mouseUv, uv, pull);
```


- **종횡비 보정** — `gl_FragCoord`를 배경 좌표로 바꿀 때 화면비와 이미지비가 다르면 별이 늘어난다. 중심 기준으로 한쪽 축을 스케일해 보정하고, 왜곡 중심(마우스) 좌표에도 똑같은 보정을 먹였다. 안 그러면 왜곡 중심이 배경과 따로 논다.
- **거리 → 당김(pull)** — `pull = u_mass / (dist² + 0.001)` 역제곱. `+0.001` epsilon이 없으면 중심(dist=0)에서 0으로 나눠 무한대로 폭발한다.
- **회전 왜곡** — `pull`을 회전각으로 써서 배경 좌표를 중심 주위로 돌린다. 여기서 의도적으로 "틀린" 행렬을 썼다. 표준 회전이면 `sin`/`cos`가 같은 각을 써야 하는데, `cos(angle*PI)`와 `sin(angle*0.5)`로 다른 각을 먹였다. 순수 회전이 아니라 비틀림이 섞이면서 더 "빨려드는" 느낌이 났다. 물리 정확성보다 룩을 택한 결정이다.
- **radius falloff** — 초기엔 화면 어디서나 미세하게 휘었다. `u_radius`를 화면 폭으로 정규화하고, 그 밖이면 왜곡 0, 안에서는 `1.0 - smoothstep(0, R, dist)`로 감쇠시켰다. 단순히 `dist > radius`로 자르면 경계에 딱딱한 원형 자국이 남는데, smoothstep으로 경계를 부드럽게 녹여준다.

여기에 질량 ease-in(목표 질량으로 지수 보간해 등장 시 서서히 커짐)과 별판 가로 스크롤을 얹었다. 스크롤은 CPU(별판)랑 GPU(셰이더 UV)가 각자 돈다. 속도가 어긋나면 왜곡 영역이랑 배경이 따로 미끄러진다. (이것도 한 번 보고 나서야 알았다.) `u_cameraSpeed` 유니폼 하나로 셰이더 쪽 속도를 별판에 묶어서 맞췄다.

## 결과

데모 해상도에서 `requestAnimationFrame` 60fps로 돌고, 클릭하면 블랙홀이 그쪽으로 천천히(lerp 2%) 따라오는 인터랙티브 씬이 나왔다. 라이브러리 없이 raw WebGL 1.0으로 shader compile → program link → buffer → uniform → texture를 전부 손으로 다뤘다. WebGL 생명주기는 React 19 커스텀 훅(`useBlackHole`)에 캡슐화했다. 저장소는 `H8njo/webgl-black-hole`.

> 이 글에서 설명하는 클릭 추적·질량 ease-in이 들어간 완전판은 저장소(`H8njo/webgl-black-hole`)에 있다. 포트폴리오 페이지에 임베드된 라이브 렌더는 성능·레이아웃을 위해 중심 고정으로 단순화한 **자립형 재현본**이다.

근데 정작 가장 오래 잡아먹은 건 셰이더 수식이 아니었다. **두 캔버스의 해상도가 안 맞는 버그.**

레티나(`devicePixelRatio > 1`) 디스플레이에서 블랙홀 왜곡 중심이랑 별판이 어긋나고 스케일도 안 맞았다. 한참 헤맸는데, 들여다보니 두 레이어가 해상도 정의를 다르게 잡고 있더라. 별판(Galaxy)은 `canvas.width = rect.width * dpr` + `ctx.scale(dpr, dpr)`로 물리 픽셀을 썼고, WebGL 쪽은 viewport를 `gl.drawingBufferWidth`로 잡았다. 한 캔버스를 다른 캔버스의 텍스처로 쓰는데 두 좌표계가 따로 노니까, 텍스처 매핑·마우스 좌표·왜곡 중심이 전부 미세하게 틀어진 거다. 이렇게 미세하게 틀어진 게 제일 잡기 어렵더라.

해결은 의외로 싱거웠다. **양쪽 다 `getBoundingClientRect()` 기반 논리 해상도로 통일하고 DPR 스케일링을 빼버렸다.** 선명함은 텍스처 필터를 `NEAREST`로 두는 걸로 챙겼고(보간을 끄니 별이 안 뭉갰다), 그걸로 충분했다.

## 회고

셰이더는 "정확"보다 "그럴듯"이 먼저인 영역이 있더라. 일부러 틀리게 짠 회전 행렬이 더 좋은 결과를 냈으니까. 대신 코드 옆에 "왜 틀리게 짰는지" 주석을 박아뒀다. 안 그러면 나중에 내가 보고 "버그네" 하고 고쳐버린다. (실제로 한 번 그럴 뻔했다.)

좌표계 얘기는 좀 허무하다. DPR 불일치 버그 하나 잡는 데 든 시간이 셰이더 전체 짜는 시간이랑 비슷했으니까. 솔직히 두 캔버스 합칠 때 DPR/스케일 규칙은 그냥 처음부터 하나로 통일하고 시작하는 게 맞더라. 한쪽만 "선명하게 하려고" DPR 곱하는 순간 정렬이 어긋난다.

다시 만든다면 매 프레임 `texImage2D` 재업로드는 손볼 거다. 별판을 GPU로 옮기거나 GPU에서 직접 합성하면 그 무거운 CPU→GPU 전송이 사라진다. 지금이야 데모라 60fps가 나오지만, 별을 더 늘리거나 고해상도로 가면 거기가 먼저 막힐 거다. 프레임 타임을 정밀하게 재보진 않았지만, 병목이 셰이더 연산이 아니라 매 프레임의 텍스처 업로드라는 건 구조만 봐도 분명하다. 사이드 프로젝트라 "60fps 나오면 됐다" 선에서 멈췄는데, 그 선을 어디에 그을지 스스로 정하는 것도 결국 엔지니어링이라고 생각한다.
