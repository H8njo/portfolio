평문 비밀번호가 프론트엔드 코드 어딘가에 잠깐이라도 머무는 게 싫었다.

EDR 보안 관제 포털을 만들었다. 보안 분석가가 위협 사건을 티켓으로 받아 악성코드·IP·해시를 분석하고, 그 자리에서 URL이나 방화벽을 차단하는 사내 관제 시스템이다. 화면(Next.js)부터 그 뒤에서 도는 서버(NestJS)까지 혼자 다 짰다.

구조부터 보자. 원래대로면 화면이 사내 보안 시스템(여기선 TOS라고 부른다)의 API를 직접 부르면 된다. 근데 그러면 안 됐다. 그래서 화면과 사내 API 사이에 **내 서버를 한 겹 더 세웠다.** 화면은 이제 사내 API가 아니라 이 중간 서버한테만 말을 걸고, 보안에 민감한 일은 전부 서버 뒤에서 처리된다. 이렇게 프론트 앞에 두는 전용 중간 서버를 흔히 BFF라고 부른다. 이 글은 그 한 겹에 뭘 맡겼는지에 대한 얘기다.

## 문제

화면이 사내 API를 직접 부르면 안 되는 이유는 분명했다. 직접 부르는 순간 인증 토큰을 화면이 들고 있어야 하고, 비밀번호를 화면에서 해시해야 하고, "이 사람이 뭘 볼 수 있는지" 권한 판단까지 화면 코드가 해야 한다. 근데 화면 코드는 사용자가 열어볼 수도, 고칠 수도 있는 자리다. 보안 시스템에선 그게 전부 위조될 수 있는 약점이 된다. 화면이 보내는 건 일단 다 의심해야 하는데, 보안 로직까지 거기 두면 의심할 게 너무 많아진다.

## 트레이드오프 결정

중간 서버를 한 겹 두면 호출이 한 단계 늘고, 같은 도메인 로직을 화면과 서버 양쪽에서 신경 써야 한다. 그래도 뒀다. 그 한 겹이 **보안 경계**가 되기 때문이다. 못 믿을 입력(화면)과 검증된 처리(서버)의 선이 거기서 딱 갈린다. 보안에 민감한 건 셋 다 서버 뒤로 넘겼다.

- **비밀번호는 BFF에서 해시한다.** 프론트는 평문 old/new를 보내고, BFF가 `crypto.SHA256()`을 돌린 뒤 TOS로 넘긴다. 프론트 코드는 해시 함수조차 모른다.
- **사용자 신원은 BFF가 주입한다.** 티켓 생성 시 `userId`를 프론트가 보내는 게 아니라, JWT에서 뽑은 `@User()`로 서버가 끼워 넣는다. 클라이언트가 남의 ID를 사칭할 자리를 없앴다.
- **권한 분기도 서버에서 한다.** 분석가(AIR 계정)는 전체 티켓을, 일반 사용자는 자기 소속 기관 것만 본다. `siteType`으로 쿼리를 서버에서 갈랐다. 화면이 "전체 줘"라고 해도 서버가 거른다.

## 접근

끼운 한 겹은 보안만 한 게 아니라 비용을 자산으로 돌려줬다. 대시보드 하나 그리는 데 내 티켓·전체 티켓·보안등급·에이전트 설치율·차단 현황이 다 필요한데, 이걸 프론트가 10번 따로 부르게 두지 않았다. BFF가 `Promise.all`로 병렬로 모아 한 응답으로 내려준다. 인시던트 상세도 IP·해시·이벤트 로그를 서버에서 합쳐서 준다. 호출 한 단계가 늘어난 대신 줄줄이 나가던 호출을 BFF가 대신 먹어줬다.

파일과 인코딩도 BFF가 흡수했다. 첨부파일 다운로드는 TOS → NestJS → axios → 브라우저 4단계를 거치는데, 각 단계의 파일 전달 방식이 하나라도 어긋나면 파일이 깨져 안 열린다. 한 군데만 봐선 안 잡혀서 제일 짜증났다. 네 단계를 한 줄로 꿰어 스트림으로 흘려보내고 나서야 겨우 열렸다. 업로드는 반대로 한글 파일명이 깨졌는데, multer가 받은 파일명 인코딩이 어긋나서 그런 거라 인코딩을 바로잡아 원래 이름으로 복원했다. 둘 다 프론트는 파일을 그냥 던지기만 한다.

```ts
@Post(':ticketId/attachments/:fileId/download')
async download(@Param('fileId') fileId, @Res({ passthrough: true }) res) {
  const stream = await this.upstream.downloadFile(fileId); // responseType: 'stream'
  res.set({ 'Content-Type': 'application/octet-stream;charset=UTF-8' });
  return new StreamableFile(stream);
}
```

데이터는 한 종류가 아니라 DB를 둘로 쪼갰다. TOS가 내려주는 조회 전용 정형 뷰는 PostgreSQL(TypeORM), 스키마가 계속 바뀌는 차단·알림은 MongoDB(Mongoose)로 보내고, 한 NestJS 서비스에서 `@InjectRepository`와 `@InjectModel`을 같이 주입받아 한 흐름에서 엮었다.

## 결과

프론트는 평문 비밀번호도, TOS 토큰도, 권한 판단도, 파일명 인코딩도 만지지 않는다. 그 책임이 전부 BFF 한 곳에 모였다. 보안 로직이 한 군데 있으니 의심할 표면이 좁아졌고, 고칠 때도 한 곳만 본다. 덤으로 대시보드와 인시던트 상세는 호출 한 번으로 그려지고, 파일은 한글 이름이든 바이너리든 안 깨진다.

가장 만족스러운 건 따로 있다. 같은 기능을 프론트와 백 양쪽에서 짜다 보니 "이 로직이 어느 쪽에 살아야 하는가"를 계속 결정해야 했는데, 그 결정의 기준이 거의 다 보안이었고, 그래서 거의 다 백엔드로 갔다.

## 회고

BFF를 끼우면 같은 도메인을 양쪽에서 신경 써야 한다는 비용이 분명히 있다. 그런데 보안 시스템에선 그 비용이 오히려 자산이었다. 경계가 명확하니 "여긴 못 믿는 입력, 여기서부턴 검증된 값"이 코드 구조로 드러났다.

다시 한다면 refresh 토큰을 서버 메모리 Map이 아니라 영속 저장소에 두고 매칭을 제대로 구현했을 거다. 그때는 "추후 구현" 주석으로 남겨두고 메모리로 갔는데, 서버를 재시작하면 토큰이 날아가는 구조였다. 그 정도 빼면, "민감한 건 전부 서버 뒤로"라는 원칙 하나로 끝까지 끌고 간 건 두고두고 편했다. 의심할 곳이 한 군데뿐이라, 나중에 보안 점검을 할 때도 거기만 보면 됐으니까.
