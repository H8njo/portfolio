평문 비밀번호가 프론트엔드 코드 어딘가에 잠깐이라도 머무는 게 싫었다.

EDR 보안 관제 포털을 여럿이 함께 만들었다. 보안 분석가가 위협 사건을 티켓으로 받아 악성코드·IP·해시를 분석하고, 그 자리에서 URL이나 방화벽을 차단하는 사내 관제 시스템이다. 나는 내가 맡은 기능을 화면(Next.js)부터 그 뒤에서 도는 서버(NestJS)까지 양쪽 다 짰다. 이 글의 BFF도 그렇게 프론트와 백을 오가며 만든 것 중 하나다.

## 쉽게 말하면, 경비실을 하나 세웠다

원래대로면 화면이 사내 보안 시스템(여기선 TOS라고 부른다)의 API를 직접 부르면 된다. 근데 그러면 안 됐다.

방문객이 금고에 직접 손을 못 대게, 그 사이에 **경비실을 하나 세운 셈**이다. 방문객(화면)은 이제 금고(사내 API)가 아니라 경비실에만 말을 건다. 신분을 확인하고, 금고를 여닫는 민감한 일은 전부 경비실이 대신 한다. 이렇게 화면 바로 앞에 세우는 전용 중간 서버를 흔히 **BFF**(Backend for Frontend)라고 부른다. 이 글은 그 경비실 한 겹에 무엇을 맡겼는지에 대한 얘기다.

## 문제 — 화면은 못 믿는 자리다

화면이 사내 API를 직접 부르면 안 되는 이유는 분명했다. 직접 부르는 순간, 원래 서버가 해야 할 일이 화면으로 넘어온다. 인증 토큰을 화면이 들고 있어야 하고, 비밀번호를 화면에서 암호화해야 하고, "이 사람이 뭘 볼 수 있는지" 권한 판단까지 화면 코드가 해야 한다.

문제는 **화면 코드가 사용자가 열어볼 수도, 고칠 수도 있는 자리**라는 거다. 손님이 직접 들고 있는 서류 같은 거라, 위조가 가능하다. 보안 시스템에선 화면이 보내는 건 일단 다 의심해야 하는데, 보안 로직까지 거기 두면 의심할 게 너무 많아진다.

## 그래서 민감한 건 전부 경비실 뒤로

중간 서버를 한 겹 두면 대가가 있다. 호출이 한 단계 늘고, 같은 도메인 로직을 화면과 서버 양쪽에서 신경 써야 한다. 그래도 뒀다. 그 한 겹이 **보안 경계**가 되기 때문이다. 못 믿을 입력(화면)과 검증된 처리(서버)의 선이 거기서 딱 갈린다.

보안에 민감한 것 셋을 전부 서버 뒤로 넘겼다.

**1. 비밀번호는 경비실이 해싱한다.** 프론트는 평문 비밀번호(옛것/새것)를 보내기만 하고, 상위 시스템(TOS) 규약이 요구하는 해싱(SHA256)은 BFF가 처리한 뒤 TOS로 넘긴다. 프론트 코드는 해싱 함수가 뭔지조차 모른다.

**2. "너 누구야"는 경비실이 정한다.** 티켓을 만들 때 사용자 ID를 프론트가 보내는 게 아니라, 로그인 토큰(JWT)에서 서버가 직접 뽑아 끼워 넣는다. 그래서 클라이언트가 남의 ID를 사칭할 자리 자체가 없다.

**3. "뭘 볼 수 있나"도 경비실이 거른다.** 분석가 계정은 전체 티켓을, 일반 사용자는 자기 소속 기관 것만 본다. 화면이 "전체 줘"라고 우겨도, 서버가 계정 종류를 보고 쿼리를 갈라 걸러낸다.

## 경비실이 잡일까지 떠안다

한 겹 끼운 김에, 경비실은 보안만 하지 않고 귀찮은 일을 대신 먹어줬다.

**흩어진 호출을 하나로.** 대시보드 하나 그리는 데 내 티켓·전체 티켓·보안등급·에이전트 설치율·차단 현황이 다 필요하다. 이것만 다섯, 인시던트 상세(IP·해시·이벤트 로그)까지 더하면 열 번 가까운 호출을 — 프론트가 따로따로 부르게 두지 않았다. BFF가 한꺼번에 병렬로(`Promise.all`) 모아 한 응답으로 내려준다. 호출 한 단계가 늘어난 대신, 줄줄이 나가던 호출을 BFF가 대신 삼켰다.

**파일 깨짐도 여기서.** 첨부파일 다운로드는 TOS → 내 서버 → 브라우저까지 여러 단계를 거치는데, 그 사이에서 파일 전달 방식이 한 군데만 어긋나도 파일이 깨져 안 열린다. 한 곳만 봐선 안 잡혀서 제일 짜증났다. 단계를 물 흐르듯 이어지는 하나의 스트림으로 꿰고 나서야 겨우 열렸다. 업로드는 반대로 한글 파일명이 깨졌는데, 파일명이 엉뚱한 인코딩으로 들어와서였다. 인코딩을 바로잡아 원래 이름으로 되돌렸다. 둘 다 프론트는 파일을 그냥 던지기만 한다.

```ts
@Post(':ticketId/attachments/:fileId/download')
async download(@Param('fileId') fileId, @Res({ passthrough: true }) res) {
  const stream = await this.upstream.downloadFile(fileId); // responseType: 'stream'
  res.set({ 'Content-Type': 'application/octet-stream;charset=UTF-8' });
  return new StreamableFile(stream);
}
```

**성격 다른 데이터는 DB도 나눠서.** TOS가 내려주는 "읽기 전용 정형 데이터"는 PostgreSQL에, 스키마가 계속 바뀌는 차단·알림 데이터는 MongoDB에 넣었다. 한 서비스 안에서 두 DB를 같이 주입받아 한 흐름으로 엮었다. 하나의 DB로 다 우겨넣지 않은 거다.

## 결과

프론트는 평문 비밀번호도, 사내 API 토큰도, 권한 판단도, 파일명 인코딩도 만지지 않는다. 그 책임이 전부 BFF 한 곳에 모였다. 보안 로직이 한 군데 있으니 의심할 표면이 좁아졌고, 고칠 때도 한 곳만 본다. 덤으로 대시보드와 인시던트 상세는 호출 한 번으로 그려지고, 파일은 한글 이름이든 바이너리든 안 깨진다.

가장 만족스러운 건 따로 있다. 같은 기능을 프론트와 백 양쪽에서 짜다 보니 "이 로직이 어느 쪽에 살아야 하는가"를 계속 결정해야 했는데, 그 기준이 거의 다 보안이었고, 그래서 거의 다 백엔드로 갔다.

## 회고

BFF를 끼우면 같은 도메인을 양쪽에서 신경 써야 하는 비용이 분명히 있다. 그런데 보안 시스템에선 그 비용이 오히려 자산이었다. 경계가 명확하니 "여긴 못 믿는 입력, 여기서부턴 검증된 값"이 코드 구조로 그냥 드러났다.

다시 한다면 refresh 토큰을 서버 메모리에 잠깐 들고 있게 한 부분은 제대로 고쳤을 거다. 그때는 "추후 구현" 주석으로 남겨두고 메모리로 갔는데, 서버를 재시작하면 토큰이 날아가는 구조였다. 영속 저장소에 두고 대조까지 구현했어야 했다.

그 정도 빼면, "민감한 건 전부 서버 뒤로"라는 원칙 하나로 끝까지 끌고 간 건 두고두고 편했다. 의심할 곳이 한 군데뿐이라, 나중에 보안 점검을 할 때도 거기만 보면 됐으니까.
