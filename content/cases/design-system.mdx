사내 디자인 시스템을 맡았다. 처음엔 컴포넌트만 잘 만들면 되는 줄 알았다.

막상 해보니 일의 절반은 그게 아니었다. 컴포넌트를 잘 만드는 거랑, 다음 사람이 그 위에서 안 헷갈리게 굴러가게 깔아두는 건 다른 일이더라. 여기서 컨벤션 하나 잘못 정하면 그게 소비 프로젝트마다 그대로 복제되니까, 사소한 것도 함부로 못 정했다.

범위는 좁지 않았다. 버튼, 폼 입력, 다이얼로그, 탭, 타이포그래피, Select, 아이콘까지 — 여러 프론트엔드 제품이 공유할 컴포넌트 전반을 Radix primitive + Tailwind + CVA 위에 세우고, Storybook으로 문서화해 사내 패키지로 배포하는 라이브러리였다. 나는 그 **토대와 공통 규칙을 깔고**(스타일 토큰, 유틸, 컴포넌트 스캐폴딩), 패턴의 기준이 되는 컴포넌트와 툴링을 직접 짰다. 그리고 설정상 **모든 PR의 코드오너**라 남이 올린 것까지 전부 리뷰·머지했고, 릴리스도 직접 cut했다. 누가 와도 같은 규율로 굴러가게 PR 템플릿, 리뷰어 로테이션, commit/pr/ship 컨벤션을 박아뒀다.

컴포넌트마다 크고 작은 판단이 있었다. 예를 들어 Button 초기 버전엔 욕심을 부려서, 클릭 함수가 오래 걸리면 로딩 상태를 알아서 켜고 꺼주게 만들었다. 데모로는 그럴듯했는데 기본 컴포넌트로는 그냥 틀린 설계였다 — 상태 주도권을 쓰는 사람한테서 뺏어오니까. 결국 되돌렸다. 기본 컴포넌트에서는 마법보다 예측 가능한 쪽이 거의 항상 맞더라.

그런 결정이 수십 번 있었다. 이 글은 그중 **제일 오래 붙들고 가장 골머리를 앓은 두 개**만 꺼낸 거다. 하나는 Select, 하나는 아이콘 자동화.

## 첫 번째 — Select: 하나만 고르기와 여러 개 고르기를 한 컴포넌트로

쉽게 말하면 이거다. 드롭다운에서 **하나만 고르는 것**(예: 국가 선택)과 **여러 개 고르는 것**(예: 관심 태그 여러 개)은 쓰는 사람 눈엔 똑같은 "선택 상자"인데, 안에서 동작하는 방식이 완전히 다르다. 이걸 같은 컴포넌트로 쓰게 만드는 게 과제였다.

기존 `Select`는 하나만 고르는 부품(Radix Select) 위에 여러 개 고르기를 억지로 흉내 내고 있었다. 그러다 보니 컴포넌트가 정작 **지금 뭐가 선택됐는지를 몰랐다.** 선택된 값 목록도, 체크 표시도, 스크린리더가 읽을 "선택됨" 표시도 없었다. 눈속임이었던 거다.

그래서 억지로 한 몸에 욱여넣는 대신, **겉(쓰는 방법)은 하나로 통일하고 속(엔진)은 둘로 갈랐다.** 하나만 고르는 건 기존 부품을 그대로 감싸서 폼 연동을 공짜로 얻고, 여러 개 고르는 건 다른 부품 위에 선택 목록·키보드 이동·체크박스를 직접 만들었다. 쓰는 사람이 작성하는 코드는 양쪽이 똑같다.

여기에 타입으로 안전장치를 걸었다.

```ts
type SelectV2SingleProps = Common & { multiple?: false; value?: string;  onValueChange?: (v: string)   => void }
type SelectV2MultiProps  = Common & { multiple: true;  value: string[];  onValueChange:  (v: string[]) => void }
export type SelectV2Props = SelectV2SingleProps | SelectV2MultiProps
```

`multiple`을 켜면 컴파일러가 "값은 배열이어야 한다"고 요구하고, 끄면 "값은 문자열"이라고 강제한다. 잘못 쓰면 실행 전에 빨간 줄로 먼저 막힌다.

여기에 하나 더. 드롭다운 너비를 늘 "가장 긴 항목"에 맞추고 싶었는데, 항목은 열리기 전엔 화면에 없어서 크기를 잴 수가 없다. 이걸 JavaScript 측정 없이 **CSS만으로** 풀었다. 안 보이는 복제본을 같은 칸에 겹쳐 두면 칸이 저절로 가장 긴 항목 크기가 되고, 보이는 버튼이 그 너비를 입는다. 리사이즈 관찰이나 리렌더 루프가 아예 없다.

보상은 쓰는 쪽에서 돌아왔다. 교재→단원을 트리로 고르고 탭까지 붙은 복잡한 선택 상자를, 스타일 한 줄 다시 안 짜고 조립할 수 있었다. 선택 상태가 아무리 깊이 중첩된 항목에도 알아서 전달되니까, 탭이나 표 안에 묻힌 항목도 체크박스가 붙고 토글된다. 디자인 시스템을 별도 패키지로 떼어낸 이유가 사실 이거였다.

## 두 번째 — 아이콘 자동화, 그리고 빌드가 두 번 죽은 이야기

이건 이런 일이다. 디자이너가 그린 아이콘 그림(SVG)을 개발자가 **손으로 하나하나 코드로 옮겨 적고** 목록 파일에 등록하던 걸, 디자이너가 **폴더에 그림을 넣기만 하면 코드가 저절로 만들어지게** 바꾼 작업.

원래는 아이콘 하나 추가할 때마다 사람이 손으로 컴포넌트를 만들고 목록에 한 줄씩 붙였다. 160개쯤 되니 이건 작업이 아니라 사고 대기였다.

그래서 뒤집었다. 디자이너가 정해진 폴더에 SVG를 드롭하고 명령어 한 번(`pnpm generate`)을 돌리면, 폴더를 스캔해서 목록을 통째로 다시 만든다. 파일을 지우면 코드에서도 사라진다. **폴더가 곧 정답지**인 셈이다.

그런데 여기서 빌드가 두 번 죽었다.

**첫 번째 죽음 — 개발할 땐 되는데 배포 빌드만 터졌다.** 아이콘을 불러오는 코드가 `'...Alert.svg?react'`처럼 파일 이름 뒤에 `?react`라는 꼬리표를 붙이는데, 개발용 도구(Vite)에선 이 꼬리표를 이해하지만 배포용 도구(Rollup)는 몰랐다. Rollup은 `Alert.svg?react`라는 (디스크에 실제로는 없는) 파일을 찾다가 길을 못 찾고 질식했다. 원인 찾는 데 한참 걸렸다. 소스를 두 벌로 가르는 대신, 그 꼬리표를 떼고 진짜 파일 경로로 바꿔주는 얇은 어댑터 하나를 배포 빌드에 끼웠다.

```js
{
  name: 'svg-query-resolver',
  resolveId(source, importer) {
    if (source.endsWith('?react')) {
      const actualPath = source.replace('?react', '')
      return path.resolve(path.dirname(importer), actualPath)
    }
    return null
  },
}
```

개발 도구와 배포 도구가 다를 땐, 둘 중 하나에 맞춰 소스를 갈라치는 것보다 얇은 어댑터 하나 끼우는 게 나았다.

**두 번째 죽음은 더 음흉했다.** 빌드는 통과하는데 화면에서 아이콘이 잘려 나왔다. 크게 키우면 커지긴커녕 일부가 잘렸다. 범인은 SVG를 자동으로 최적화해 주는 도구(SVGO)였다. 그림의 "좌표계"(viewBox) 정보를 불필요하다고 판단해 지워버렸는데, 이게 없으면 크기를 키울 때 기준이 사라져서 그림이 잘린다. 이것도 한참 째려봤다. 디자이너 쪽에서 그림을 좌표계 기준으로 다시 뽑고, 빌드 설정에도 "그 정보는 지우지 말라"고 못을 박았다. **소스와 빌드 두 층 다 막아야** 입력이 흔들려도 안 터진다.

첫 번째 문제는 타입 검사만으론 안 잡혔다(문법상으론 멀쩡해 보이니까). 그래서 push 전에 타입 검사 + 실제 배포 빌드를 강제로 돌리는 게이트를 걸었다. 같은 부류의 실패는 이제 내 노트북을 못 벗어난다.

## 결과

Select는 여러 개 고르기가 눈속임이 아니라 정식 기능이 됐다. 아이콘은 손으로 쓰던 등록 코드가 통째로 사라져서, 디자이너가 폴더에 그림을 드롭하면 타입까지 붙은 아이콘 컴포넌트가 나오고 Storybook 갤러리에 바로 뜬다.

## 회고

컴포넌트 하나 잘 만드는 거랑, 다음 사람이 잘 만들 수밖에 없게 깔아두는 건 확실히 다른 일이더라.

이 두 문제의 공통점이 있었다. 둘 다 "겉으로는 되는 것처럼 보이는데 속은 틀린" 상태에서 시작했다. Select는 선택된 걸 아는 척했고, 아이콘 빌드는 개발 환경에서만 되는 걸 다 되는 척했다. 눈속임을 진짜로 바꾸는 게 대부분의 시간을 먹었다. 그리고 그 눈속임은 타입 검사나 개발 빌드만으론 안 드러나서, 결국 "실행 전에 실패하게 만드는 게이트"를 걸어야 했다. 디자인 시스템에서 제일 중요한 건 예쁜 컴포넌트가 아니라, 다음 사람이 틀리게 쓸 수 없게 막아두는 안전장치라는 걸 이 두 개에서 제일 크게 배웠다.
