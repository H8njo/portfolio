사내 디자인 시스템을 맡았다. 처음엔 컴포넌트만 잘 만들면 되는 줄 알았다.

막상 해보니 일의 절반은 그게 아니었다. 컴포넌트를 잘 만드는 거랑, 다음 사람이 그 위에서 안 헷갈리게 굴러가게 깔아두는 건 다른 일이더라. 여기서 컨벤션 하나 잘못 정하면 그게 소비 프로젝트마다 그대로 복제되니까, 사소한 것도 함부로 못 정했다.

버튼, 타이포그래피, 폼 입력, 탭 같은 기본 컴포넌트를 만드는 건 크게 막히지 않았다. Radix로 접근성을 깔고, Tailwind와 CVA로 variant를 정리하고, Storybook에 문서를 붙이고 — 대체로 손에 익은 흐름이었다. 리뷰랑 릴리스, 컨벤션 정리를 거들다 보니 라이브러리 전반을 늘 들여다보고 있기도 했다.

그렇다고 매끈하기만 했던 건 아니다. 예를 들어 Button 초기 버전엔 욕심을 부려서, 클릭 함수가 오래 걸리면 로딩 상태를 알아서 켜고 꺼주게 만들었다. 데모로는 그럴듯했는데 기본 컴포넌트로는 그냥 틀린 설계였다 — 상태 주도권을 쓰는 사람한테서 뺏어오니까. 결국 되돌렸다. 기본 컴포넌트에서는 마법보다 예측 가능한 쪽이 거의 항상 맞더라.

그런 작은 판단은 늘 있었지만, **진짜 오래 붙들린 건 두 개**였다. 하나는 Select, 하나는 아이콘 자동화.

## 첫 번째 — Select: 하나만 고르기와 여러 개 고르기를 한 컴포넌트로

쉽게 말하면 이거다. 드롭다운에서 **하나만 고르는 것**(예: 국가 선택)과 **여러 개 고르는 것**(예: 관심 태그 여러 개)은 쓰는 사람 눈엔 똑같은 "선택 상자"인데, 안에서 동작하는 방식이 완전히 다르다. 이걸 같은 컴포넌트로 쓰게 만드는 게 과제였다.

기존 `Select`는 하나만 고르는 부품(Radix Select) 위에 여러 개 고르기를 억지로 흉내 내고 있었다. 그러다 보니 컴포넌트가 정작 **지금 뭐가 선택됐는지를 몰랐다.** 선택된 값 목록도, 체크 표시도, 스크린리더가 읽을 "선택됨" 표시도 없었다. 눈속임이었던 거다.

그래서 억지로 한 몸에 욱여넣는 대신, **겉(쓰는 방법)은 하나로 통일하고 속(엔진)은 둘로 갈랐다.** 하나만 고르는 건 기존 부품을 그대로 감싸서 폼 연동을 공짜로 얻고, 여러 개 고르는 건 다른 부품 위에 선택 목록·키보드 이동·체크박스를 직접 만들었다. 쓰는 사람이 작성하는 코드는 양쪽이 똑같다.

여기에 타입으로 안전장치를 걸었다.

```ts
type SelectV2SingleProps = Common & {
  multiple?: false; value?: string; onValueChange?: (v: string) => void }
type SelectV2MultiProps = Common & {
  multiple: true; value: string[]; onValueChange: (v: string[]) => void }
export type SelectV2Props = SelectV2SingleProps | SelectV2MultiProps
```

`multiple`을 켜면 컴파일러가 "값은 배열이어야 한다"고 요구하고, 끄면 "값은 문자열"이라고 강제한다. 잘못 쓰면 실행 전에 빨간 줄로 먼저 막힌다.

여기에 하나 더. 드롭다운 너비를 늘 "가장 긴 항목"에 맞추고 싶었는데, 항목은 열리기 전엔 화면에 없어서 크기를 잴 수가 없다. 이걸 JavaScript 측정 없이 **CSS만으로** 풀었다. 안 보이는 복제본을 같은 칸에 겹쳐 두면 칸이 저절로 가장 긴 항목 크기가 되고, 보이는 버튼이 그 너비를 입는다. 리사이즈 관찰이나 리렌더 루프가 아예 없다.

보상은 쓰는 쪽에서 돌아왔다. 교재→단원을 트리로 고르고 탭까지 붙은 복잡한 선택 상자를, 스타일 한 줄 다시 안 짜고 조립할 수 있었다. 선택 상태가 아무리 깊이 중첩된 항목에도 알아서 전달되니까, 탭이나 표 안에 묻힌 항목도 체크박스가 붙고 토글된다. 디자인 시스템을 별도 패키지로 떼어낸 이유가 사실 이거였다.

## 두 번째 — 아이콘 자동화: 폴더가 곧 정답지

이건 이런 일이다. 디자이너가 그린 아이콘 그림(SVG)을 개발자가 **손으로 하나하나 코드로 옮겨 적고** 목록 파일에 등록하던 걸, 디자이너가 **폴더에 그림을 넣기만 하면 코드가 저절로 만들어지게** 바꾼 작업.

원래는 아이콘 하나 추가할 때마다 사람이 손으로 컴포넌트를 만들고 목록에 한 줄씩 붙였다. 160개쯤 되니 이건 작업이 아니라 사고 대기였다.

그래서 뒤집었다. 디자이너가 정해진 폴더에 SVG를 드롭하고 명령어 한 번(`pnpm generate`)을 돌리면, 폴더를 스캔해서 목록을 통째로 다시 만든다. 파일을 지우면 코드에서도 사라진다. **폴더가 곧 정답지**인 셈이다.

핵심은 목록 파일을 사람이 손대지 않는다는 거다. `generate`가 폴더를 읽어 export 목록을 통째로 새로 쓴다.

```js
const files = await fs.promises.readdir(iconsDir)        // 폴더의 SVG를 스캔
const exports = files
  .filter((f) => f.endsWith('.svg'))
  .map((file) => {
    const name = file.replace(/\.svg$/, '')              // Alert_1px.svg → Alert_1px
    const pascal = name[0].toUpperCase() + name.slice(1)
    return `export { default as ${pascal} } from '../../assets/icons/design-system/${file}'`
  })
await fs.promises.writeFile(outputFile, exports.join('\n') + '\n')  // 목록을 통째로 덮어씀
```

파일명이 곧 컴포넌트 이름이 되니 따로 매핑표를 둘 필요가 없다(`_1px`·`_fill` 같은 굵기도 이름에서 파생된다). 목록을 매번 새로 쓰기 때문에, SVG를 지우면 export도 같이 사라진다. 손으로 관리하던 목록이 폴더 상태를 그대로 따라오는 거울이 됐다.

여기에 두 가지를 더 챙겼다.

**색을 코드가 아니라 CSS가 정하게.** Figma에서 뽑을 때 아이콘 색을 `currentColor`로 고정했다. 그러면 아이콘이 자기 색을 들고 있는 게 아니라, 놓인 자리의 글자색을 상속한다. 아이콘마다 라이트용·다크용을 따로 만들 필요 없이, 테마가 바뀌면 색도 저절로 따라온다.

**예외를 위한 뒷문.** 스피너처럼 자동 스캔 규칙에 안 맞는 특수 아이콘은 `extra/` 폴더에 손으로 두고, 자동 생성된 목록이 마지막에 그걸 끌어와 합친다. 코드젠이 소유하는 영역과 사람이 손대는 영역을 폴더로 깔끔히 갈랐다.

흐름이 폴더 하나로 닫혔다. 드롭 → 타입 붙은 아이콘 컴포넌트 → Storybook 갤러리에 즉시. 갤러리가 목록을 통째로 순회하도록 짜서, `generate` 한 번이면 스토리를 안 고쳐도 새 아이콘이 바로 뜬다.

## 결과

Select는 여러 개 고르기가 눈속임이 아니라 정식 기능이 됐다. 컴포넌트가 무엇이 선택됐는지를 진짜로 알게 되니, 그 위에서 조립이 열렸다. 교재→단원을 트리로 고르고 탭까지 붙은 복잡한 선택 상자를, 소비 제품이 trigger 스타일 한 줄 다시 안 짜고 만들 수 있었다. `multiple` 키 하나로 값 타입과 콜백이 컴파일 단계에서 강제되니, 잘못 쓴 코드는 애초에 빌드를 못 넘긴다. "쓰기 쉬운데 틀리기 어려운" 컴포넌트가 목표였고, Select가 그 기준선이 됐다.

아이콘 쪽 변화는 개발자보다 디자이너가 먼저 반겼다. 예전엔 "이 아이콘 새로 추가했어요"를 한 땀 한 땀 개발자한테 알려줘야 했다. 이제는 그럴 필요가 없다. 개발자가 Figma에서 전체 아이콘을 통째로 긁어 폴더에 넣고 `generate`만 돌리면 끝이라, 디자이너는 그냥 Figma에서 그리기만 하면 된다. 뭐가 바뀌었는지 목록을 맞춰보는 커뮤니케이션 자체가 사라졌다. 디자이너가 이 흐름을 제일 좋아했다.

## 회고

컴포넌트 하나 잘 만드는 거랑, 다음 사람이 잘 만들 수밖에 없게 깔아두는 건 확실히 다른 일이더라.

이 두 문제의 공통점이 있었다. 시작은 둘 다 "사람이 조심하면 되는" 상태였다. Select는 소비자가 선택 개수를 직접 세서 넘겨야 했고, 아이콘은 누군가 목록에 한 줄씩 손으로 붙이고 그걸 서로 알려줘야 했다. 조심으로 굴러가는 건 결국 누군가 한 번은 틀린다. 그래서 둘 다 "조심"을 "구조"로 옮겼다. Select는 타입이 잘못된 사용을 실행 전에 막게, 아이콘은 폴더가 목록의 유일한 정답이 되게. 디자인 시스템에서 제일 중요한 건 예쁜 컴포넌트가 아니라, 다음 사람이 틀리게 쓸 수 없게 만들어 두는 안전장치라는 걸 이 두 개에서 제일 크게 배웠다.
