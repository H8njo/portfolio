사내 디자인 시스템을 맡았다. 처음엔 컴포넌트만 잘 만들면 되는 줄 알았는데, 실제 일은 여러 사람이 그 위에서 안 헷갈리게 굴러가도록 만드는 거였다.

여러 프론트엔드 제품이 같은 UI·접근성을 공유해야 했다. React 컴포넌트 라이브러리를 Radix primitive + Tailwind + CVA 위에 올리고, Storybook으로 문서화해 사내 패키지로 배포하는 구조다. 나는 초기 인프라를 깔고, 첫 컴포넌트인 Button으로 패턴을 정하고, 아이콘 파이프라인을 자동화했고, 가장 최근엔 Select를 두 계층으로 다시 설계했다. 그러면서 코드오너이자 릴리스 매니저가 됐다.

## 문제

라이브러리는 개별 화면이 아니다. 한 번 잘못 정한 컨벤션이 모든 소비 프로젝트에 복제된다.

구체적인 통증은 세 군데에서 왔다. 첫째, 기존 `Select`가 Radix Select 위에 multi-select를 흉내 내고 있었다. Radix Select는 본질적으로 단일 값이라, multi를 만들려고 `value`를 빈 문자열로 비우고 `shouldKeepOpenRef`로 자동 닫힘을 가로채는 우회를 썼다. 컴포넌트가 정작 무엇이 선택됐는지를 몰랐다. 진짜 선택 집합도, 체크박스도, `aria-checked`도 없었다.

둘째, 아이콘이 수작업이었다. 디자이너가 Figma에서 뽑은 SVG를 누군가 손으로 컴포넌트로 옮기고 barrel에 export를 추가했다. 160개가 되면 그건 일이 아니라 사고다.

셋째, 손으로 만든 컴포넌트는 폴더 구조가 매번 미묘하게 달랐다. 누구는 barrel을 만들고 누구는 안 만들고, stories가 있는 것과 없는 것이 섞였다.

## 트레이드오프 결정

Button 초기 버전에 야심을 부렸던 적이 있다. `isLoading` prop을 두는 것까지는 좋은데, async `onClick`이 promise를 반환하면 그걸 가로채서 로딩 상태를 자동으로 켜고 끄게 만들었다. 데모로는 멋졌다.

그런데 라이브러리 primitive로는 틀린 설계였다. 컴포넌트가 stateful해지고, 핸들러가 promise를 반환할 때만 동작하며, async 생명주기의 주도권을 호출자에게서 뺏는다. 결국 controlled `isLoading`으로 되돌렸다. Button은 순수하게 보여주기만 하고, 상태는 소비자가 쥔다. 마법보다 예측 가능성이다 — primitive에서는 이게 거의 항상 맞다.

Select에서도 같은 종류의 판단을 했다. single과 multi를 하나의 엔진으로 억지로 합치지 않고, **API는 통합하되 내부는 두 엔진으로 분기**시켰다. single은 Radix Select를 그대로 래핑해 폼 바인딩(`name`/`required`/hidden input/자동 닫힘)을 공짜로 가져오고, multi는 Radix Popover 위에 선택 모델·키보드 내비·체크박스를 직접 구현했다. 소비자가 쓰는 JSX는 양쪽이 똑같다.

```ts
type SelectV2SingleProps = Common & { multiple?: false; value?: string;  onValueChange?: (v: string)   => void }
type SelectV2MultiProps  = Common & { multiple: true;  value: string[];  onValueChange:  (v: string[]) => void }
export type SelectV2Props = SelectV2SingleProps | SelectV2MultiProps
```

`multiple`을 주면 컴파일러가 `value: string[]`와 배열 콜백을 요구하고, 빼면 `string` 버전을 강제한다. 잘못 쓰면 런타임이 아니라 타입 체크에서 막힌다.

## 접근

아이콘은 코드젠으로 뒤집었다. 디자이너가 `assets/icons/design-system/`에 SVG를 드롭하고 `pnpm generate`를 돌리면, plop이 디렉토리를 스캔해 barrel을 통째로 재생성한다. 파일을 지우면 export도 사라진다 — 디렉토리의 충실한 거울이다. Figma export preset에서 `Black => currentColor`를 켜둔 덕에 모든 아이콘이 컨테이너의 CSS `color`를 상속한다. per-icon variant 없이 테마잉이 된다.

여기서 빌드가 두 번 죽었다.

처음엔 `?react` 한 줄이면 끝날 줄 알았다. barrel이 `'...Alert_1px.svg?react'`로 import하는데, Storybook(Vite)에선 잘 되고 라이브러리 빌드(Rollup)만 죽었다. 원인 찾는 데 한참 걸렸다. `?react`는 vite-plugin-svgr가 만든 컨벤션이라 Vite에서만 의미가 있었다. Rollup의 `@svgr/rollup`은 그 쿼리를 못 읽고, `Alert_1px.svg?react`라는 디스크에 없는 파일을 찾다가 resolution에서 질식했다. barrel을 두 개로 가르는 대신, 쿼리를 떼고 실제 경로로 넘기는 작은 Rollup 플러그인을 끼웠다.

```js
{
  name: 'svg-query-resolver',
  resolveId(source, importer) {
    if (source.endsWith('?react')) {
      const actualPath = source.replace('?react', '')
      return path.resolve(path.dirname(importer), actualPath)
    }
    return null
  },
}
```

단일 barrel이 Vite(네이티브)와 Rollup(투명 rewrite) 양쪽에서 돈다. 개발 toolchain과 배포 toolchain이 다르다는 건, 둘 중 하나에 맞춰 소스를 갈라치는 것보다 얇은 어댑터 하나를 두는 게 낫다.

두 번째 죽음은 더 음흉했다. 빌드는 통과하는데 화면에서 아이콘이 잘렸다. `className="h-7 w-7"`로 키우면 스케일이 안 되고 클리핑됐다. SVGO가 범인이었다. `width`/`height`와 `viewBox`가 둘 다 있으면 `removeViewBox`가 viewBox를 redundant로 보고 지운다. viewBox가 사라지면 좌표계가 없어져 CSS 스케일이 path를 잘라버린다. Figma를 viewBox-only로 재출력하고, 빌드 쪽 SVGO 설정에서 `removeViewBox: false`로 못 박았다. 소스와 빌드 두 층에서 막아야 입력이 흔들려도 안전하다.

`?react` 사건은 타입 체크만으론 못 잡혔다(ambient `*.svg` 선언이 통과시킨다). 그래서 push 전에 type-check + 실제 Rollup 빌드를 강제하는 게이트를 걸었다. 같은 부류의 실패는 다시 내 로컬을 떠나지 못한다.

컴포넌트 구조 드리프트는 plop 제너레이터로 막았다. 프롬프트 하나로 컴포넌트·stories·barrel 3-파일 폴더를 컨벤션대로 찍어낸다. 이름은 정규식으로 PascalCase를 강제하고, stories는 `autodocs`가 미리 배선돼 생성 즉시 문서가 된다. 아이콘 코드젠도 결국 이 제너레이터의 `icon` 분기로 합쳐졌다.

## 결과

Select는 multi-select가 우회가 아니라 일급 모델이 됐다. 가장 자랑스러운 건 mirror mode다. trigger를 최장 옵션 너비에 맞추고 싶은데 옵션은 열리기 전 portal 안에 있어 측정이 안 된다. 숨긴 DOM mirror를 trigger와 같은 단일 grid 셀에 겹쳐서, `inline-grid w-fit`이 셀을 mirror에 맞추고 trigger가 그 너비를 입게 했다. ResizeObserver도 `useLayoutEffect`도 reflow 루프도 없는 순수 CSS 측정이다.

이게 보상받는 지점은 소비자 쪽이다. 교재→단원 tree-select에 탭까지 붙은 복잡한 selectbox를, trigger 스타일을 한 줄도 다시 짜지 않고 조립할 수 있었다. multi context가 중첩 깊이와 무관하게 전파되니, grid와 Tabs 안에 묻힌 `SelectV2.Item`도 자동으로 체크박스가 붙고 토글된다. 디자인 시스템 분리의 목표가 바로 이거였다.

아이콘은 export 보일러플레이트가 사라졌다. 디자이너가 폴더에 드롭하면 타입 있는 아이콘 컴포넌트가 나오고, Storybook 갤러리에 바로 반영된다.

역할로 보면, 설정상 모든 PR의 코드오너로 [확인]개 PR을 리뷰·머지했고, 0.0.6·0.0.16 릴리스를 직접 cut했다. PR 템플릿(버그는 AS-IS/TO-BE 표 강제)과 리뷰어 로테이션 정책을 만들고, commit/pr/ship 컨벤션을 Claude Code 스킬로 박아 누가 와도 같은 규율이 강제되게 했다.
