게임 하나에 업적이 2,222개 있다. 이걸 손으로 관리하는 사람들을 위한 스프레드시트가 커뮤니티에 돌아다녔다. 나는 그게 답답했다.

스프레드시트는 "내가 지금 뭘 했는지"는 보여줘도 "그래서 다음에 뭘 해야 가장 효율적인지"는 못 알려준다. 업적마다 주는 경험치가 다르고, 선행 조건이 얽혀 있고, 같은 작업으로 여러 업적이 한 번에 깨지기도 한다. 이건 사람이 눈으로 풀 문제가 아니라 그래프 최적화 문제였다. 그래서 직접 만들었다. 프론트(Next.js)부터 백엔드(NestJS + Prisma), 데이터 시딩까지 혼자.

## 문제

진짜 어려운 건 두 가지였다.

첫째, **다음에 뭘 할지 추천하는 게 단순 정렬이 아니다.** 업적 2,222개는 10개 대분류, 50개+ 서브카테고리로 계층 분류돼 있고, 각각 레어도(NORMAL 1,615 / RARE 476 / EPIC 131), 예상 소요 시간, 필요 아이템 레벨, 필요 재화가 AND/OR 복합 조건으로 붙는다. "경험치 높은 순"으로 정렬하면 끝날 것 같지만, 어떤 업적은 다른 업적을 깨는 길에 자동으로 딸려 온다. 목표 칭호까지 남은 경험치를 최소 비용으로 채우는 경로를 찾아야 했다.

둘째, **진행 상황을 사람이 일일이 체크하게 두면 아무도 안 쓴다.** 2,000개를 손으로 체크박스 누르라고 하면 그게 스프레드시트랑 뭐가 다른가. 게임 화면을 읽어서 자동으로 동기화하고 싶었다.

## 트레이드오프 결정

데이터 소스를 정하는 데서 한 번 갈렸다.

로스트아크는 공식 API(`developer-lostark.game.onstove.com`)를 제공한다. 캐릭터 레벨·아이템 레벨·서버 같은 건 여기서 정확하게 가져온다. 그런데 **개별 업적 달성 여부는 공식 API가 안 준다.** 여기서 선택지는 둘이었다. (1) 유저가 직접 2,000개를 체크하게 둔다, (2) 게임 화면을 캡처해서 OCR로 읽는다.

(2)는 정확도 100%가 아니다. OCR은 오인식이 나고, 게임 UI 폰트·해상도·스킨에 따라 흔들린다. 그래도 (1)의 "아무도 안 쓴다"가 더 치명적이라고 봤다. **부정확하지만 자동인 쪽**을 골랐다. 대신 OCR 결과는 곧바로 확정하지 않고 유저가 검수·수정하는 단계를 끼워서, 자동의 편함과 수동의 정확함을 반반 가져가기로 했다. 정답을 못 맞히는 자동화여도, 사람이 처음부터 입력하는 것보단 빠르다.

API 키 문제도 솔직히 적어둔다. 초기엔 키를 프론트에서 직접 썼다. 빨리 동작하는 게 우선이었기 때문인데, 이건 명백히 잘못된 결정이라 백엔드 릴레이 전환을 이슈로 남겨뒀다. [확인: 전환 완료 여부]

## 접근

**최적 루트는 그래프 문제로 모델링했다.** 업적을 노드로, 경험치를 가중치로 두고, 목표(삼라만상/천리한 칭호)까지 필요한 경험치를 최소 작업량으로 채우는 경로를 계산한다. 여기에 "같은 던전을 돌면 묶여 깨지는 업적" 같은 묶음을 반영해서, 단순히 효율 높은 순으로 줄 세우는 게 아니라 *함께 처리되는 것들을 한 덩어리로* 추천했다. 거기에 평균 진행 속도를 곱해 예상 달성 날짜까지 냈다. [확인: 알고리즘 정확 명칭 — 가중 그리디 / DP 여부]

**OCR 동기화는 게임 화면 캡처 → 텍스트 추출 → 업적 매칭 파이프라인**으로 짰다. 화면에서 읽어낸 업적명을 DB의 2,222개와 매칭하는데, 오인식 때문에 정확히 일치하는 경우가 드물어서 근사 매칭이 필요했다. 매칭된 결과를 유저 진행도(`completedStages Int[]`)에 반영한다. 이 `completedStages`는 다단계 업적에서 1단계만 깨고 2단계 건너뛴 뒤 3단계를 깨는 비선형 진행도 표현하려고 배열로 뒀다.

데이터 규모도 그냥 두면 느렸다. 업적 1,200개+를 한 번에 불러오면 응답이 늘어져서, 커서 기반 페이지네이션 + 필요한 필드만 `select` + 카테고리별 캐싱(`staleTime` 5분)으로 끊어 받았다.

여기에 **인터랙티브 맵**을 붙였다. 208개 맵 위에 모코코·업적·수집품 위치를 react-leaflet 마커로 찍고, 동선을 폴리라인으로 그린다. 원본 맵 이미지가 268MB라 그대로는 못 쓰니 sharp로 256px 타일을 뜨고 WebP로 변환해 **10.7MB까지(약 96% 감소)** 줄여 R2 CDN에 올렸다.

맵에서 제일 오래 잡은 버그가 마커였다. 처음엔 마커가 가끔 엉뚱한 좌표에 찍히거나 안 그려졌다. Leaflet이 컨테이너를 측정·배치하기 *전에* 마커를 먼저 렌더하면, 아직 좌표계가 안 잡힌 지도 위에 마커를 얹게 된다는 걸 한참 뒤에 알았다. 결국 "맵 준비 완료" 플래그를 두고, 레이아웃이 끝난 뒤에만 마커를 그리도록 게이트를 걸어 해결했다. 측정이 끝나야 좌표가 의미를 갖는다 — 이건 내가 다른 프로젝트의 DOM 측정 기반 조판에서도 똑같이 겪은 원리였다.

## 결과

혼자 만들어서 실제로 돌아가는 서비스가 됐다. 업적 2,222개 데이터베이스, 계층 필터·검색, OCR 일괄 등록, 208개 맵 뷰어, 그리고 최적 루트 추천까지 풀스택으로 엮였다. [확인: 사용자 수 / 트래픽 등 공개 가능한 지표]

수치로 잡히는 것 하나는 맵 에셋이다. 268MB → 10.7MB. 타일링 + WebP만으로 96%를 깎아서, 무거운 월드맵을 모바일에서도 끊김 없이 띄울 수 있게 됐다.

## 회고

가장 배운 건 "정확도 100%가 아니어도 자동이 이긴다"는 판단이었다. OCR을 처음엔 완벽하게 만들려고 욕심냈는데, 그 욕심이 기능을 못 내보내게 막고 있었다. 검수 단계를 끼워 불완전함을 인정하고 나서야 출시할 수 있었다.

아쉬운 건 최적 루트의 데이터 의존성이다. "평균 진행 속도로 달성 날짜 계산"은 유저 데이터가 충분히 쌓여야 신뢰도가 나오는데, 개인 프로젝트라 그 양을 모으는 게 가장 어려웠다. 알고리즘보다 데이터 확보가 병목이었다. 다시 만든다면 추천 근거를 유저에게 더 투명하게 보여줘서, 데이터가 적을 때도 "왜 이걸 추천하는지"로 신뢰를 살 것 같다. [확인]
