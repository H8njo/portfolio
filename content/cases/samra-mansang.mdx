게임 하나에 업적이 2,222개 있다. 이걸 손으로 관리하려고 커뮤니티에서 스프레드시트가 돌아다녔는데, 나는 그게 좀 답답했다.

스프레드시트는 "내가 지금 뭘 했는지"는 보여줘도 "그래서 다음에 뭘 해야 가장 효율적인지"는 안 알려준다. 업적마다 주는 경험치가 다르고, 선행 조건이 얽혀 있고, 같은 작업 한 번으로 여러 업적이 같이 깨지기도 한다. 눈으로 풀 문제가 아니었다. 그래프 최적화 문제에 가까웠다. 그래서 그냥 만들었다. 프론트(Next.js)부터 백엔드(NestJS + Prisma), 데이터 시딩까지 혼자.

## 문제

막상 손대보니 걸리는 게 두 군데였다.

하나는 **다음에 뭘 할지 추천하는 게 단순 정렬이 아니라는 점.** 업적 2,222개는 10개 대분류, 50개+ 서브카테고리로 계층 분류돼 있고, 각각 레어도(NORMAL 1,615 / RARE 476 / EPIC 131), 예상 소요 시간, 필요 아이템 레벨, 필요 재화가 AND/OR 복합 조건으로 붙는다. "경험치 높은 순"으로 줄 세우면 끝날 것 같지만 아니다. 어떤 업적은 다른 업적 깨는 길에 알아서 딸려 온다. 목표 칭호까지 남은 경험치를 최소 비용으로 채우는 경로를 찾아야 했다.

다른 하나는 **진행 상황을 사람이 일일이 체크하게 두면 아무도 안 쓴다는 것.** 2,000개를 손으로 체크박스 누르라고 하면 그게 스프레드시트랑 뭐가 다른가. 게임 화면을 읽어서 자동으로 동기화하고 싶었다.

## 트레이드오프 결정

데이터 소스를 정하는 데서 한 번 막혔다.

로스트아크는 공식 API(`developer-lostark.game.onstove.com`)를 준다. 캐릭터 레벨·아이템 레벨·서버 같은 건 여기서 정확하게 가져온다. 그런데 **개별 업적 달성 여부는 공식 API가 안 준다.** 그래서 남은 선택지가 둘이었다. (1) 유저가 직접 2,000개를 체크하게 두거나, (2) 게임 화면을 캡처해서 OCR로 읽거나.

(2)는 정확도 100%가 안 된다. OCR은 오인식이 나고, 게임 UI 폰트·해상도·스킨 따라 결과가 흔들린다. 그래도 (1)의 "아무도 안 쓴다"가 더 치명적이라고 봤다. 그래서 **부정확하지만 자동인 쪽**을 골랐다. 대신 OCR 결과를 곧바로 확정하지 않고 유저가 검수·수정하는 단계를 끼웠다. 자동의 편함이랑 수동의 정확함을 반반 가져가는 식이다. 다 못 맞히는 자동화여도 처음부터 손으로 입력하는 것보단 빠르니까.

API 키 문제도 솔직히 적어둔다. 초기엔 키를 프론트에서 그냥 직접 썼다. 일단 빨리 돌아가는 게 우선이었어서. 당연히 잘못된 결정이라 백엔드 릴레이 전환을 이슈로 남겨뒀다. [확인: 전환 완료 여부]

## 접근

**최적 루트는 그래프 문제로 모델링했다.** 업적을 노드로, 경험치를 가중치로 두고, 목표(삼라만상/천리한 칭호)까지 필요한 경험치를 최소 작업량으로 채우는 경로를 계산한다. 여기에 "같은 던전 돌면 묶여서 깨지는 업적" 같은 묶음을 반영했다. 효율 높은 순으로 줄 세우는 게 아니라 *함께 처리되는 것들을 한 덩어리로* 추천하는 쪽으로. 거기에 평균 진행 속도를 곱해서 예상 달성 날짜까지 냈다. [확인: 알고리즘 정확 명칭 — 가중 그리디 / DP 여부]

**OCR 동기화는 게임 화면 캡처 → 텍스트 추출 → 업적 매칭 파이프라인**으로 짰다. 화면에서 읽어낸 업적명을 DB의 2,222개랑 매칭해야 하는데, 오인식 탓에 정확히 일치하는 경우가 드물었다. 근사 매칭이 필요했다. 매칭된 결과는 유저 진행도(`completedStages Int[]`)에 반영한다. 이 `completedStages`를 배열로 둔 이유는 다단계 업적에서 1단계만 깨고 2단계는 건너뛴 뒤 3단계를 깨는, 그런 비선형 진행을 표현해야 했기 때문이다.

```ts
model UserAchievement {
  status          String  // not_started / in_progress / completed
  currentStage    Int     // 하위호환용. completedStages.length와 동기화
  completedStages Int[]   // 개별 완료 단계 인덱스 — 비선형 진행 표현
  // ...
  @@unique([userId, achievementId])
}
```

데이터 규모도 그냥 두면 느렸다. 업적 1,200개+를 한 번에 불러오면 응답이 축 늘어져서, 커서 기반 페이지네이션 + 필요한 필드만 `select` + 카테고리별 캐싱(`staleTime` 5분)으로 끊어 받았다.

여기에 **인터랙티브 맵**을 붙였다. 208개 맵 위에 모코코·업적·수집품 위치를 react-leaflet 마커로 찍고, 동선을 폴리라인으로 그린다. 원본 맵 이미지가 268MB라 그대로는 못 쓴다. sharp로 256px 타일을 뜨고 WebP로 변환해서 **10.7MB까지(약 96% 감소)** 줄여 R2 CDN에 올렸다.

맵에서 제일 오래 잡은 버그가 마커였다. 처음엔 마커가 가끔 엉뚱한 좌표에 찍히거나 아예 안 그려졌다. 한참 헤맸는데, Leaflet이 컨테이너를 측정·배치하기 *전에* 마커를 먼저 렌더하면 아직 좌표계가 안 잡힌 지도 위에 마커를 얹게 되는 거였다. 결국 "맵 준비 완료" 플래그를 두고 레이아웃이 끝난 뒤에만 마커를 그리도록 게이트를 걸었다. (측정이 끝나야 좌표가 의미가 있으니까.) 다른 프로젝트의 DOM 측정 기반 조판에서도 똑같이 당했던 거라, 이번엔 좀 빨리 알아챘어야 했는데.

## 결과

혼자 만든 게 어쨌든 실제로 돌아가는 서비스가 됐다. 업적 2,222개 데이터베이스, 계층 필터·검색, OCR 일괄 등록, 208개 맵 뷰어, 최적 루트 추천까지 풀스택으로 엮였다. [확인: 사용자 수 / 트래픽 등 공개 가능한 지표]

수치로 깔끔하게 잡히는 건 맵 에셋 정도다. 268MB → 10.7MB. 타일링 + WebP만으로 96%를 깎았으니 무거운 월드맵도 모바일에서 끊김 없이 뜬다.

## 회고

OCR을 처음엔 완벽하게 만들려고 욕심을 냈었다. 근데 그 욕심이 기능을 계속 못 내보내게 막고 있더라. 검수 단계를 끼워서 불완전함을 그냥 인정하고 나서야 출시가 됐다. 정확도 100%가 아니어도 자동이 이긴다는 걸, 머리로는 알았는데 손으로 깨닫는 데 시간이 좀 걸렸다.

아쉬운 건 최적 루트의 데이터 의존성이다. "평균 진행 속도로 달성 날짜 계산"이라는 게 유저 데이터가 충분히 쌓여야 신뢰가 가는데, 개인 프로젝트라 그 양을 모으는 게 제일 힘들었다. 알고리즘보다 데이터 확보가 병목이었던 셈이다. 다시 만든다면 추천 근거를 유저한테 더 투명하게 보여줘서, 데이터가 적을 때도 "왜 이걸 추천하는지"로 신뢰를 사보고 싶다. [확인]
