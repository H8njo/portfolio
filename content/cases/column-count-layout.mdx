`column-count: 2` 한 줄이면 끝날 줄 알았다. 결과적으로 같은 문제를 세 번 다시 풀었다.

문제지 편집기에서 문제들을 A4 2단으로 깔아야 했다. 긴 지문 하나가 1열 끝과 2열 시작에 반으로 잘려 걸쳐졌다. 일반 웹이면 넘어가지만 이건 인쇄돼서 학생한테 가는 문제지다. 문제가 중간에 잘리면 못 쓴다. 게다가 카드마다 호버·선택·오류제보 같은 인터랙션이 붙어야 했고(CSS 컬럼으론 카드 단위 제어가 안 된다), 서버에서 같은 HTML을 PDF로 변환하는데 CSS 다단은 화면이랑 PDF가 달라졌다.

## 실패한 시도들

이게 한 번에 풀린 게 아니다. 부끄럽지만 거쳐온 길을 적는 게 이 작업의 핵심이다.

처음엔 CSS multi-column 좌표 기반으로 갔다가, 안 맞아서 JS로 높이를 직접 재는 방식으로 틀었다. 그러다 "글자 수평 잘림"을 잡겠다고 **다시 CSS multi-column으로 회귀**했다. `break-before: column`, `column-fill: auto`, 잘린 조각에 `.cropped` 클래스 주고 `translateX`로 위치 보정까지. 그때 커밋 메시지를 "글자 수평 잘림 해결 (아마도)"라고 적었는데, **'아마도'를 붙인 시점에 사실 진 거였다.**

거기서부터는 예외처리 지옥이었다. 첫 페이지만 헤더가 164px라 컬럼 분할이랑 충돌했고, `index === 3`을 하드코딩으로 밀어내는 보정이 자꾸 늘었다. 하드코딩 예외가 늘기 시작하면 접근 자체가 틀렸다는 신호다. 결국 롤백하고 JS 측정 + 3단 폴백(의미 블록 → 줄 → 픽셀 순으로 글자 안 잘리는 경계를 우선)으로 돌아왔다. V1에서만 접근법을 다섯 번쯤 갈아치웠다.

## 뒤집은 한 끗

실패에서 건진 건 하나였다. **CSS multi-column을 '표시 레이아웃'으로 쓰면 안 된다.** 근데 그걸 버리는 대신 각도를 틀었다 — 표시 말고, 큰 아이템 하나를 **자르는 도구**로만 쓰면 어떨까.

키 큰 콘텐츠를 `column-count: 1` 컨테이너에 넣고 고정 높이로 클립하면, 세로로 넘친 내용이 멀티컬럼 규칙에 따라 가로로 흐른다. 그 흐름을 슬라이스마다 `translateX`로 하나씩 들여다본다. 콘텐츠를 진짜로 자르거나 복제하지 않고, 한 흐름을 여러 창으로 보는 거다.

```
[세로로 긴 콘텐츠]        [column-count:1 + 고정높이 클립]
┌──┐                     ┌──┬──┬──┐
│ 1│                     │ 1│ 3│ 5│  ← 가로로 흐른 슬라이스
│ 2│        →            │ 2│ 4│ 6│
│ …│                     └──┴──┴──┘
└──┘                      각 칸을 translateX로 들여다봄
```

표시 레이아웃으로 쓰면 통제가 안 되던 그 CSS를, 단일 아이템 내부 크롭으로만 쓰니 안전했다. 망한 기술을 버린 게 아니라 쓰는 각도를 바꾼 거였다.

## 세 번 풀면서 다듬은 것

- **v1**(사내 문제 에디터): 도메인에 박힌 채 위 6번의 시행착오를 다 거친 원형.
- **v2**(사내 다른 제품): 임의 콘텐츠로 일반화하고 크롭 트릭을 안정화. 근데 핵심 훅이 210줄짜리 god-hook이라 손대기 무서운 코드였다.
- **v3**(독립 npm 패키지 `column-pager`): 앱에서 떼어내 순수 코어 / 측정 어댑터 / React 렌더 3계층으로 다시 짰다.

v3에서 가장 신경 쓴 건 **검증된 수학(크롭·측정)은 그대로 이식하고 아키텍처만 새로 짠 것**이다. 그 수학은 수많은 버그를 거치며 맞춰놓은 자산이라, 백지에서 다시 짜면 회귀가 쏟아진다. 위험을 진 곳(구조)과 안 진 곳(검증된 계산)을 갈라놨다.

서브픽셀도 한 번 데였다. `2090 / 697 = 2.998`인데 실제론 3컬럼이다. `Math.floor`면 2로 내려가 마지막 컬럼이 통째로 사라진다. DOM 측정값은 폰트 로드·줌·레이아웃 압박으로 정수배에서 늘 살짝 빗나가서, "정수 N배여야 하는" 값엔 `round`를 쓴다.

배포는 `semantic-release`로 main 머지가 곧 publish다. 첫 배포 땐 코드가 아니라 메타데이터에서 막혔다 — `repository` 필드 누락으로 provenance publish 실패, 새 패키지라 `publishConfig.access: public` 미설정으로 거부. 자동 파이프라인은 "코드가 맞아도 배포 메타에서 막힌다"는 걸 그때 배웠다.

## 결과

지금은 라이브러리 없이 임의의 React 콘텐츠를 고정 높이 페이지로 나누는 오픈소스 패키지가 됐다. 측정은 실제 브라우저 레이아웃에 의존해서 단위 테스트로는 한계가 있고, 그래서 시각 검증은 Storybook 13개 스토리로 한다(health 96). 이 사이트의 라이브 데모가 그 패키지로 돈다.
