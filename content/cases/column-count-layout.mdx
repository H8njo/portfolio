## 문제

교육 콘텐츠 편집 도구에서 문제지를 **A4 다단(2열) 레이아웃**으로 조판해야 했다. 그런데 브라우저 표준인 CSS `column-count`는 긴 지문을 **컬럼 경계에서 임의로 잘라버린다**. 출판물인 문제지에서 한 문제가 두 컬럼에 어중간하게 걸쳐 깨지는 건 치명적이었다. 동료들도, 당시 떠올려본 접근들도 이 "잘림"을 깔끔하게 푸는 방법을 찾지 못했다.

브라우저의 자동 컬럼 분배를 포기하고, **DOM을 직접 측정해 수동으로 배치하는 길**을 택했다. 구현 복잡도는 크게 올라가지만, "정확히 어디서 끊고 어디로 이어붙일지"를 코드가 통제할 수 있어야 문제지 품질을 보장할 수 있다고 판단했다.

## 접근

- **오프스크린 측정:** 숨긴 영역에 콘텐츠를 단일 컬럼으로 렌더하고 각 아이템의 `getBoundingClientRect()` 높이를 측정 → 열 높이를 누적하며 넘치면 다음 열/페이지로 배치.
- **노드 분할:** 한 컬럼보다 큰 아이템은 노드를 복제해 두 조각으로 자르고, `transform`으로 끌어올려 "잘리지 않고 이어지는 것처럼" 표현.
- **악명 높은 버그들:** 첫 페이지 헤더 높이 차이 보정, 잘린 노드가 2열로 갈 때의 가로 위치 어긋남(`translateX` 보정), 그리고 정교화 단계에서 만난 **서브픽셀 반올림으로 마지막 컬럼이 누락되던 버그**(`Math.floor` → `Math.round`)까지.
- **정직하게:** 최초 발상과 v1 구현은 직접 했고, 이후 성능 최적화 일부는 **AI를 페어 도구로 활용**해 측정·해결했다. 발상은 사람이, 반복 최적화는 도구를 레버리지하는 방식.

## 결과

- 측정 병목을 잡아 페이지 높이 계산을 **lazy batch + 캐시**로 전환(반복 측정 수백 회 → 일괄). {/* TODO(수치): 실제 ms 절감값 (NDA 확인 후) */}
- 신뢰성 확보를 위해 **브라우저 모드 테스트 45개**를 직접 작성(단위 + DOM 측정 + 페이지네이션 통합 + 분할 크롭).
- 사내 v1에서 출발 → 정교화 → **독립 npm 패키지 `ColumnPager`로 재설계·오픈소스화**. (컴파운드 컨트롤, `onPagesGenerated`로 렌더 HTML 방출 → 서버 PDF 변환 연동)
- 이 사이트의 **라이브 데모**에서 "남들 방법(잘림) vs 내 방법"을 직접 토글해 볼 수 있다.
