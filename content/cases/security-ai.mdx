AI 기반 보안관제 시스템의 어드민을 맡았다. 보안 관제사가 네트워크 공격을 자동으로 탐지하는 머신러닝 모델을, 코드 한 줄 안 쓰고 GUI만으로 직접 학습시키는 화면이다. 처음엔 "그냥 폼 몇 개 모은 마법사겠지" 했다. 그게 이 프로젝트에서 제일 오래 붙잡은 화면이 됐다.

마법사는 큰 단계가 5개다. 데이터 선택 → 특징 추출 → 알고리즘 학습 → 테스트 → 요약. 그런데 큰 단계 안에 작은 단계가 또 있었다. 예를 들어 첫 단계 '데이터 선택'만 해도, 새로 수집할지·이미 수집된 걸 쓸지·학습해둔 걸 쓸지·추천받은 걸 쓸지에 따라 화면이 네 갈래로 갈라진다. 그래서 실제 이동 경로는 1-1, 1-2, 1-3 … 2-1, 2-2 … 처럼 **2층 구조**였다. 큰 단계(step) 밑에 작은 단계(substep)가 매달린 셈이다.

화면으로 보면 폼이지만, 실제로는 ML 파이프라인 하나를 통째로 운영하는 콘솔이었다. 관제사는 각 단계에서 설정만 채우고 마지막에 실행을 누르면, 뒤에서 수집·전처리·정규화·학습·테스트가 줄줄이 돌아간다.

## 문제

마법사라는 게 단순한 듯 안 단순하다. 사용자는 이 2층 단계를 앞뒤로 들락거린다. 1-1에서 데이터를 고르고 3단계까지 갔다가, "아 데이터 잘못 골랐네" 하고 1-1로 돌아온다. 새로고침도 누르고, 뒤로가기도 누르고, 링크를 북마크해놓고 다음 날 다시 들어오기도 한다.

여기서 세 가지가 한꺼번에 얽힌다.

**첫째, 지금 몇 층 몇 단계에 서 있는지를 잃으면 안 된다.** 새로고침 한 번에 맨 앞으로 튕겨 나가면 관제사는 처음부터 다시다.

**둘째, 뒤로 갔을 때 앞서 채운 게 그대로 남아 있어야 한다.** 그리고 그 입력이 얕지 않다. '알고리즘 학습' 한 단계만 봐도 옵티마이저, 탐색 방식, 교차검증 횟수, 비율… 설정이 몇 겹으로 중첩된다.

**셋째, 그 반대로 앞 단계를 바꾸면 거기 기대던 뒤 단계는 무효가 돼야 한다.** 1단계 데이터를 갈아끼웠는데 2단계 특징·3단계 알고리즘이 옛 데이터 기준으로 남아 있으면, 관제사는 모르고 그대로 학습을 돌린다. 그러면 몇 시간짜리 워커 작업이 엉뚱한 조합으로 통째로 낭비된다.

흔히 하듯 `currentStep` 숫자 하나로 이걸 들고 가면 다 무너진다. 큰 단계·작은 단계·단계별 중첩 데이터·완료 여부·앞뒤 의존이 서로 맞물려 돌아야 했다. 그러니까 이 화면에 진짜로 필요했던 건 예쁜 폼이 아니라, 그 다섯 개가 **유기적으로 연결된 상태 관리자** 하나였다.

그리고 더 근본적인 문제. 학습은 비동기다. 실행을 누른다고 결과가 바로 안 나온다. 수집이 끝나야 특징을 뽑고, 특징이 나와야 학습을 돌리는데, 각 단계가 몇 분에서 몇 시간씩 걸린다. 그동안 화면은 "지금 어디까지 됐는지"를 정직하게 보여줘야 했다. 안 그러면 관제사는 멈춘 건지 도는 건지 몰라서 계속 새로고침만 하니까.

## 트레이드오프 결정

이 상태 관리자를 어떻게 짤지부터 정해야 했다. 붙들고 보니 상태가 성격이 다른 두 덩어리였다. 하나는 "지금 어디 서 있나"(내비게이션), 하나는 "지금까지 뭘 채웠나"(데이터). 이 둘을 한 군데 뭉쳐 놓으면 서로 발을 밟는다. 그래서 책임을 갈랐다 — **내비게이션은 URL에, 데이터는 리듀서에.**

먼저 내비게이션. 보통은 그냥 React state에 `currentStep`을 들고 간다. 쉬우니까. 근데 그러면 새로고침하면 날아가고, 뒤로가기는 마법사를 빠져나가버리고, 딥링크도 안 된다. 그래서 **지금 위치(큰 단계 + 작은 단계)를 React state가 아니라 URL을 단일 소스로** 삼았다. 단계 전환은 곧 `router.push`다.

```js
// context.js — 현재 위치(step/substep)는 URL에서 파생
const currentStep = Number(router.query.step);
const currentSubstep = router.query.substep;

const goToStep = ({ step, substep }) => {
  if (step === currentStep && substep === currentSubstep) return;
  const query = { step };
  if (substep) query.substep = substep;           // 2층: 큰 단계 밑에 작은 단계
  router.push({ pathname: router.pathname, query });
};
```

`step`과 `substep`이 URL에 같이 실리니, 1-3에 있다가 새로고침하든 링크를 공유하든 정확히 그 자리로 돌아온다. 위치는 URL이 진실이고, 화면은 그걸 읽어서 그리기만 한다.

다음은 데이터. 수집명·특징 목록·알고리즘 파라미터 같은 실제 입력은 URL에 다 못 넣으니 리듀서가 들고 갔고, `useImmerReducer`를 썼다. 단계별 객체가 워낙 깊게 중첩돼서(`step3.algorithmOptions.explorationDatas.cvFold` 이런 식) 불변 갱신을 손으로 펼치면 코드가 지옥이 됐을 거다. immer로 "그냥 직접 바꾸듯" 쓰고 불변성은 라이브러리에 맡겼다.

둘로 쪼개는 게 처음엔 마음에 안 들었는데, 막상 갈라 놓으니 각자 단순해졌다. 그리고 이 둘을 하나로 묶는 게 다음 장의 규칙들이었다.

## 접근

URL과 리듀서를 하나의 살아 있는 상태 관리자로 엮은 건 몇 가지 규칙이었다. 이 규칙들이 앞서 말한 "완료 여부"와 "앞뒤 의존"을 실제로 굴러가게 만든다.

첫 규칙은 "리소스 id가 생기면 그 단계가 완료된다"였다. 수집을 만들면 `collectId`가, 특징그룹을 만들면 `featureGroupId`가 백엔드에서 내려온다. 그 id가 세팅되는 순간 해당 단계 `completed`를 같이 켜는 패턴을 리듀서 전체에서 일관되게 박았다.

```js
case 'SET_COLLECT_ID':
  state.collectId = action.payload;
  state.step1.completed = !!action.payload; break;
case 'SET_FEATURE_GROUP_ID':
  state.featureGroupId = action.payload;
  state.step2.completed = !!action.payload; break;
```

"앞이 바뀌면 뒤를 비운다"도 리듀서에 명시했다. Step1 데이터가 바뀌면 거기 의존하는 Step2·Step3 상태를 초기화한다. 이것도 한 번 데여서 넣은 거다. 처음엔 데이터만 갈아끼우고 뒤 단계는 그대로 뒀더니, 화면상으론 멀쩡한데 안에선 옛 특징그룹 id가 살아 있더라. 생긴 게 같다고 같은 게 아니니까.

완료 여부를 들고 있으니 네비게이션 헤더에서 바로 써먹을 수 있었다. **이미 지난 단계인데 미완료면 `error`로 표시**해서, 빠뜨린 설정을 즉시 빨갛게 드러냈다.

```js
const step1Status = step > 1 && !step1.completed && 'error';
// description: step1Status === 'error' && '수집데이터 설정하세요.'
```

비동기 학습 추적은 별도 모델이 필요했다. 모델 1건은 **crawl → feature → normalize → train → test** 5단계를 거치고, 각 단계가 `S/R/C/E`(대기/실행중/완료/에러) 상태를 갖는다. 백엔드는 각 단계의 상태 컬럼을 한 글자씩 이어붙여 progress 문자열 하나로 만들어 내려줬다.

```sql
concat(
  NVL(CONCAT('C', crawl_step), ''),     -- Crawl
  NVL(CONCAT('F', feature_step), ''),   -- Feature
  NVL(CONCAT('N', normalize_step), ''), -- Normalize
  NVL(CONCAT('R', train_step), ''),     -- tRain
  NVL(CONCAT('T', test_step), '')       -- Test
) as progress_step
```

재밌었던 건 이 상태 조합만으로 "지금 어느 워커가 일하고 있나"까지 읽어낸다는 거였다. 수집·특징추출(crawl/feature)은 Java 워커가, 정규화·학습·테스트(normalize/train/test)는 Python 워커가 맡는 이종 분산 구조였다. 대시보드 작업 큐는 단계 상태의 조합만 보고 어느 큐에 일이 쌓였는지를 분류했다.

```ts
// 1: 대기 Java, 2: 작업중 Java, 3: 대기 Python, 4: 작업중 Python
const workerCondition = {
  1: "and crawl_step = 'S'",
  2: "and ((crawl_step='R' or feature_step='R') or (crawl_step='E' and feature_step='R'))",
  3: "and (crawl_step='C' and feature_step='C' and normalize_step is null and train_step is null and test_step is null)",
  4: "and (normalize_step='R' or train_step='R' or test_step='R')",
};
```

NestJS에서 Python 워커로 작업을 넘기는 다리는 Celery였다. RabbitMQ를 브로커로 두고 `celery-ts`로 작업을 보냈다. 마법사가 실행을 누르면 여기로 작업이 떨어지고, 화면은 그 진행률을 폴링해서 위 progress 문자열을 다시 그렸다.

마법사 데이터의 범위 지정에는 Elasticsearch가 들어갔다. 관제사가 직접 ES 쿼리를 짤 순 없으니 `react-awesome-query-builder`를 ES 스키마에 물려서, GUI로 bool 쿼리를 조립하게 했다. 여기서 한 번 호되게 당했다. 검색 폼에서 기간(range) 조건을 줘도 필터가 안 먹고 전체가 나오는 버그가 있었는데, range를 `bool.should`에 넣고 있었던 거다.

```js
// should는 must가 있으면 '가점용 OR'일 뿐 필수 조건이 아니다
query.bool.should.push({ range: { time: { gte, lte } } }); // ← 잘못
```

ES `bool`에서 `must`가 하나라도 있으면 `should`는 "있으면 가점, 없어도 통과"라 조건이 아니게 된다. 그래서 기간 조건이 사실상 무시됐던 거다. 솔직히 처음엔 ES가 고장난 줄 알고 한참 엉뚱한 데를 봤는데, range를 `must`로 옮기니 바로 됐다. (지금 다시 짠다면 점수가 필요 없는 기간·동등 필터는 `filter`에 넣었을 거다. 캐시도 타고.)

## 결과

- 관제사가 코드 없이 5단계 GUI만으로 탐지 모델 1건을 끝까지 학습·배포할 수 있게 됐다. URL을 단일 소스로 둔 덕에 새로고침·뒤로가기·딥링크가 다 자연스럽게 동작한다.
- "앞이 바뀌면 뒤를 무효화"를 리듀서에 명시하고, 지난 단계 미완료를 `error`로 드러내서, 옛 설정으로 몇 시간짜리 학습을 헛돌리는 일을 막았다.
- 비동기 학습은 5단계 × 4상태 progress 문자열로 추적해, 관제사가 "멈춘 건지 도는 건지" 새로고침으로 확인할 필요가 없어졌다. 같은 상태 조합으로 Java/Python 워커 큐 현황까지 한 화면에 보여줬다.
- ES 기간 필터 버그를 잡아, 기간 조건이 사실상 무시되던 검색이 의도대로 좁혀지게 됐다.

## 회고

이 프로젝트로 "GUI로 비동기 파이프라인을 운영한다"는 게 뭔지 몸으로 알았다. 화면은 폼처럼 생겼지만, 본질은 백엔드에서 길게 도는 작업의 상태를 정직하게 비추는 거울이었다. 그래서 제일 신경 쓴 건 예쁜 UI가 아니라 "지금 어디까지 됐고, 뭐가 빠졌고, 어디서 깨졌나"를 거짓 없이 보여주는 거였다.

습관도 몇 개 붙었다. 마법사·탭처럼 단계가 있는 화면은 그 위치를 — 2층이면 큰 단계와 작은 단계 둘 다 — URL에 둔다. state로 들고 있으면 새로고침 한 방에 무너지니까. 그리고 한 단계의 데이터가 뒷단계를 무효화하는 의존이 있으면, 그 무효화를 UI가 아니라 상태 갱신 로직(리듀서)에 못 박는다. 화면에서만 가리면 안에선 옛 값이 살아 있다. 이건 한 번 당하고 막았다.

ES `bool`의 `must`/`should`/`filter` 의미 차이는 그날 이후로 안 헷갈린다. 검색이 "필터를 줬는데 전부 다 나온다" 싶으면 제일 먼저 should에 잘못 넣었나부터 본다.

하나 더. 시작할 땐 AI/ML 지식이 사실상 없었다. 특징 추출이 뭔지, 모델 검증이 뭔지도 몰랐다. 근데 이 화면을 제대로 짜려면 각 단계가 뭘 하는 건지를 알아야 폼 하나도 똑바로 못 만들어서, 백엔드·데이터 쪽 어깨너머로 ML 파이프라인이 어떻게 도는지를 계속 주워들었다. crawl → feature → normalize → train → test가 왜 그 순서인지, 정규화가 왜 따로 한 단계로 떨어져 있는지 같은 걸 알아가는 게 의외로 재밌었다. 프론트 화면 하나 만들겠다고 붙었다가, ML 파이프라인의 생김새를 덤으로 배운 셈이다.

Elasticsearch도 이때 처음 만졌다. 색인이 어떻게 잡히고 `bool` 쿼리가 어떻게 점수를 매기는지가 평소 쓰던 RDB랑은 사고방식 자체가 달라서, 그 `must`/`should`/`filter`에 호되게 당하긴 했어도 동작 원리를 새로 알아가는 게 재밌었다. 모르던 영역이 일하면서 하나씩 채워지는 그 감각이, 이 프로젝트에서 제일 좋았던 부분이다.
