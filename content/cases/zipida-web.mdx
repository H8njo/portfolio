글 하나 올리는 데 개발자를 부르는 회사가 제일 비효율적이다.

입사 초였다. AI 솔루션 회사의 공식 홈페이지를 거의 혼자 맡아 만들었다. 회사 소개, 프로젝트, 아카이브, 채용, 블로그. 근데 문제는 블로그였다. 디자이너랑 기획자가 글을 쓰는데, 그걸 화면에 띄우는 건 매번 나였다. 글 받고, HTML로 옮기고, 배포하고. 글 하나에 사람 둘이 붙는 셈이다.

## 문제

콘텐츠를 코드 안에 박아두면, 콘텐츠를 바꿀 때마다 개발자가 필요하다.

처음엔 그냥 페이지마다 마크업을 짰다. 새 글이 나오면 컴포넌트를 하나 더 만들고. 두세 개까지는 그럭저럭 버틸 만했다. 그런데 블로그라는 건 글이 계속 쌓이는 거고, 글마다 개발자가 손을 대야 한다면 그건 블로그가 아니라 그냥 정적 페이지 더미다. 작성자가 코드를 못 만지니, 코드와 콘텐츠를 떼어놓지 않으면 답이 없었다.

## 트레이드오프 결정

런타임에 글을 불러올까, 빌드할 때 굳혀버릴까.

CMS를 붙여서 DB에서 글을 읽어오는 길도 있었다. 그런데 어차피 회사 홈페이지였다. 글이 하루에 수십 개 올라오는 것도 아니고, 트래픽이 폭주하는 것도 아니다. 서버를 한 대 더 띄우고 DB를 관리하는 비용이, 얻는 것에 비해 너무 컸다. 그래서 Gatsby로 갔다. 빌드할 때 모든 글을 GraphQL로 긁어 정적 HTML로 굳히는 방식. 글은 `content/<카테고리>/*.mdx` 파일로 두고, 파일을 추가하면 빌드 타임에 페이지가 생긴다.

대신 트레이드오프가 있었다. 정적 사이트라 "글을 올린다 = 빌드를 다시 돌린다"가 된다. 작성자가 파일을 올리면 누군가는(혹은 CI가) 빌드를 한 번 더 돌려야 한다. 실시간은 아니다. 그래도 회사 홈페이지엔 그 정도 지연이 문제가 안 됐다. 운영 부담을 줄이는 쪽을 택했다.

## 접근

MDX를 고른 게 핵심이었다. 마크다운인데 안에서 React 컴포넌트를 쓸 수 있다. 작성자는 평범하게 글을 쓰다가, 필요하면 미리 만들어둔 컴포넌트를 끼워넣는다.

`gatsby-node.js`에서 빌드 시 모든 MDX 노드를 조회해 페이지를 하나씩 찍어냈다. 파일 경로에서 `slug`랑 `folder`(카테고리)를 뽑아 필드로 박아두면, 이후 GraphQL에서 카테고리별로 글을 갈라 쓸 수 있다.

```js
// gatsby-node.js — 파일 하나당 페이지 하나
result.data.allMdx.edges.forEach(({ node }) => {
  createPage({
    path: node.fields.slug,
    component: path.resolve(`./src/components/posts-page-layout.js`),
    context: { id: node.id },
  });
});
```

그런데 정작 시스템을 만드는 것보다 어려운 게 따로 있었다. 작성자가 이걸 어떻게 쓰는지를 알려주는 일. 그래서 README에 비개발자용 작성 매뉴얼을 직접 썼다. 프론트매터에 `title`, `author`, `image`, `categories`, `date`를 어떻게 적는지. 이미지는 `./images/파일이름` 규약을 지켜야 한다는 것. 그리고 한 줄을 굵게 남겼다. "HTML 태그를 문법에 안 맞게 쓰면 그 글이 통째로 안 보인다." 이건 내가 당해본 거다. 시스템을 만든 사람만 아는 함정을, 쓰는 사람은 모른 채 밟는다. 그래서 적어뒀다.

## 결과

작성자가 mdx 파일과 이미지만 폴더에 넣으면 글이 떴다. 나를 거치지 않고. 만들려던 게 그거였다.

배포는 한 방에 정리되지 않았다. CI가 갖춰지지 않은 환경이라, 처음엔 빌드 결과물인 `public/`을 `.gitignore`에서 빼고 git에 통째로 커밋했다. 커밋 하나에 빌드된 HTML/CSS/JS 수천 줄이 들어간다. 정석은 아니다. 그래도 서버에서 `git pull`만 하면 바로 서빙되니, "push = 배포"가 만들어졌다. 이후엔 `ecosystem.config.js`로 `serve`를 PM2에 물려 self-host 했고, 최종적으로 Netlify Git 연동 CI로 옮겼다. 정적 사이트 하나를 세 가지 방식으로 배포해본 셈이다.

빌드는 자주 깨졌다. Gatsby v2는 플러그인이랑 sharp 같은 네이티브 의존이 많아서, `node_modules` 상태가 조금만 어긋나도 `gatsby build`가 터졌다. 원인 추적이 길어질 때가 잦아서, 결국 이것도 README에 운영 매뉴얼로 남겼다. `npm update` 하고 `node_modules` 통째로 지운 뒤 재설치. 클린 인스톨이 가장 빠른 회복책일 때가 많았다. (지금이라면 lockfile 믿고 `npm ci`를 썼겠지만, 그땐 거기까지 몰랐다.)

## 회고

이 프로젝트의 진짜 결과물은 코드가 아니라 README였던 것 같다.

기능 자체는 Gatsby가 깔아둔 길을 따라간 거다. 어려운 건 시스템을 비개발자 손에 쥐여주는 일이었고, 그건 코드가 아니라 문서로 풀렸다. 내가 옆에 없어도 디자이너가 글을 올릴 수 있게 만드는 것. 입사 초에 한 일치고는, "내가 빠져도 굴러가게 만든다"는 감각을 일찍 배운 셈이다.
