공공기관 보안 포털을 두 명이 만들었다. 도메인은 100개가 넘었다.

산하 기관의 자산·취약점·침해사고·악성 IP·모의훈련을 한곳에서 관리하는 관제센터용 시스템이었다. 자산 목록, 취약점 진단 결과, 블랙리스트, 통보 이력... 화면마다 도메인은 다른데 생긴 건 다 똑같다. 목록 보여주고, 검색하고, 정렬하고, 엑셀로 내리고, 추가/수정/삭제 모달 띄우고. 이걸 100번 손으로 짤 수는 없었다.

## 문제

같은 모양 화면을 100개 만든다는 건, 같은 버그를 100군데 심는다는 뜻이기도 하다.

처음엔 화면을 하나씩 만들었다. 표 그리고, 검색 폼 붙이고, 엑셀 버튼 달고, 모달 폼 짜고. 두 번째 화면도 거의 같은 코드. 세 번째도. 이쯤 되니 보였다. 이 화면들이 다른 건 결국 "어떤 컬럼을 보여주느냐"뿐이라는 거. 제목 셀이냐 날짜 셀이냐, 검색은 텍스트냐 기간이냐, 엑셀에 넣을 거냐 말 거냐.

그러면 컬럼 정의만 데이터로 빼고 나머지는 한 컴포넌트가 다 처리하게 하면 되지 않나. 문제는 컬럼 하나가 다섯 군데서 서로 다르게 해석돼야 한다는 거였다. 같은 `status` 컬럼이 표에선 색깔 뱃지로 보이고, 검색에선 select가 되고, 엑셀에선 코드값을 한글 라벨로 풀고, 폼에선 입력칸이 되고, 어떤 화면에선 아예 폼에서만 숨긴다. 하나의 정의, 다섯 개의 얼굴.

## 트레이드오프 결정

추상화를 키우면 누수도 같이 큰다. 그걸 알면서 키웠다.

선택지는 둘이었다. 화면마다 컴포넌트를 따로 두고 손이 많이 가도 명시적으로 가느냐, 아니면 컬럼 메타 하나로 다 찍어내고 추상화의 누수를 감수하느냐. 두 명이서 도메인 100개를 감당해야 하는 상황에선 답이 정해져 있었다. 후자.

대신 누수는 각오했다. 컬럼 메타가 다섯 곳에서 분기되니까, 표에선 맞는데 엑셀 enum에서만 깨지는 케이스가 반드시 나온다. 실제로 그랬고, 코드에 `//TODO : valueEnum 문제로 인한 임시 방편` 같은 주석을 남겨가며 우회로를 뚫었다. 추상화의 힘과 누수가 한 몸이라는 걸 이 프로젝트에서 제대로 배웠다. 그래도 안 했으면 화면 100개를 직접 짰을 거다. 그게 더 나쁘다.

## 접근

컬럼 객체 하나에 플래그를 달아 다섯 갈래로 흘려보냈다. `searchDefine`이면 검색 입력, `export: false`면 엑셀에서 제외, `excelValueEnum`이면 엑셀에서만 다른 라벨, `editable: false`면 폼에서 disabled. 표 한 열로 시작한 정의가 검색·정렬·엑셀·모달폼까지 전부 만든다. 신규 화면은 "컬럼 배열 작성 + GraphQL 쿼리 연결" 수준으로 줄었다. 이 Table 컴포넌트를 결국 `pages/` 아래 59개 파일이 import했다.

추가/수정 모달은 표에 쓴 그 컬럼 배열을 그대로 폼으로 넘긴다. 같은 정의니까 표랑 폼이 어긋날 일이 없다. 수정할 땐 바뀐 필드만 골라 보냈다. 초기값과 다른 키만 추려서. 안 그러면 안 건드린 필드까지 통째로 덮어쓴다.

```jsx
// RecordModal.jsx — 수정은 diff만
if (isUpdate) {
  R.keys(value).forEach((key) => {
    if (value[key] !== initialValues[key]) valueForUpdate[key] = value[key];
  });
}
```

### 권한을 컴포넌트가 스스로 판단하게

추가·수정·삭제 버튼은 권한이 있을 때만 그려야 했다. 기관마다 "이 메뉴는 조회만, 저 메뉴는 수정까지"가 다 다르니까. 권한 목록을 따로 테이블로 관리하면 메뉴 바뀔 때마다 어긋난다. 메뉴는 있는데 권한 키가 없는 상태. 그래서 라우트를 권한의 원천으로 삼았다.

라우트 정의 한 벌에서 메뉴 트리도 나오고, 권한 설정 트리도 나오고, 런타임 권한 체크 키도 나온다. 권한 키는 그냥 `경로/액션`이다. `/policy/blacklist/UPDATE` 같은 식. Table은 현재 경로로 `hasPermission('${path}/CREATE')`만 물어보면 버튼을 그릴지 말지 안다.

```jsx
// Table/index.jsx — 버튼은 권한이 있을 때만
const permissionCreate = user.hasPermission(`${path || pathname}/CREATE`);
```

메뉴를 추가하면 권한이 자동으로 따라 생긴다. 라우트 객체에 한 줄 넣는 게 전부다. 권한 정의를 코드 바깥에 둘 때 늘 생기는 불일치를 구조로 막았다.

## 결과

같은 추상화를 쓴다는 건, 한 번 잡은 버그가 59개 화면에서 한꺼번에 고쳐진다는 뜻이기도 했다. 문제는 그 버그들이 추상화 깊은 곳에 숨어 있었다는 거고.

제일 끈질겼던 건 페이지 리셋이었다. 3페이지를 보다가 검색하면 결과가 비어 보였다. 검색 결과는 한 페이지뿐인데 화면은 여전히 3페이지 offset을 서버에 요청하니까. 검색·정렬할 때 페이지를 1로 안 돌려놓으면 직전 페이지 번호가 살아남아 엉뚱한 `start`로 쿼리가 나간다. 같은 버그가 한 화면에서 검색 핸들러, 필터 핸들러, 정렬 핸들러마다 따로 재발했다. 핸들러를 새로 달 때마다 사람이 리셋을 빠뜨린 거다. 결국 검색·필터·정렬을 바꾸면 무조건 1페이지로 가도록 추상화 한 곳에 강제했다. 핸들러가 개별로 기억해야 하는 계약을 없애고, 입력이 바뀌는 모든 경로가 같은 리셋을 통과하게 묶었다. 빠뜨리기 쉬운 계약일수록 사람한테 맡기면 안 된다. 페이지네이션 다중선택도 같은 결의 버그였다 — 전체선택 후 일부 해제하면 타 페이지 선택분이 날아가던 걸, "현재 페이지 이벤트"와 "전역 선택 상태"를 분리해 훅 한 곳에서 보정했다.

보안 점검은 기능 개발만큼 반복되는 상수였다. 정부 포털이라 응답에서 새는 정보를 계속 틀어막아야 했다. nginx 버전과 백엔드 `X-Powered-By`가 응답 헤더로 새는 건 `server_tokens off`만으론 부족해서, 프록시 환경에선 업스트림 헤더를 nginx에서 명시적으로 죽였다(`proxy_hide_header X-Powered-By`). GraphQL이 내부 DB id를 그대로 직렬화해 내보내는 것도 막아야 했는데, 응답 스키마와 저장 스키마를 분리하고 통계는 ORM N+1 대신 knex raw 서브쿼리 한 방으로 묶어 내부 id 노출과 쿼리 폭증을 동시에 잡았다.

화면 59개, 도메인 100여 개를 두 명이 굴렸다. [확인: 페이지 수·도메인 수 정확 수치]

## 회고

이 프로젝트는 처음부터 끝까지 "정의를 어디 한 군데로 모으느냐" 싸움이었다.

화면의 정의는 컬럼 배열로 모았고, 권한의 정의는 라우트로 모았고, 페이지네이션 리셋은 훅으로 모았다. 흩어진 진실을 한 곳으로 끌어모으면 버그도 한 곳에서 잡힌다. 대신 그 한 곳이 무너지면 59개가 같이 무너진다. 그 무게를 견디는 게 추상화를 만드는 사람의 몫이라는 걸 여기서 배웠다.
