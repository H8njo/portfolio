공공기관 보안 포털을 두 명이 만들었다. 산하 기관의 자산·취약점·침해사고·악성 IP·모의훈련을 한곳에서 관리하는 관제센터용 시스템이었고, 도메인은 100개가 넘었다.

화면은 이름만 다르지 골격이 다 똑같았다. 목록 보여주고, 검색하고, 정렬하고, 엑셀로 내리고, 추가·수정·삭제 모달 띄우고. 도메인 100개를 두 명이 감당하려면 이 골격을 화면마다 다시 짜선 안 됐다. 그래서 화면의 다른 점을 딱 하나로 줄였다 — **어떤 컬럼을 보여주느냐.** 나머지 전부는 **테이블 컴포넌트 하나**가 처리하게 했다. `pages/` 아래 59개 파일이 결국 이 하나를 import했다.

## 화면마다 필요한 게 다 달랐다

문제는 "똑같은 표"가 사실 하나도 안 똑같았다는 거다. 화면을 모아 놓고 보니 필요한 기능이 이만큼이었다.

- **목록 표시** — 타입별 셀 렌더(날짜/숫자/코드값 뱃지), 말줄임+툴팁, 행 번호
- **검색** — 화면에 따라 텍스트 한 칸부터 기간 + select 다섯 개까지. 접었다 펴기, 조건 딥링크
- **정렬** — 컬럼별 오름/내림차순, 정렬 제외 컬럼
- **엑셀** — 내보내기(코드값→한글 라벨 변환), 가져오기(행 단위 등록)
- **CRUD 모달** — 컬럼 정의로 자동 생성되는 추가/수정 폼, 다열 레이아웃
- **행 선택 / 일괄 작업** — 페이지 넘어가는 선택 유지, 일괄 삭제 + 커스텀 일괄 액션
- **컬럼 조작** — 표시/숨김 토글, 순서 드래그 변경, 너비 리사이즈, 그 설정의 화면별 저장
- **인라인 편집** — 목록 위에서 셀 값만 바로 수정·저장
- **권한** — 기관마다 다른 조회/수정 범위에 따라 버튼 노출 제어

한 컴포넌트가 이걸 다 하되, 화면마다 **필요한 것만 골라 켜지게** 해야 했다. 안 그러면 "이 화면만 좀 달라요"가 나올 때마다 컴포넌트 밖에 특수 코드가 쌓이고, 결국 복사 붙여넣기 지옥으로 돌아간다.

## 설계 축: 하나의 정의, 다섯 개의 얼굴

핵심 통찰은 이거였다.

> 컬럼 하나가 다섯 군데에서 서로 다르게 해석돼야 한다.

같은 `status` 컬럼이 **표**에선 색깔 뱃지로, **검색**에선 select로, **엑셀**에선 코드값을 한글 라벨로 풀어서, **폼**에선 입력칸으로 보이고, 어떤 화면에선 아예 **폼에서만 숨는다.** 그래서 컬럼 객체 하나에 플래그를 달아 다섯 갈래로 흘려보냈다. 정의는 한 벌, 해석은 다섯 갈래.

```jsx
// 컬럼 한 벌이 표·검색·엑셀·폼을 전부 만든다
{
  dataIndex: 'status',
  title: '상태',
  render: (v) => <StatusBadge value={v} />,   // 표: 뱃지
  searchDefine: { type: 'select', options },  // 검색: select
  excelValueEnum: STATUS_LABELS,              // 엑셀: 코드→한글
  editable: false,                            // 폼: 수정 시 잠금
  hideSorting: true,                          // 정렬: 제외
}
```

추가·수정 모달도 표에 쓴 그 배열을 그대로 폼으로 넘긴다. 같은 정의니까 표와 폼이 어긋날 일이 없다. 수정할 땐 바뀐 필드만 — 초기값과 다른 키만 추려 보냈다. 안 그러면 안 건드린 필드까지 통째로 덮어쓰니까.

```jsx
// 수정은 diff만
if (isUpdate) {
  R.keys(value).forEach((key) => {
    if (value[key] !== initialValues[key]) valueForUpdate[key] = value[key];
  });
}
```

신규 화면은 여기까지 오면 "컬럼 배열 작성 + 쿼리 연결"로 끝난다.

## 화면 하나는 이만큼이면 끝난다

악성 IP 블랙리스트 화면 하나를 통째로 옮기면 이렇다. 컬럼 배열이 표·검색·엑셀·폼을 다 만들고, `<Table>`은 데이터와 CRUD 함수만 받는다.

```jsx
// pages/policy/blacklist.jsx — 이 파일이 화면의 전부다

const columns = [
  { dataIndex: 'ip', title: 'IP 주소',
    searchDefine: { type: 'text' } },              // 표 + 검색(텍스트)

  { dataIndex: 'status', title: '상태',
    render: (v) => <StatusBadge value={v} />,      // 표: 색깔 뱃지
    searchDefine: { type: 'select', options },     // 검색: select
    excelValueEnum: STATUS_LABELS },               // 엑셀: 코드→한글

  { dataIndex: 'reason', title: '차단 사유' },       // 표 + 폼(기본 입력칸)

  { dataIndex: 'blockedAt', title: '차단일시',
    valueType: 'dateTime',                         // 표: 날짜 포맷
    searchDefine: { type: 'dateRange' },           // 검색: 기간 선택
    editable: false },                             // 폼: 수정 시 잠금
];

export default function BlacklistPage() {
  const { data, loading, refetch } = useBlacklist();  // 쿼리만 연결

  return (
    <Table
      tableItemName="블랙리스트"
      columns={columns}
      dataSource={data}
      loading={loading}
      refetch={refetch}
      onAdd={(v) => createBlacklist(v)}            // CRUD 주입
      onEdit={(id, v) => updateBlacklist(id, v)}
      onDelete={(id) => deleteBlacklist(id)}
    />
  );
}
```

이게 화면 파일 전부다. 검색 폼, 정렬, 추가·수정·삭제 모달, 페이지네이션, 행 선택은 한 줄도 안 짰지만 다 나온다 — 컬럼 배열에 이미 다 적혀 있으니까. 추가·수정·삭제 버튼은 경로(`/policy/blacklist`)에서 권한을 물어 스스로 나타나고 사라진다.

엑셀도 마찬가지다. **다운로드**는 컬럼과 `dataSource`가 있으니 자동으로 뜬다 — 컬럼의 `title`을 헤더로, `excelValueEnum`으로 코드값을 한글로 풀어서 `블랙리스트_목록.xlsx`를 내린다. **업로드**는 이미 연결한 `onAdd`에 그대로 물린다. 사용자가 올린 시트의 각 행이 `onAdd`로 흘러 들어가 그대로 등록된다. 화면은 파싱도 매핑도 신경 쓸 게 없다.

```jsx
// 엑셀 커스터마이징이 필요할 때만 한 줄 (기본은 자동)
<Table
  columns={columns}
  dataSource={data}
  onAdd={(v) => createBlacklist(v)}                    // 업로드가 이 함수로 흘러든다
  excelExport={{ fileName: '악성IP_현황' }}              // 다운로드 파일명 override
  excelImport={{ template: BLACKLIST_TEMPLATE }}        // 업로드 양식 지정
/>
```

화면 59개가 이 모양으로 생겼다.

## 조립식으로 확장한다

"필요한 것만 골라 켠다"를 세 층의 조립 지점으로 나눴다. 화면은 이 셋을 조합해 자기 모양을 만든다.

**1. 컬럼 플래그 — 선언으로 켜고 끈다.** 컬럼 객체에 플래그만 얹으면 동작이 바뀐다. `searchDefine`이면 검색 입력으로, `export: false`면 엑셀에서 제외, `excelValueEnum`이면 엑셀에서만 다른 라벨, `editable: false`면 폼에서 잠금, `table: false`면 표에선 숨기고 폼에만, `hideSorting`이면 정렬 후보에서 제외. 화면 하나가 표 30컬럼짜리든 검색 한 칸짜리든, 코드가 아니라 데이터로 표현된다.

**2. 슬롯 — 자리를 열어두고 끼워 넣는다.** 툴바에 이 화면만의 버튼이 필요하면 `customToolbar`로 끼우고, 행 오른쪽에 "상세보기·재발송" 같은 액션이 필요하면 액션 컬럼 렌더러로 넣는다. 검색 폼·툴바·헤더·선택 알림줄이 전부 열린 슬롯이라, 컴포넌트를 건드리지 않고 화면에서 채운다.

**3. 훅 — 흐름을 가로챈다.** 기본 동작으로 안 되는 화면을 위해 가로채기 지점을 뒀다. `beforeAdd`·`beforeEdit`로 모달을 열기 전에 끼어들고, `checkValidation`으로 제출을 막고, `allowEdit(record)`로 특정 행만 수정 불가로 만들고, `handleMultipleRows.addon`으로 일괄 작업 메뉴에 임의 액션을 더한다. CRUD 자체는 `onAdd`·`onEdit`·`onDelete`로 주입하니 GraphQL이든 REST든 상관없다.

앞의 블랙리스트 화면에 이 화면만의 요구가 붙어도, 확장은 한 줄씩이다.

```jsx
<Table
  columns={columns}
  dataSource={data}
  // ...
  {...useColumnsState()}                               // 프리셋: 컬럼 설정 저장
  allowEdit={(row) => row.locked}                       // 훅: 잠긴 행은 수정 불가
  customToolbar={() => <SyncFromThreatFeed />}          // 슬롯: 이 화면만의 버튼
  handleMultipleRows={{ addon: [releaseAction] }}       // 훅: 일괄 "차단 해제" 추가
/>
```

자주 쓰는 조합은 프리셋으로 굳혔다. 읽기 전용 게시판은 `CommonTable`이 추가·수정·삭제를 다 꺼둔 채 데이터만 물어오고, `useColumnsState`를 끼우면 표시/숨김·순서·너비 설정이 **화면 경로별로 localStorage에 남아** 새로고침해도 유지된다.

## 그래서 모든 케이스가 흡수됐다

이 조립 구조 덕에, "이 화면만 달라요"가 컴포넌트 밖으로 새지 않았다.

- 컬럼 30개짜리 관제 화면 → 컬럼 표시 토글 + 순서 드래그 + 너비 리사이즈로 사용자가 직접 화면을 맞추고, 그 설정은 경로별로 저장된다.
- 검색이 복잡한 화면 → 텍스트·숫자·기간·select를 컬럼이 직접 고르고, 조건을 URL 쿼리에 실어 "특정 조건 3페이지"를 그대로 링크로 공유한다.
- 모달이 과한 화면 → 인라인 편집 셀로 목록 위에서 값 하나만 바꿔 저장한다.
- 권한이 화면마다 다른 상황 → 아래 방식으로 버튼이 스스로 나타나고 사라진다.

새 요구가 오면 컴포넌트에 플래그·슬롯·훅 중 맞는 확장점을 하나 더 열었다. 화면 쪽 코드는 늘 컬럼 배열 수준으로 얇게 유지됐다.

## 권한을 컴포넌트가 스스로 판단하게

기관마다 "이 메뉴는 조회만, 저 메뉴는 수정까지"가 달랐다. 권한 목록을 코드 바깥 테이블로 관리하면 메뉴가 바뀔 때마다 어긋난다 — 메뉴는 있는데 권한 키가 없는 상태. 그래서 **라우트를 권한의 원천**으로 삼았다.

라우트 정의 한 벌에서 메뉴 트리도, 권한 설정 트리도, 런타임 체크 키도 나온다. 권한 키는 그냥 `경로/액션`이다. Table은 현재 경로로 물어보기만 하면 버튼을 그릴지 안다.

```jsx
// 버튼은 권한이 있을 때만 그린다
const permissionCreate = user.hasPermission(`${path || pathname}/CREATE`);
```

메뉴를 추가하면 권한이 자동으로 따라 생긴다. 라우트 객체에 한 줄 넣는 게 전부다.

## 계약은 사람이 아니라 구조가 지키게

추상화 한 곳에 몰아넣은 만큼, 계약도 한 곳에서 지켜야 했다.

페이지네이션이 대표적이었다. 검색·정렬·필터를 바꿀 때 페이지를 1로 안 돌려놓으면, 3페이지를 보다 검색했을 때 직전 offset이 살아남아 엉뚱한 `start`로 쿼리가 나가 결과가 비어 보인다. 핸들러마다 "여기도 리셋"을 사람이 챙기면 반드시 빠뜨린다. 그래서 **입력이 바뀌는 모든 경로가 무조건 1페이지를 통과하도록** 한 곳에 강제했다. 페이지 넘어가는 다중 선택도 마찬가지 — "현재 페이지 이벤트"와 "전역 선택 상태"를 분리해 훅 한 곳에서 병합·차감하게 묶었다. 빠뜨리기 쉬운 계약일수록 컴포넌트가 대신 지킨다.

## 결과

화면 59개, 도메인 100여 개를 두 명이 굴렸다. 새 CRUD 화면 하나가 "컬럼 배열 작성 + 쿼리 연결"로 끝나니, 화면을 찍어내는 속도가 확 빨라졌다. 한 번 잡은 버그는 59개 화면에서 한꺼번에 고쳐졌다.

그리고 이 테이블은 이 포털 안에서 끝나지 않았다. 검색·정렬·엑셀·모달폼이 붙는 리스트/디테일 화면은 어디든 컬럼만 다를 뿐 골격이 똑같으니까, 이후 다른 프로젝트들에서도 컬럼 배열만 갈아끼워 그대로 가져다 썼다. 확장점을 플래그·슬롯·훅으로 열어둔 덕에, 새 프로젝트의 낯선 요구도 컴포넌트를 다시 만들지 않고 확장점 하나를 더 여는 걸로 흡수됐다.

정의를 한곳에 모으면 값이 한곳에서 나온다. 화면의 정의는 컬럼 배열로, 권한의 정의는 라우트로, 흐름의 계약은 훅으로 모았다. 흩어질 진실을 한곳에 묶고, 다양성은 조립 지점으로 열어두는 것 — 재사용성과 확장성은 그 둘을 동시에 잡을 때 나왔다.

## 회고 — 이 테이블은 책임이 너무 많다

솔직히 말하면, 이 `Table`은 God component다. 렌더링, 검색, 정렬, 페이지네이션, 엑셀 입출력, CRUD 모달, 권한 판정, 행 선택, 컬럼 상태 저장을 한 컴포넌트가 다 들고 있다. prop이 40개를 넘고, "테이블"이라는 이름 하나로 부를 수 없는 일을 한다. 두 명이 도메인 100개를 쳐내야 하는 상황에선 이게 가장 빠른 답이었지만, 빠른 것과 옳은 것은 다르다.

증상은 두 군데서 드러났다. 하나는 **prop 폭발** — 화면이 기능을 "골라 켜는" 게 아니라, 안 쓰는 기능까지 전부 들고 있는 하나의 거대한 인터페이스에서 필요한 것만 채우는 꼴이 됐다. 다른 하나는 **컬럼 플래그의 런타임 누수** — 컬럼 하나가 표·검색·엑셀·폼에서 다르게 해석되는데, 그 분기가 전부 한 컴포넌트 안에 숨어 있어 "표에선 멀쩡한데 엑셀에서만 깨지는" 걸 타입이 못 잡고 런타임에 가서야 알았다.

다시 만든다면, **오케스트레이션과 관심사를 분리**하는 쪽으로 갈 것 같다.

**1. 헤드리스 코어 — 상태는 훅으로, 그림은 밖으로.** 쿼리 파라미터·페이지네이션·선택 상태 같은 "머리"를 `useTable()` 훅으로 내리고, `Table`은 그 상태를 그리기만 하게 한다. 지금 컴포넌트 안에 박아둔 페이지 리셋·전역 선택 계약은 그대로 훅 안에 살아 있으니, 로직은 잃지 않고 렌더링에서만 떼어낸다.

```jsx
// 지금: Table이 상태도 렌더도 CRUD도 권한도 다 안다
<Table columns={...} onAdd={...} sorting excelImport={...} /* +30 props */ />

// 다시 만든다면: 상태는 훅, 화면은 필요한 조각만 조립
const table = useTable({ columns, query });
<TableView {...table}>
  <TableView.Search />       {/* 검색 필요한 화면만 */}
  <TableView.Excel />        {/* 엑셀 필요한 화면만 */}
  <TableView.Pagination />
</TableView>
```

**2. 관심사를 컴파운드 컴포넌트로.** 검색·엑셀·툴바·페이지네이션을 `Table`의 prop이 아니라 **자식 컴포넌트**로 만든다. 안 쓰면 안 붙인다. prop 40개짜리 하나가 아니라, 필요한 조각만 트리에 얹는 진짜 조립식이 된다. (성숙한 버전에서 실제로 이 방향의 컴파운드 구조를 부분적으로 썼는데, 다음엔 이걸 기본값으로 삼겠다.)

**3. CRUD 모달과 권한은 테이블 밖으로.** 폼 생성과 권한 판정은 "표를 그리는 일"이 아니다. CRUD는 컬럼 스키마를 공유하는 별도 `<RecordForm>`으로, 권한은 `Table`이 직접 `hasPermission`을 부르는 대신 상위 경계(라우트 가드/`<Can>` 래퍼)에서 결정해 액션만 내려주게 한다. 테이블은 "그릴지 말지"만 받으면 된다.

**4. 컬럼 스키마를 표면별로 타입 분리.** "하나의 정의, 다섯 개의 얼굴"은 유지하되, 그 정의가 표·검색·엑셀·폼에서 각각 어떤 타입으로 해석되는지를 어댑터로 갈라놓는다. 그러면 엑셀 enum 누수 같은 게 주석(`// TODO`)이 아니라 **컴파일 에러**로 먼저 드러난다. 추상화를 줄이는 게 아니라, 새는 자리를 눈에 보이게 만드는 쪽으로.

요지는 이렇다. 이 프로젝트에서 "정의를 한곳에 모은다"는 옳았다. 틀린 건 **그리는 책임까지 같은 곳에 모은 것**이다. 다음엔 상태와 계약은 한곳에 모으되, 렌더링과 관심사는 조립하는 사람 손에 돌려주겠다.
