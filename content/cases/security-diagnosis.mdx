폐쇄망 보안 진단 시스템에서 일을 시작하고 제일 먼저 체감한 건, 여기선 모든 게 엑셀로 오간다는 거였다. 진단 항목도, 결과도, 통계도. 담당자들은 화면에서 직접 입력하기보다 엑셀로 받아서 엑셀로 다시 올렸다. 그래서 엑셀 입출력은 부가 기능이 아니라 사실상 이 시스템의 본업이었다.

이게 내 초기 커리어 때 일이다. 그땐 "엑셀 다운로드, 그냥 데이터 받아서 파일로 떨구면 되는 거 아닌가" 했다. 근데 양방향으로 제대로 만들어 보니 입력 쪽이 훨씬 더럽더라. 엑셀은 사람이 보는 값이랑 안에 들고 있는 값이 다르니까.

## 문제

다운로드는 그럭저럭이었다. antd Table에 이미 `columns` 정의가 다 있으니, 그걸 그대로 받아서 화면에 보이는 값으로 변환해 `xlsx`로 내보내면 됐다. 근데 문제는 코드값이었다. 화면엔 "조치완료"라고 보여도 DB엔 코드 id로 저장돼 있어서, 엑셀에 그 코드가 그대로 찍히면 사람이 못 읽는다. 그래서 `columns`의 `editor`(select/text), `render` 함수를 보고 변환기를 동적으로 만들어, 화면에 보이는 값과 엑셀 값을 일치시켰다. select 컬럼은 `options`에서 `value → text`를 역매핑하고.

진짜 골치는 업로드였다. 여기서 세 가지가 따로 터졌다.

하나, **대량 POST가 서버를 눕혔다.** 담당자가 엑셀 한 장에 수천 건을 담아 올리는 일이 흔했는데, 그걸 한 방에 POST하면 서버가 못 버텼다.

둘, **올릴 때마다 데이터가 불어났다.** 진단 결과 회신 화면에서 같은 엑셀을 다시 올리면, 기존 레코드가 갱신되는 게 아니라 매번 새 레코드로 또 생겼다. 두 번 올리면 두 배가 됐다.

셋, **날짜가 날짜가 아니었다.** 진단 마법사에서 팀·장비 엑셀을 올리면 시작일·종료일이 `44179` 같은 정수로 그리드에 들어왔다. 화면에선 분명 `2020-12-14`였는데. 그게 그대로 DB에 박혔다.

## 트레이드오프 결정

대량 POST부터. 가장 단순한 건 그냥 서버 타임아웃을 늘리는 거였다. 근데 그건 미루는 거지 푸는 게 아니다. 건수가 더 늘면 또 터질 거고. 그래서 클라이언트에서 `limit=50`씩 잘라 순차로 보내기로 했다.


```js
const limit = 50;
const total = dataWillCreateToDB.length;
const result = [];
for (let current = 0; current <= total; current += limit) {
  setUploadProgress({ percent: Math.round((current / total) * 100) });
  const dividedArray = dataWillCreateToDB.slice(current, current + limit);
  result.push(await diagApi.post('/diag-results/reply', dividedArray));
}
setUploadProgress({ percent: 100 });
```

`await`로 한 묶음씩 끝내고 다음을 보내니 서버에 한 번에 실리는 양이 50건으로 고정됐다. 덤으로 진행률이 생겼다. 수천 건 올라가는 동안 화면이 멈춘 것처럼 보이면 담당자가 불안해서 새로고침을 누르고, 그러면 또 사달이 난다. 청크로 자르면서 `setUploadProgress`로 퍼센트를 같이 그려 주니 그 불안이 사라졌다. 성능 때문에 자른 건데 UX가 따라온 셈이다.

병렬로 한 번에 다 던지면 더 빠르지 않냐고 할 수 있다. 맞다. 근데 그럼 서버를 눕히는 원래 문제로 돌아간다. 여기선 "조금 느리지만 안 죽는" 쪽을 골랐다. 폐쇄망 내부 업무 도구라 초당 처리량보다 안 죽는 게 훨씬 중요했고.

## 접근

중복 생성은 원인을 알고 나니 좀 허무했다. 회신(reply) 업로드인데, 신규 생성용 엔드포인트(`/upload`)로 POST하고 있었다. 서버는 그 엔드포인트를 무조건 "새로 만들기"로 처리하니까, 이미 있는 결과인지 아닌지 따지질 않고 같은 걸 올려도 계속 insert가 됐던 거다.

```diff
- result.push(await diagApi.post('/diag-results/upload', dividedArray));
+ result.push(await diagApi.post('/diag-results/reply', dividedArray));
```

고친 건 한 줄이다. 근데 여기서 배운 건 한 줄짜리가 아니었다. 클라이언트에선 둘 다 똑같은 "엑셀 올리기" 버튼이다. 생긴 게 같다. 그런데 서버에선 "새로 만들기"랑 "있는 거 갱신하기"는 완전히 다른 연산이다. 생긴 게 같다고 같은 게 아니다. 그 뒤로 같은 입력 UI라도 의도(create / upsert / reply)에 맞는 엔드포인트로 보내고 있는지부터 본다. 일반 결과랑 회신 결과의 업로드 경로를 아예 분리하는 작업도 같이 했다.

날짜가 제일 재밌었다. `44179`. 처음엔 이게 대체 뭔가 했는데, 알고 보니 엑셀이 날짜를 1900-01-01 기준 일수, 즉 시리얼 정수로 저장하더라. 셀을 raw로 읽으면 사람이 보던 날짜가 아니라 그 정수가 그냥 따라온다. 화면에서 멀쩡해 보였던 건 엑셀이 표시할 때만 날짜로 포맷해 줬던 거고, 안에 든 진짜 값은 줄곧 숫자였다.

그래서 그리드 모달에 시리얼 → 날짜 보정 함수를 넣었다.

```js
const getFormatDate = excelDate => {
  let date = new Date((excelDate - 25569) * 86400 * 1000);
  let year = date.getFullYear();
  let month = 1 + date.getMonth();
  month = month >= 10 ? month : '0' + month;
  let day = date.getDate();
  day = day >= 10 ? day : '0' + day;
  return `${year}-${month}-${day}`;
};

if (data.startDate || data.endDate) {
  data.startDate = getFormatDate(data.startDate);
  data.endDate = getFormatDate(data.endDate);
}
```

`25569`는 그냥 마법의 숫자가 아니라 엑셀 epoch(1900-01-01)랑 Unix epoch(1970-01-01) 사이의 일수 차이다. 엑셀의 그 악명 높은 1900년 윤년 버그까지 포함한 표준 보정값이고. 거기에 `86400 * 1000`을 곱해 밀리초로 만들면 JS `Date`가 받아먹는다. 나중에 알았지만 `xlsx`를 읽을 때 `cellDates: true`로 파싱 단계에서 아예 Date로 받는 방법도 있었다. 그땐 들어온 정수를 거꾸로 환산하는 쪽으로 풀었다.

## 결과

- 대량 업로드: 한 번에 던지던 걸 50건 청크 + 진행률로. 서버에 실리는 양이 묶음 크기로 고정돼서, 건수가 늘어도 안 죽게 됐다.
- 중복 누적: 엔드포인트를 `reply`로 바로잡고 일반/회신 경로를 분리. 같은 엑셀을 다시 올려도 데이터가 불어나지 않는다.
- 날짜: 시리얼 정수로 박히던 시작일·종료일이 정상 날짜로 들어간다.
- 엑셀 다운로드는 처음 한 화면에만 있던 걸, `columns`에 `render`가 있으면 그걸 적용한다는 공통 규약을 세워 진단·해킹·통계·네트워크 전 페이지로 넓혔다.

수치로 크게 자랑할 만한 일은 아니다. 폐쇄망 안에서 담당자들이 매일 쓰는 도구가 안 죽고, 데이터가 안 불어나고, 날짜가 날짜로 들어가게 만든 것. 그게 결과다.

## 회고

여기서 걸린 것들은 결국 "보이는 값이랑 진짜 값은 다르다"는 한 가지였다. 화면의 "조치완료"는 DB엔 코드 id고, 화면의 `2020-12-14`는 엑셀 안에선 `44179`고, 화면의 "엑셀 올리기" 버튼 하나는 서버에선 create랑 reply 두 개고. 전부 같은 함정이었다.

특히 엔드포인트 건은 오래 남았다. 클라이언트에서 똑같이 생긴 두 동작이 서버에선 전혀 다른 의미일 수 있다는 거. 이런 게 제일 짜증난다. 생긴 게 같다고 같은 게 아니니까. (한 번 당하고 막았다.)

엑셀 다루는 습관도 그때 붙었다. 스프레드시트 날짜는 일단 시리얼 정수일 거라고 의심하기. 대량 전송은 청크로 잘라 peak를 잡고 진행률을 같이 보여 주기. 같은 입력 UI라도 어떤 엔드포인트로 가는지 확인하기. 초기 커리어에 데이터 집약형 관리자 시스템을 제대로 한번 밟아 본 게, 지금도 엑셀이나 대량 데이터만 보면 반사적으로 이 세 개를 먼저 떠올리게 만든다.
