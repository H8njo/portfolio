위저드로 빨리 출시했다가, 회사 공통 결제에 합치려고 통째로 갈아엎었다.

교육 콘텐츠 마켓의 주문제작 도서 기능이었다. 학원 강사가 자기가 만든 교재 PDF를 올리면, 내지 컬러랑 제본 방식, 표지, 수량을 골라서 실물로 제본·배송받는 거다. 가격이 `페이지 수 × 컬러 단가 + 제본 단가`라, 올리자마자 페이지 수부터 알아야 가격이 떴다. 그건 서버 왕복 없이 `pdf.js`로 클라이언트에서 직접 셌고(손상 파일 검증도 겸했다), 파일은 S3가 발급한 임시 URL로 브라우저가 직접 올렸다. 서버를 파일 중계기로 쓰지 않았다.

## 트레이드오프 결정

제일 큰 결정은 폼 구조였다. 처음엔 3-step 위저드로 띄웠다. 제본신청 → 배송지 → 결제, 한 흐름. 빨리 내야 했고 위저드가 빨랐다. POD 상태는 페이지를 벗어나면 다 버려져야 하는 페이지-스코프 상태라, 전역 store 대신 Context로 들었다(언마운트되면 알아서 비워지니까).

근데 문제는 결제였다. 회사엔 이미 장바구니·쿠폰·포인트가 붙은 공통 결제 인프라가 있었는데, 위저드가 결제까지 자기 흐름으로 끌고 가버려서 그걸 못 썼다. 합치려면 POD를 "장바구니에 담기는 상품"으로 일반화하는 게 맞았다. 그래서 결국 통째로 갈아엎었다. 결제 스텝을 떼서 공통 퍼널에 흘려보내고, POD 페이지는 옵션 입력만 하는 단일 폼이 됐다. `FormProvider` + `useForm`.

위저드를 후회하진 않는다. 그게 있어서 일단 출시가 됐고, 통합이 필요해진 건 그다음 일이었다. 빨리 내는 도구랑 합치는 도구가 달랐을 뿐이다.

## 접근

리팩토링하면서 위저드 시절 손으로 짜둔 게 줄줄이 한 줄로 줄었다. 가장 인상 깊었던 건 에러 포커스였다.

위저드 때는 첫 에러 필드로 점프시키려고 input ref를 네 개 박아놓고 우선순위대로 `focus()`에 `scrollIntoView()`를 돌렸다. 숨겨둔 file input은 포커스가 안 먹어서, 잠깐 `display:block`으로 보였다가 스크롤하고 다시 숨기는 트릭까지 썼다. 그게 RHF에선 `shouldFocusError: true` 한 줄이었다.

```tsx
// Content.tsx
const methods = useForm<PODForm>({
  mode: 'onTouched',
  shouldFocusError: true, // 위저드 시절 ref 4개 + scrollIntoView가 이 한 줄
  defaultValues: { color: '흑백', binding: '무선', quantity: 1, /* ... */ },
})
```

그러다 진짜 짜증나는 버그를 하나 만났다. 비로그인으로 옵션을 다 채워놓고 "다음"에서 로그인하면, 인증 리다이렉트로 페이지가 재마운트되면서 입력이 싹 날아가버렸다. 전부 메모리에만 있었으니 돌아올 길이 없었다. `localStorage`에 동기화해서 복원하게 했는데, 여기에 함정이 하나 더 있었다.

```tsx
const [isLoaded, setIsLoaded] = useState(false)
useEffect(() => {                 // ① 마운트 시 복원
  const saved = localStorage.getItem('pod_draft_settings')
  if (saved) setSettings(JSON.parse(saved))
  setIsLoaded(true)
}, [])
useEffect(() => {                 // ② 변경 시 저장
  if (!isLoaded) return           // 복원 전 빈 값으로 덮어쓰기 방지
  localStorage.setItem('pod_draft_settings', JSON.stringify(settings))
}, [settings, isLoaded])
if (!isLoaded) return null        // 서버 빈값 vs 클라 복원값 hydration mismatch 차단
```

`isLoaded` 가드 하나가 두 개를 동시에 막는다. 복원 effect가 돌기 전에 저장 effect가 빈 값으로 덮어쓰는 것, 그리고 서버가 그린 빈 값이랑 클라 복원값이 달라 나는 hydration 에러. 복원 끝났다는 플래그로 첫 렌더를 막아야 둘 다 안 난다. (한 번 당하고 막았다.)

## 결과

PDF를 올리면 페이지 수가 즉시 잡혀 가격이 실시간으로 갱신되고, 파일은 서버를 안 거치고 S3로 바로 올라간다. 로그인하러 떠났다 와도 입력은 그대로 남는다. 위저드로 출시했다가 단일 폼으로 갈아엎으면서 결제는 공통 인프라에 합쳐졌고, 손으로 짜둔 검증·포커스·상태 관리는 RHF로 넘어가 코드가 한참 줄었다.

## 회고

이 기능은 회사 코드베이스의 축소판이었다. 전체가 레거시 스택에서 신규 스택으로 점진적으로 넘어가는 중이었는데, POD도 똑같이 위저드(Context, 수동 폼)에서 단일 폼(RHF)으로 넘어갔다. 마이그레이션이 왜 필요한지, ref 네 개가 한 줄이 되는 그 대비에서 직접 봤다.

다시 한다면 위저드부터 안 만들었을까? 아니다. 빨리 내야 했던 그때는 위저드가 맞았고, 통합이 필요해진 건 그다음 문제였다. 페이지를 벗어나면 버려질 상태를 전역 store가 아니라 Context로 둔 초기 판단도 지금 봐도 맞다. 다만 이탈에 견디는 폼(localStorage 복원 + `isLoaded` 가드)은 처음부터 깔고 갔을 것 같다. 로그인하러 떠났다 돌아오면 입력이 날아가는 건, 위저드든 단일 폼이든 어차피 겪었을 문제였으니까.
