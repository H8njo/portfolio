"전체 페이지 자동 선택"을 누르면 탭이 그대로 얼었다.

수업 관리 도구의 학습자료 제작 기능이었다. 교사가 교재 PDF를 올리고, 페이지에서 문제 단위로 사각형을 드래그해 잘라낸 다음 시험지나 과제로 등록하는 흐름. 한 문제씩 직접 긋는 것도, 페이지를 자동으로 문제별로 쪼개는 것도, 선택한 모든 페이지에 그 자동 분할을 한 번에 거는 것도 됐다. 그 마지막 "전체 페이지" 버튼이 문제였다. 페이지 몇 장짜리 PDF면 잘 돌았는데, 수십·수백 페이지짜리 교재를 넣고 누르면 메모리가 치솟다가 탭이 얼어붙었다.

처음엔 PDF 렌더 라이브러리가 무거운가 했다. 아니었다. 내가 다 끌어안고 있던 거였다.

## 문제

원래 구조는 단순했다. 페이지를 전부 미리 `HTMLImageElement`로 렌더해서 통째로 React state에 들고 있었다. 각 이미지는 `scale = 3 × dpr`로 `canvas.toDataURL()`한 거대한 base64 문자열이다. 한 장도 작지 않은데 그걸 페이지 수만큼.

자동 선택 코드는 거기서 한 술 더 떴다.

```ts
// before — 전체 페이지를 한 번에
const results = await Promise.all(
  pageImages.map((img) => splitRectsByPage(img, ...)),
);
```

모든 페이지의 풀해상도 이미지에다 OpenCV 크롭 결과까지 **동시에** 메모리에 올렸다. peak 메모리가 페이지 수 N에 그대로 비례했으니, 큰 교재에서 안 죽는 게 이상한 거였다.

거기에 UX 구멍이 하나 더 있었다. 페이지 선택 UI가 **이미지가 전부 렌더될 때까지** 막혀 있었다. 수백 장을 다 굽는 동안 교사는 아무것도 못 하고 기다렸다.

## 트레이드오프 결정

메모리부터 잡았다. 전체를 한 번에 돌리는 대신 5페이지씩 청크로 묶어 순차 처리하고, 각 페이지는 렌더한 직후 `page.cleanup()`으로 바로 버렸다. peak가 N이 아니라 청크 크기에 묶이게.

```ts
const imgChunk = chunk(pageImages, CHUNK_LENGTH /* 5 */);
for (const c of imgChunk) {
  const results = await makeRect(c, pageNumberStart); // 렌더 → split → page.cleanup()
  pageNumberStart += c.length;
  setFullPageRectProgress(pageNumberStart); // 원형 진행률
  resultRects.push(...results);
}
```

여기서 한 번 욕심을 부렸다. 청크 재렌더 품질을 `0.2`까지 낮추면 메모리를 더 아낄 수 있겠다 싶었다. 했더니 저해상도라 OpenCV가 문제 경계를 제대로 못 잡았다. 칼선이 한 칸씩 안쪽으로 먹혀 문제 윗줄이 잘리고. 메모리 다이얼이랑 정확도 다이얼이 따로 노는 줄 알았는데, 같은 다이얼의 양 끝이었던 거다. 결국 품질을 `1`로 도로 올렸다. 청크화만으로 탭은 이미 안 죽었으니, 굳이 정확도를 깎을 이유가 없었다.

순차 `for`문도 일부러 그대로 뒀다. 나중에 누가 보면 "이거 `Promise.all`로 병렬화하면 빠른데?" 하고 고치고 싶어질 코드다. 근데 병렬화하는 순간 peak 메모리가 다시 N으로 돌아간다. 청크를 순차로 도는 게 이 코드의 핵심이라 주석으로 박아뒀다.

## 접근

다음은 그 "전부 로드될 때까지 막는" gate였다.

원래 자동 선택은 배경에서 미리 채워둔 `pageImages` 배열을 그대로 썼는데, 배경 렌더가 아직 진행 중이면 배열 중간에 `undefined` 구멍이 났다. 그 상태로 누르면 결과가 부분적으로 틀어졌다.

처음엔 제일 쉬운 방어를 골랐다. 전부 로드되기 전엔 못 누르게.

```ts
// 1차 — gate
const isImageAllLoaded = pageImageLength === loadedImages.length;
// 체크박스 disabled, 누르면 "이미지가 모두 로드 된 후 다시 시도하세요."
```

동작은 했다. 근데 이게 결국 사용자한테 "내가 일 끝낼 때까지 기다려"라고 떠넘기는 거였다. 수백 장짜리 교재에선 그 대기가 길었다.

다시 보니 막을 일이 아니었다. 구멍 난 페이지는 그 시점에 즉석으로 렌더하면 그만이었다. 캐시에 있으면 쓰고, 없으면 just-in-time으로 굽고.

```ts
let image = imgElement;            // 배경 캐시에 있으면 그대로
if (!imgElement) {                 // 구멍이면 지금 렌더
  const page = await pdf.getPage(pageNumberStart + pageIndex + 1);
  image = await getViewportImage(page, RENDER_IMAGE_QUALITY, true);
}
```

이렇게 바꾸고 `isImageAllLoaded`·`loadedImages`·`pageImageLength` prop을 전부 지웠다. 막는 코드보다 그냥 되게 만드는 코드가 더 적었다. 막는 게 방어 같았는데 사실은 빚이었던 거다.

그 와중에 스피너 하나가 끝까지 말썽이었다. PDF 최초 로드 중에 슬라이더가 준비된 듯 보이다가, 어떤 순서로 들어가면 스피너가 영영 안 꺼졌다. 추적해보니 "PDF 문서 자체가 로딩 중"이라는 상태를 직접 들고 있는 값이 없었다. 스피너를 `pageImageLength !== pageImages.length`나 `!pdf` 같은 파생값 조합으로 짰는데, 이건 문서가 resolve된 *이후* 구간만 커버하고 그 전 구간을 못 담았다. effect가 `setLoading(true)` 해놓고 `if (!pdf) return`으로 빠지면 `loading`이 true로 고착되고.

그래서 `isPDFLoading` boolean 하나를 1급으로 뒀다. 초기 true, 문서 패칭이 settle되면 `.finally()`에서 false. 끝.

```ts
const [isPDFLoading, setIsPDFLoading] = useState(true);
getPdfDocument({ url })
  .then((res) => { handleLoad(res); setPdfDocument(res); })
  .catch(setError)
  .finally(() => setIsPDFLoading(false));
// 소비처: spinning={loading || isPDFLoading}, disabled={isPDFLoading}
```

"로딩 중"을 다른 값들에서 추론하면 그 값들이 아직 의미 없는 초기 구간이 꼭 구멍으로 남는다. 비동기의 시작과 끝을 직접 표현하는 boolean 하나가 그 구멍을 메운다.

이 작업 내내 임시 진행률 바를 여기저기 박았다. 청크가 정말 5개씩 도는지, 배경 로딩이 어디까지 찼는지 눈으로 봐야 믿을 수 있어서. 그러다 온디맨드+캐시우선으로 로딩 중에도 UI를 쓸 수 있게 되니까 그 바들이 더는 필요 없어져서 다 거뒀다. 보이는 신호로 검증하고, 검증 끝나면 치우는 식이었다.

## 결과

- 메모리: 전 페이지 동시 처리 → 5페이지 청크 + 즉시 `page.cleanup()`. peak가 페이지 수에 비례하던 게 청크 크기로 고정돼, 대용량 교재에서 탭이 안 죽는다. (처리 시간 변화 [확인])
- 자동 선택: 캐시 미스 페이지를 즉석 렌더로 채워서, "전부 로드까지 대기" 단계 자체가 사라졌다. 교사는 로딩 중에도 바로 누른다.
- 로딩 상태: `isPDFLoading` 단일 플래그로 모든 스피너/비활성화가 한 진실을 보게 돼, 순서 탄 스피너 고착 버그가 없어졌다.
- 크롭 정확도: 품질 `0.2 → 1` 복귀로 OpenCV 경계 검출이 다시 정확해짐.

## 회고

이 일은 거의 다 "내가 한 번에 뭘 얼마나 쥐고 있나"의 문제였다. 전 페이지를 동시에 쥐어서 메모리가 터졌고, 전부 로드되길 기다리게 해서 UX를 깎았고, 로딩 상태를 여러 파생값에 흩어 쥐어서 스피너가 고착됐다. 셋 다 "한 번에 다"를 "필요한 만큼만, 그때그때"로 바꾸는 일이었다.

습관 하나가 남았다. 메모리 줄였으면 그 자리에서 결과물부터 다시 본다. 품질 `0.2`로 내렸을 때 OpenCV가 조용히 엉뚱한 경계를 잡고 있던 걸 한참 못 봤다. 메모리는 숫자로 바로 보이는데 정확도는 결과를 들여다봐야 보인다. 이건 한 번 당하고 배웠다.
