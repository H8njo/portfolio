300페이지짜리 교재 PDF를 열면, 모든 페이지가 다 그려질 때까지 화면이 그냥 멈춰 있었다.

페이지 한 장 그리는 데 약 2초(2,132ms). 300장이면 단순히 곱해도 10분이 넘는다. 실제로 첫 조작이 가능해지기까지(TTI) 약 639,000ms, 그러니까 10분 30초가 걸렸다. 작업자는 그동안 아무것도 못 하고 멍하니 기다려야 했다.

이건 서버에서 PDF를 굽는 일이 아니라, 브라우저 쪽 PDF 뷰어였다. 교재 PDF의 각 페이지를 canvas로 래스터해서, 그 위에서 작업자가 문제 단위로 영역을 잡아 잘라내는 도구다. 근데 그 "각 페이지를 그리는" 비용이 페이지 수만큼 통째로 앞단에 쌓여 있었다.

## 문제

원래 구조는 단순했다. 페이지를 전부 미리 이미지로 렌더해서 통째로 state에 들고, 그게 다 끝나야 화면을 열어줬다. 여기서 세 가지가 한꺼번에 발목을 잡았다.

**문제 1 — 첫 화면이 페이지 수에 비례했다.** 전부 다 그려야 화면을 열어주니, 초기 렌더 시간이 페이지 수에 그대로 비례했다. 페이지가 많을수록 첫 화면이 한없이 밀렸다.

**문제 2 — 메모리를 통째로 쥐고 있었다.** 각 페이지가 `scale = 3 × dpr`로 키운 거대한 base64라, 수백 장을 동시에 들고 있으면 대용량에서 탭이 그대로 얼었다.

**문제 3 — 페이지를 넘길 때마다 다시 그렸다.** 이미 그려둔 이미지를 Step 넘길 때마다 또 재렌더하고 있었다.

## 전체를 기다리지 말고, 보이는 것부터

전체 렌더가 끝날 때까지 기다리는 구조 자체를 버렸다.

먼저 페이지를 슬라이더로 골라 보는 UI로 바꿨다. 그리고 렌더링은 앞단에서 다 끝내는 대신 백그라운드로 돌렸다. 백그라운드 렌더는 2페이지씩 청크로 잘라서 순차로 처리했는데, 한 번에 다 그리던 걸 잘게 나누니 메인 스레드가 안 막혔다. 렌더링 블락이 사라진 거다.

핵심은 우선순위였다. 1,000페이지면 1번부터 순차로 백그라운드에서 미리 그려두는데, 슬라이더로 특정 페이지가 포커싱되면 그 페이지를 먼저 그린다. 그게 다 그려지면, 이어서 나머지를 다시 백그라운드에서 계속 채운다. 사용자가 지금 보는 페이지가 항상 제일 먼저 나오고, 나머지는 뒤에서 조용히 따라오는 식이다.

로딩 인디케이터로 진행 상황은 보여주되, 로딩 중에도 조작을 막지 않았다. 아직 안 그려진 페이지를 건드리면 그 페이지만 그 자리에서 즉석으로 렌더해서 바로 보여줬다. 전부 그려질 때까지 기다릴 이유가 없으니까.

엘리베이터에 거울을 다는 이유가 있다. 층이 바뀌길 기다리는 시간은 똑같은데, 거울에 시선을 뺏기면 그 시간이 훨씬 짧게 느껴진다. 여기서 노린 것도 정확히 그거였다. 사용자의 시선을 지금 렌더된 첫 페이지에 붙잡아 두고, 나머지 수백 장은 뒤에서 조용히 그린다. 슬라이더로 다른 페이지를 포커싱하면 그 페이지가 곧바로 떠오르니, 사용자 입장에선 수백 페이지짜리 PDF가 로딩 없이 즉석에서 그려지는 것처럼 — 거의 마법처럼 느껴진다. 실제로는 뒤에서 부지런히 청크를 돌리고 있을 뿐인데.

```ts
// 2페이지씩 청크로, 백그라운드에서 순차 렌더 — 그리고 바로 해제
const makeImage = async (pdf, n) => {
  const page = await pdf.getPage(n);
  const img = await getViewportImage(page, scale ?? devicePixelRatio, true);
  page.cleanup();                 // 래스터 직후 pdf.js 리소스 해제 (메모리 누수 차단)
  return img;
};
for (const c of chunk(pageNumbers, 2)) {
  const results = await Promise.all(c.map((n) => makeImage(pdf, n)));
  setPageImages((prev) => [...prev, ...results]); // 점진 push → 진행률
}
```

```ts
// 슬라이더가 포커싱한 페이지: 캐시에 있으면 그대로, 없으면 지금 즉석 렌더
let image = imgElement;
if (!imgElement) {
  const page = await pdf.getPage(focusedPage);
  image = await getViewportImage(page, RENDER_IMAGE_QUALITY, true);
}
```

청크로 나눠 그리면서 `page.cleanup()`으로 페이지를 바로바로 버리니, peak 메모리도 전체 페이지 수가 아니라 청크 크기에 묶였다. 블락이 풀린 김에 탭이 죽던 것도 같이 잡힌 셈이다.

## 자동 영역 선택은 더 무거웠다

"전체 페이지 자동 영역 선택"은 따로 손이 갔다. 페이지 이미지를 OpenCV로 넘겨 문제 경계를 크롭하는 기능인데, `addFullPageRects`가 전체 페이지를 한 번에 `Promise.all`로 돌리면서 풀해상도 이미지랑 크롭 결과를 동시에 쌓았다. 여기도 5페이지 청크 + 즉시 해제로 peak를 묶었다.

여기서 한 번 욕심을 부렸다. 청크 재렌더 품질을 `0.2`까지 낮추면 메모리를 더 아끼겠다 싶었는데, 막상 해보니 저해상도라 OpenCV가 문제 경계를 제대로 못 잡더라. 메모리랑 정확도가 따로 노는 줄 알았는데 아니었던 거다. 결국 품질을 `1`로 도로 올렸다. 청크화만으로 이미 탭은 안 죽었으니까.

## 결과

- **첫 화면**: 300페이지 기준, 첫 조작까지 639,000ms(약 10분 30초) 걸리던 게 1,310ms(약 1초)로 떨어졌다. 약 488배다. 초기 로딩이 페이지 수에 비례하던 구조 자체가 사라진 거다(페이지당 2,132ms로 쌓이던 게 1,527ms로).
- **UX**: 로딩 중에도 인터랙션을 열어둬서, 다 그려지길 기다리는 단계가 없어졌다. 포커싱한 페이지가 항상 먼저 떠서 체감이 확 달라졌다.
- **작업 속도**: 작업자들의 이미지 처리 작업이 대용량 PDF에서 약 4배 빨라졌다. 실제로 PDF를 읽어 문제를 추출하던 작업이 몇 배로 빨라졌다는 현장 피드백을 받았다 — 숫자보다 이 말이 더 반가웠다.
- **메모리**: peak가 청크 크기로 고정돼, 대용량 교재에서도 탭이 죽지 않는다.

## 회고

결국 "한 번에 다 하려다 멈춘다"는 얘기였다. 전 페이지를 동시에 그려서 첫 화면이 밀렸고, 동시에 메모리에 쥐어서 탭이 얼었고. 둘 다 "한 번에 다"를 "지금 필요한 것부터, 나머지는 뒤에서"로 바꾸는 일이었다. 사용자가 보는 한 페이지만 빨라도 체감은 다 빨라지더라.

습관도 하나 남았다. 메모리를 줄였으면 그 자리에서 결과물부터 다시 본다. 품질을 `0.2`로 내렸을 때 OpenCV가 조용히 엉뚱한 경계를 잡고 있던 걸 한참 못 봤다. 메모리는 숫자로 바로 보이는데 정확도는 결과를 들여다봐야 보이니까. 이건 한 번 당하고 배웠다.
