# content/cases 작성 규칙

이 폴더의 `*.mdx`는 포트폴리오 케이스(사례) 글이다. 이직용이므로 **읽는 사람을 끌어당기고, 쉽게 이해되고, 정직해야** 한다. 새 케이스를 쓰거나 기존 케이스를 고칠 때 이 규칙을 따른다. 규칙은 `hoonjo`가 실제로 준 피드백에서 뽑았다.

---

## 1. 프론트매터 스키마

검증은 [lib/cases.schema.ts](../../lib/cases.schema.ts)가 빌드 타임에 한다(틀리면 빌드 실패).

```yaml
---
title: 문자열                 # 필수. 목록·상세·<title> 메타에 쓰임
summary: 문자열               # 필수. 목록 카드의 "떡밥" 한 줄
featured: false              # 홈/상단 노출 여부. 기본 false
order: 9                     # /work 정렬 (작을수록 먼저)
demo: false                  # 또는 "column-pager" | "blackhole" | "range-editor"
tags: [보안, API, 인증]       # string 배열
metrics:                     # { label, value } 배열, 보통 3개
  - label: 내 기여
    value: 맡은 기능 프론트+백 양쪽
date: 2026-06-08
---
```

- **demo**: 지원되는 세 값만 라이브 데모가 임베드된다([app/work/[slug]/page.tsx](../../app/work/%5Bslug%5D/page.tsx)). 그 외엔 `false`.
- **demoUrl**: 브라우저에 못 담는 네이티브 앱 등은 외부 링크(GitHub 등)를 넣으면 "저장소 열기" 블록이 뜬다.
- **metrics**: `value`는 짧고 임팩트 있게. 수치가 있으면 대비를 보여준다 — 예: `639,000ms → 1,310ms`, `10개 호출 → 1응답`, `59개 페이지`. 오만한 라벨 금지(3번 참고).
- 제목에 따옴표가 들어가면 YAML을 `"..."`로 감싸고 안쪽 `"`는 `\"`로 이스케이프.

## 2. 글 구조

두 가지 템플릿이 있다. **내러티브형을 선호**한다(이쪽이 잘 읽힌다).

**클래식형** — 정형 사례: `문제 → 트레이드오프 결정 → 접근 → 결과 → 회고`

**내러티브형** — 섹션 제목을 서술 문장으로. 이야기가 헤더만 읽어도 흐른다:
- 예) `쉽게 말하면, 경비실을 하나 세웠다` / `문제 — 화면은 못 믿는 자리다` / `그래서 민감한 건 전부 경비실 뒤로` / `경비실이 잡일까지 떠안다`
- 예) `화면마다 필요한 게 다 달랐다` / `설계 축: 하나의 정의, 다섯 개의 얼굴` / `조립식으로 확장한다`

공통 규칙:
- **`## 회고`는 거의 항상 마지막**에 둔다. 배운 것·다시 한다면·아쉬운 점을 정직하게.
- 첫 문단은 **한 문장 훅**으로 시작(예: "평문 비밀번호가 프론트엔드 코드 어딘가에 잠깐이라도 머무는 게 싫었다.").
- 문제가 여럿이면 **번호로 쪼갠다**: `**문제 1 — …**`, `**첫째, …**`. 한 덩어리로 뭉치지 않는다.
- 핵심 하위 항목은 **볼드 리드 문장**으로: `**1. 비밀번호는 경비실이 암호화한다.** …`

## 3. 목소리 & 톤 (가장 중요)

### 정직하고 겸손하게 — 오만한 표현 금지
- ❌ "코드오너로 맡았다", "혼자 다 짰다", "단독 기여", "내가 전부 설계했다"
- ✅ 실제 기여를 정직하게. "여럿이 함께 만들었고, 내가 맡은 기능은 프론트+백 양쪽을 다 짰다"
- breadth(넓게 관여했다)는 **넌지시** 흘린다. 자랑으로 박지 않는다. metrics 라벨도 겸손하게("공동 메인테이너", "맡은 기능 프론트+백").
- **사실은 Obsidian 노트로 대조한다** → `~/H8njo-second-brain/02 Areas/Career/<프로젝트>/`. 팀 규모·기여율·누가 뭘 했는지가 거기 있다. 기억으로 쓰지 말고 확인하고 쓴다.

### 1인칭 스토리텔링
- 겪은 사람 시점. 실감 표현 허용("제일 짜증났다", "한참 째려봤다", "겨우 열렸다").
- **반전 훅**을 잘 쓴다: "그냥 폼 몇 개겠지 했다 → 제일 오래 붙든 화면이 됐다."
- 짧은 문단과 2~3문장 문단을 섞는다. em대시(—) 자유롭게 쓴다.

### 하나의 주제에 집중
- 한 케이스는 **한 주제**. 두 개를 섞으면 독자가 혼란("Select인 줄 알았는데 아이콘이 왜?"). 여러 개를 다뤄야 하면 "여럿 중 제일 어려웠던 둘"처럼 **관계를 명시**한다.
- 제목과 본문 주제가 일치해야 한다. 주인공이 컴포넌트면 제목도 컴포넌트("보안 포털" ❌ → "테이블 컴포넌트" ✅).

## 4. 제목 규칙

- **끌리게, 그러나 쉽게.** 도메인 전문용어로 시작하지 않는다("관제사가 탐지 모델을…" ❌).
- 잘 먹히는 훅: 숫자(`CRUD 화면 59개`), 반전(`"폼 몇 개겠지" 했던 화면이…`), 통증 공감(`수백 페이지 PDF가 탭을 얼리던`), 비유(`엘리베이터 거울의 마법`).
- 대기업 테크블로그 톤을 지향: 구체적 결과 + 궁금증. 과장·낚시는 금물.
- 제목을 바꿀 땐 **본문 첫 문단·주제와 맞물리는지** 확인.

---

## 5. 비개발자도 이해되게

- 기술 용어가 나오면 **먼저 쉬운 말로 무엇을 하는지** 설명하고, 그 다음에 용어/코드를 붙인다.
- 강력한 비유를 쓴다. 실제 사용된 예:
  - BFF = "화면과 금고 사이에 세운 **경비실**"
  - 백그라운드 렌더링 = "**엘리베이터 거울** — 기다림을 숨긴다"
  - 코드젠 = "**폴더가 곧 정답지**"
  - 컬럼 정의 = "하나의 정의, **다섯 개의 얼굴**"
- 코드 블록 앞엔 한 줄 설명을 둔다. 코드는 개념적으로 단순화해도 되지만, 거짓이면 안 된다.

## 6. NDA / 회사 정보

- **회사명·내부 시스템명은 추상화**한다: "사내 디자인 시스템", "사내 보안 시스템(TOS로 부른다)", "공공기관 보안 포털".
- 내부 캡처·실제 사명·비공개 수치는 넣지 않는다.
- 상단에 NDA 안전 주석을 남긴다: `{/* NDA 안전: 회사명은 "…"으로 추상화. */}`

## 7. 이미지 / 코드

- 이미지는 마크다운 `![alt](/cases/xxx.png)`로. `alt`가 캡션(figcaption)이 된다.
  - 홀로 있는 이미지가 `<p>`로 감싸져 하이드레이션 에러 나던 문제는 [page.tsx](../../app/work/%5Bslug%5D/page.tsx)의 `rehypeUnwrapImages`가 처리한다. 신경 안 써도 됨.
- 코드 펜스는 언어를 명시(` ```ts `, ` ```js `). 하이라이팅은 rehype-pretty-code가 한다.

## 8. 검증

- **`tsc` 직접 돌리지 않는다**(프로젝트 규칙). 빌드나 브라우저로 확인한다.
- dev 서버가 떠 있으면 `curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/work/<slug>`로 200 확인.
- ⚠️ dev 서버가 도는 동안 `next build`를 돌리면 `.next`가 덮여 dev가 500 난다. 검증은 curl/브라우저로, 전체 빌드가 꼭 필요하면 dev를 멈추고 한다.
