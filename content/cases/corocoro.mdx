오래된 작업이다. 2020년, 코로나가 한창일 때 만든 React Native 앱이고 솔직히 지금 보면 손대고 싶은 코드가 많다.

내 위치 반경 2km 안의 확진자 동선이랑 지역 위험도를 지도랑 카드로 보여주는 앱이었다. 저장소명은 corocoro인데 스토어엔 두기두기라는 이름으로 올렸다. 100개쯤 되는 내 커밋 중에서 제일 오래 붙잡고 있었던 건 지도 컴포넌트 하나(`Map.js`, 580줄)다. 줌을 당기고 풀 때마다 마커가 다른 걸로 갈아끼워지는 그 화면.

## 문제

지도에 동선 마커를 다 찍으면 축소했을 때 화면이 핀으로 덮인다.

확진자 한 명이 동선을 여러 개 남기니까, 구 단위로 보면 핀이 수십 개씩 겹친다. 처음엔 당연히 클러스터링이지 싶었다. `react-native-map-clustering`을 깔고 붙여봤다. 그런데 내가 원한 건 단순히 "핀을 묶어서 숫자로 보여주는" 게 아니었다. 당겨서 가까이 보면 개별 동선 핀이 떠야 하고, 적당히 빼면 구 단위 위험도 핀, 더 빼면 그냥 점만 찍히길 바랐다. 줌 단계마다 마커의 종류 자체가 달라지는 거다. 클러스터링 라이브러리로 이걸 표현하려니 자꾸 라이브러리랑 싸우게 됐다.

## 트레이드오프 결정

라이브러리를 걷어내고 줌 레벨을 직접 계산하기로 했다.

지도가 멈출 때(`onRegionChangeComplete`) `latitudeDelta` 하나로 줌을 역산했다. 임계값 두 개를 두고 1·2·3 단계로 나눈다. 레벨 1은 개별 동선 핀에 클릭하면 상세 카드, 레벨 2는 구별 위험도 핀에 통계 카드, 레벨 3은 점만 찍고 카드는 숨긴다. 빼는 것도 손해다 — 클러스터링이 공짜로 주던 핀 묶기, 줌 임계값 튜닝을 전부 내가 떠안는 거니까. 그래도 골랐다. 라이브러리 추상화랑 싸우는 시간보다 임계값 두 개 만지는 게 빨랐다. 지금도 `Map.js` 맨 위엔 그때 걷어낸 import가 주석으로 남아있다. package.json엔 의존성도 그대로다. (지우는 걸 까먹었다.)

## 접근

지도 상태는 `useReducer`로 묶었다.

줌 역산은 임계값 두 개로 끝난다. 지도가 멈추면 `latitudeDelta` 하나만 보고 단계를 돌려준다.

```js
const LEVEL_1_LIMIT = { latitudeDelta: 0.052 };
const LEVEL_2_LIMIT = { latitudeDelta: 0.448 };

const convertZoomLevel = ({ latitudeDelta }) => {
  if (LEVEL_1_LIMIT.latitudeDelta > latitudeDelta) return 1; // 확대(동선)
  else if (LEVEL_2_LIMIT.latitudeDelta > latitudeDelta) return 2; // 중간(구)
  else return 3; // 축소(점)
};
```

당시 RN 0.62, hooks가 막 자리잡던 때라 `animating` 플래그랑 `region`, `zoomLevel`을 한 reducer에 모았다. 줌 인/아웃 버튼은 직접 만들었는데 현재 delta를 2배·절반으로 조정해서 `animateToRegion`을 부른다. 여기서 한 번 데였다. 애니메이션이 도는 중에 버튼을 또 누르면 이동이 겹치면서 지도가 튄다. 그래서 호출 전마다 `if (state.animating) return`으로 재진입을 막았다.

지도에 들어가는 데이터는 따로 떠 있는 미들웨어가 만든다. 그땐 redux-thunk를 안 쓰고 `action.type`을 가로채는 커스텀 미들웨어 세 개(위치·확진자·초기화)로 비동기를 직렬로 엮었다. 핵심은 순서였다. 위치(시/구)가 확정돼야 Firestore에서 `where('district', '==', ...)`가 성립하니까, `await dispatch(fetchLocationBegin())` 다음에 `await dispatch(fetchInfectorBegin())`을 부르는 식으로 강제했다. 스플래시는 데이터가 다 모일 때까지 떠 있다가 마지막에 닫는다. 확진자 미들웨어는 Firestore 컬렉션 대여섯 개를 조회해서 최근 2주 동선만 거르고, 각 마커에 내 위치와의 거리(Haversine)를 붙여 2km 안이면 위험으로 표시한 다음 person별 동선까지 묶어 넘긴다. 화면이 쓰는 위험도는 reducer가 아니라 selector에서 계산했다. 위험 장소 수 더하기 주변 확진자 수, 그걸 한 값(`riskValue`)으로 매핑해서 앱 전체 색을 그 값 하나로 돌렸다.

그리고 가장 마지막에 박은 게 마커 렌더 가드다. 출시 직전 커밋에 "shield codes"라고만 적어놨는데, MapView가 화면에 측정되기(`onLayout`) 전에 마커를 먼저 그리면 좌표가 어긋났다. 지도는 네이티브 레이아웃이 끝나야 좌표계가 잡히는데 그 전에 `markers.map`을 돌린 거다. `isMapReady` 플래그를 `onLayout`에서 켜고, 켜진 뒤에만 마커를 렌더하도록 막았다. 이런 게 dev에선 안 보이다가 실기기에서만 튀어서 제일 짜증난다.

## 결과

줌을 당기고 풀 때 마커가 동선→구→점으로 자연스럽게 바뀌는 지도가 나왔다. 동선 화면에서 확진자 옆 핀을 누르면 그 사람 동선 전체로, 개별 동선 핀을 누르면 그 한 지점으로 지도가 날아가서 잡힌다. 위치는 당겨서 새로고침(pull-to-refresh)하면 다시 측정하고, 구가 바뀌면 확진자 정보도 다시 조회한다. iOS·안드로이드 둘 다 스토어에 올렸다. [확인: 출시 후 지표·다운로드 수치는 기억이 흐리다]

지나고 보니 release 빌드에서만 SVG 아이콘 때문에 강제종료되던 것도 이 앱이었다(ProGuard가 클래스명을 난독화해서 죽었고 `-keep` 규칙으로 막았다). 그런 종류의, dev에선 멀쩡한데 배포본에서만 죽는 버그를 이때 처음 제대로 겪었다.

## 회고

클러스터링을 걷어낸 건 지금 봐도 맞는 결정이었다. 라이브러리가 주는 추상화가 내가 그리려는 그림이랑 어긋나면, 그거랑 싸우는 것보다 작은 규칙(임계값 두 개)을 직접 갖는 게 빠를 때가 있다.

대신 미들웨어 쪽은 자랑이 아니라 반성이다. thunk를 안 쓰고 미들웨어로 직렬화한 건 그땐 영리해 보였는데, 미들웨어가 진짜 사이드이펙트 레이어가 되면서 비즈니스 로직이 거기 쌓였다. Firestore 조립도, 거리 계산도, 순서 강제도 다 미들웨어 안이다. 지금이면 이 무게를 이렇게 한곳에 안 몰 거다. 6년 지난 코드를 다시 읽으니 잘한 결정이랑 어린 결정이 같이 보여서, 그게 좀 머쓱하다.
