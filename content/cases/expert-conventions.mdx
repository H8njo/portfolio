교육 콘텐츠 제작 스튜디오 프로젝트에 합류했다. 내 첫 커밋은 기능이 아니라 **ESLint·Prettier 설정**이었다.

입사하고 코드를 열어 보며 제일 크게 걸린 건, 이 프로젝트만이 아니라 **회사의 모든 프로젝트에 일관된 설정이 없다**는 거였다. 스튜디오의 `.eslintrc.json`은 딱 한 줄이었다.

```json
{ "extends": "next/core-web-vitals" }
```

Next.js 스캐폴드가 깔아준 기본값 그대로다. 포맷도, import 순서도, 커밋 전 검사도 없었고, 상태관리는 XState·Zustand가 섞여 있었다. 이건 단순히 "미관상 지저분하다" 수준이 아니라, 실제로 사람들의 시간을 매일 까먹고 있었다.

특히 이 스튜디오는 유독 심했다. [멀티컬럼 페이지네이션](/work/column-count-layout)이라는 난제를 풀겠다고 그동안 여러 프리랜서와 사내 개발자가 번갈아 투입됐는데, 강제되는 규칙이 없으니 사람이 바뀔 때마다 코드 스타일도, 상태를 다루는 방식도 같이 바뀌었다. 그 결과 한 파일 안에서도 손이 여러 번 바뀐 흔적이 제각각으로 남아, 새로 합류한 사람이 읽어내기가 특히 힘들었다.

## 왜 이게 비용인가

두 군데서 비용이 샜다.

**하나, 코드리뷰가 남의 시간을 잡아먹었다.** 규칙이 없으니 PR마다 포맷·import 순서·네이밍이 제각각으로 올라왔고, 리뷰어는 정작 봐야 할 로직 대신 "여기 세미콜론", "import 순서" 같은 걸 손으로 짚느라 시간을 썼다.

**둘, 같은 프로젝트 코드가 한눈에 안 들어왔다.** 파일마다 스타일이 다르고, 어떤 화면은 XState, 어떤 화면은 Zustand로 상태를 다뤄서, 옆 사람 코드를 읽을 때마다 그 사람의 방식에 새로 적응해야 했다.

그래서 프로젝트를 위아래로 다시 세우기로 했다. 아래 순서로 뼈대를 갈았고, 시작과 기반은 내가 잡았다. 큰 리팩토링은 동료 한 명과 상의하며 나눠서 진행했고, 내가 떠난 뒤의 마이그레이션은 이때 세운 표준 위에서 팀이 이어받았다.

## 1. 상태관리 — XState를 걷어내고 Zustand로 통일

가장 큰 덩어리였다. XState는 상태 전이를 엄격하게 모델링해 주지만, 이 팀에는 러닝커브가 높았다. 새로 합류한 사람이 상태 머신 정의부터 해독해야 했고, 간단한 상태 하나 바꾸는 데도 문턱이 컸다. **개발자 경험이 좋은 Zustand로 통일**하기로 하고, `@xstate/react`·`xstate` 의존성을 걷어냈다.

문제는 "그럼 복잡한 상태는 어떻게 푸느냐"였다. 시험지 생성, 문제 에디터, 변형문제 설정처럼 상태가 얽힌 화면이 많았으니까. 하나의 거대한 store로 뭉치면 XState를 Zustand로 이름만 바꾼 꼴이 된다. 그래서 **feature별로, 그 안에서 다시 관심사별로 작은 store를 쪼갰다.** 예를 들어 변형문제 기능 하나가 `configure`(설정) · `edit`(편집) · `passage-selection`(지문 선택) · `tooltip` store로 나뉘고, 문서 렌더러는 `pages` · `action` · `header` · `references` store로 나뉜다. 각 store는 자기 상태와 `set`, 그리고 초기값으로 되돌리는 `reset`만 갖는 작고 납작한 단위다.

```ts
// feature별·관심사별로 쪼갠 작은 store — 큰 머신 대신
export const useKoVariantQuestionConfigureStore = create<KoVariantConfigureStore>((set) => ({
  ...INITIAL_STATE,
  setVariantQuestionCount: (count) => set({ variantQuestionCount: count }),
  setIsIncludeExternalPassage: (value) => set({ isIncludeExternalPassage: value }),
  reset: () => set(INITIAL_STATE),
}));
```

복잡한 상태를 "정교한 상태 머신 하나"로 푸는 대신 "책임이 분명한 작은 store 여럿"으로 풀었다. 화면이 커져도 store가 커지지 않고 늘어나기만 하니, 다음 사람이 필요한 store만 열어 보면 됐다.

## 2. 코드 룰 — 기계가 강제하게

`next/core-web-vitals` 한 줄에 실제 룰을 얹었다.

- **`import/order`** — builtin·external → 내부(`@/`) → 형제·부모 → index, 그룹 사이엔 빈 줄. import만 봐도 의존 방향이 읽힌다.
- **컴포넌트는 화살표 함수로 통일** (`react/function-component-definition`).
- **안 쓰는 변수는 에러**, 단 `_` 접두사는 예외. "일부러 안 쓰는 것"과 "실수로 남은 것"을 구분.
- **Prettier + Tailwind 클래스 자동 정렬** — 포맷과 클래스 순서를 손이 아니라 도구가 맞춘다.

## 3. 컨벤션 — 이미 모두 머릿속에 있는 것에 맞춘다

**폴더**는 feature 기준으로 나누고, API 훅은 `hooks/api/` 아래로 모아 이름과 위치만 봐도 뭔지 알게 했다.

**파일 이름엔 사연이 있다.** 처음엔 전부 케밥케이스로 통일했다 — 그렇게 하자는 의견을 따랐다. 그런데 리액트 컴포넌트 파일은 대부분의 개발자가 UpperPascalCase를 당연하게 여긴다. 이미 다들 머릿속에 `ConfirmDialog.tsx`를 기대하는데 `confirm-dialog.tsx`로 강제하니 오히려 매번 낯설었다. 그래서 공통 룰에선 **컴포넌트 파일을 UpperPascalCase로 되돌렸다**. 규칙은 내가 옳다고 여기는 쪽보다, 이미 모두의 머릿속에 있는 암묵적 표준에 맞추는 게 마찰이 적었다.

## 4. 강제 게이트 & PR 규칙

룰은 "지켜주세요"로는 안 지켜진다. 사람이 매번 챙기는 건 결국 빠뜨리니까, 기계가 막게 했다.

```bash
# .husky/pre-commit  — 바뀐 파일만 lint·포맷 + 타입 체크
pnpx lint-staged
pnpm type-check
# .husky/pre-push    — 테스트 통과해야 푸시
pnpm test --run
```

여기에 **PR 템플릿**(기본·핫픽스·버그용 3종)과 **CODEOWNERS**를 더했다. PR엔 요약·완료작업·스크린샷·참고사항 칸이 강제되고, 변경 영역마다 리뷰어가 자동 지정된다. 리뷰가 "무슨 PR인지 파악하는 것"부터 시작하지 않게.

이 게이트를 처음 세울 때 한 번 데였다. 룰을 **프로젝트 전체에 한 방에 적용**해 올렸더니 그 PR이 당일 되돌려졌다. 수백 파일이 포맷 때문에 한꺼번에 바뀌어 다른 사람 작업과 충돌이 났으니까. 규칙이 틀린 게 아니라 들이는 방식이 틀린 거였다. 그래서 별도의 "리팩토링 베이스 브랜치"에 변화를 모아 팀과 맞춘 뒤 병합했다. 컨벤션은 혼자 옳은 것보다 같이 굴러가게 만드는 게 중요했다.

## 5. 안 쓰는 라이브러리·코드 걷어내기

패치워크로 쌓인 코드베이스엔 "한때 쓰던 것"이 잔뜩 남아 있었다. 라이브러리와 코드를 하나씩 검토해, 더는 안 쓰거나 다른 것으로 대체된 걸 걷어냈다.

- **의존성**: 대체된 라이브러리는 통째로 제거했다. XState(→Zustand로 통일), 옛 사내 UI 라이브러리(→공용 디자인 시스템), 그리고 Tailwind로 옮겨간 뒤엔 MUI·emotion까지. package.json에서 안 쓰는 무게를 덜어냈다.
- **데드코드**: 안 쓰는 legacy API 인프라, 화면 전환으로 버려진 옛 PDF 생성 코드, store에 남은 미사용 setter, 어디서도 안 부르는 컴포넌트를 지웠다.

"안 쓰는 게 확실한가"를 매번 확인하며 지웠는데, 룰(미사용 변수 에러)과 타입 체크가 이걸 안전하게 만들어 줬다 — 지워도 되는지 컴파일러가 먼저 답해 주니까. (MUI·emotion 제거는 Tailwind 마이그레이션과 함께 팀이 이어받아 마무리했다.)

## 6. 컴포넌트 — 완성형 라이브러리 대신 Radix 헤드리스

스타일도 제각각이었다. MUI, shadcn, 손으로 짠 Tailwind가 섞여서 같은 버튼도 화면마다 다르게 스타일됐다. 완성형 컴포넌트 라이브러리(MUI)는 디자인이 조금만 벗어나도 오버라이드가 쌓였고, shadcn처럼 통째로 복사해 온 컴포넌트는 우리 것도 남의 것도 아닌 채로 굴러다녔다.

그래서 컴포넌트를 짜는 방향을 하나로 못 박는 룰을 세웠다. **완성형에 기대지 말고, Radix 헤드리스 위에 Tailwind로 직접 짠다.** 동작과 접근성(포커스·키보드·`aria`)은 Radix 프리미티브에서 빌려오고, 겉모습은 Tailwind(+`cva`)로 우리가 온전히 통제한다. 자주 쓰는 컴포넌트는 같은 원칙으로 만든 공용 디자인 시스템(역시 Radix 기반)으로 모았다.

MUI·shadcn을 걷어내고 이 한 갈래로 통일해 가는 큰 전환은 내가 떠난 뒤 팀이 이어받았지만, "완성형을 붙이지 말고 헤드리스 위에 직접 짠다"는 방향과 룰은 이때 세운 기반 위에 있었다.

## 7. 오류 모니터링 — 안 보이던 버그를 보이게

그때까지 이 프로젝트엔 오류 모니터링이 없었다. 프로덕션에서 뭔가 터져도 사용자가 말해 주기 전엔 알 방법이 없었다. 특히 [멀티컬럼 페이지네이션](/work/column-count-layout) 리뉴얼 전에는 버그가 많았는데, 이 코드가 브라우저 측정에 의존하다 보니 환경에 따라 조용히 어긋나서 재현도 추적도 어려웠다.

그래서 Sentry를 붙여 프로덕션 에러가 잡히게 했다. React Query와 연동해 API 실패까지 함께 추적하고, 로컬/배포 환경을 갈라 개발 중 노이즈는 끄고 실제 배포에서만 수집하게 했다. 이 셋업은 동료가 나와 상의해 맡았고, 목적은 하나였다 — 안 보이던 걸 보이게.

## 결과

이 프로젝트는 여덟 명이 붙어 만드는 큰 코드베이스였고, 내 커밋 비중 자체는 크지 않다. 그래도 "규칙도 일관성도 없던 상태"를 "룰이 강제되고 상태관리가 통일된 상태"로 돌린 출발점은 내 첫 커밋이었고, 큰 리팩토링은 동료와 상의하며 함께 세웠다.

제일 확실한 증거는, 내가 그 프로젝트를 떠난 한참 뒤의 커밋들도 이 뼈대 위에서 돈다는 거다. 공통 로직을 유틸로 추출하고, 커스텀 컴포넌트를 디자인 시스템 것으로 교체하고, 네이밍을 통일하는 리팩토링이 계속됐다. 내가 세운 import 정리 규칙은 지금 팀의 작업 규칙 문서에도 그대로 들어가 있다.

그리고 이 방식은 스튜디오 하나에서 끝나지 않았다. 시작할 때 걸렸던 게 "회사의 모든 프로젝트에 일관된 설정이 없다"였으니, 해결도 한 곳으로 끝나면 안 됐다. ESLint·Prettier를 git 훅으로 강제하고, 컴포넌트는 UpperPascalCase로 두고, 상태는 feature별 Zustand store로 쪼개는 이 골격이 형제 프론트엔드 서비스들(콘텐츠 제작 도구, 디자인 시스템 등)에도 같은 모양으로 들어갔다. 서비스마다 세부 룰은 다듬어졌지만 뼈대는 같다. 그래서 서비스를 옮겨 다녀도 코드가 같은 얼굴로 읽힌다.

## 회고

규칙 없는 프로젝트를 다시 세우는 일에서 제일 크게 배운 건, 룰의 내용보다 **들이는 방식**이었다.

처음엔 "이게 맞으니까 전부 적용"이라고 밀어붙였다가 당일 되돌려졌다. 옳은 규칙도 팀의 진행을 한 번에 덮치면 거부당한다. 결국 베이스 브랜치로 충돌을 모으고, 동료와 나눠서 병합하고, 사람 대신 git 훅이 강제하게 만들고 나서야 자리를 잡았다. 컨벤션은 문서로 선언하는 게 아니라 구조로 강제될 때만 지켜진다는 것, 그리고 그 구조는 혼자보다 같이 세워야 오래간다는 것. 이 두 개가 남았다.
