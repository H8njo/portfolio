교육 콘텐츠 제작 스튜디오 프로젝트에 합류했다. 내 첫 커밋은 기능이 아니라 **ESLint·Prettier 설정**이었다.

입사하고 코드를 열어 보며 제일 크게 걸린 건, 이 프로젝트만이 아니라 **회사의 모든 프로젝트에 일관된 설정이 없다**는 거였다. 스튜디오의 `.eslintrc.json`은 딱 한 줄이었다.

```json
{ "extends": "next/core-web-vitals" }
```

Next.js 스캐폴드가 깔아준 기본값 그대로다. 포맷도, import 순서도, 커밋 전 검사도 없었다. 이건 단순히 "룰이 없으니 미관상 지저분하다" 수준의 문제가 아니었다. 실제로 사람들의 시간을 매일 까먹고 있었다.

특히 이 스튜디오는 유독 심했다. [멀티컬럼 페이지네이션](/work/column-count-layout)이라는 난제를 풀겠다고 그동안 여러 프리랜서와 사내 개발자가 번갈아 투입됐는데, 강제되는 규칙이 없으니 사람이 바뀔 때마다 코드 스타일도 같이 바뀌었다. 그 결과 한 파일 안에서도 손이 여러 번 바뀐 흔적이 제각각으로 남아, 새로 합류한 사람이 읽어내기가 특히 힘들었다.

## 문제 — 규칙 없음은 미관이 아니라 비용이다

두 군데서 비용이 샜다.

**하나, 코드리뷰가 남의 시간을 잡아먹었다.** 규칙이 없으니 PR마다 포맷·import 순서·네이밍이 제각각으로 올라왔고, 리뷰어는 정작 봐야 할 로직 대신 "여기 세미콜론", "import 순서" 같은 걸 손으로 짚느라 시간을 썼다. 리뷰어 여러 명의 시간이 매 PR마다 그렇게 새어 나갔다.

**둘, 같은 프로젝트 코드가 한눈에 안 들어왔다.** 파일마다 스타일이 달라서, 옆 사람이 짠 코드를 읽을 때 매번 그 사람의 방식에 새로 적응해야 했다. 컴포넌트가 어디는 함수 선언, 어디는 화살표. import가 어디는 정렬, 어디는 뒤죽박죽. 같이 일하는 사람들끼리도 코드가 낯설었다.

그러니까 이건 "규칙이 없어서 규칙을 넣자"가 아니었다. **매일 새어 나가는 팀의 시간과, 서로 못 읽는 코드를 멈추자**는 거였다. 그리고 사람이 매번 챙기는 규칙은 결국 빠뜨리니, 기계가 강제하게 만드는 게 목표였다.

## 접근 — 룰을 세우고, 기계에 맡긴다

`next/core-web-vitals` 한 줄에 다음을 얹었다.

- **`import/order`** — builtin·external → 내부(`@/`) → 형제·부모 → index 순서로, 그룹 사이엔 빈 줄. import만 봐도 의존 방향이 읽힌다.
- **컴포넌트는 화살표 함수로 통일** (`react/function-component-definition`). 스타일이 갈리던 걸 하나로.
- **안 쓰는 변수는 에러**, 단 `_` 접두사는 예외 (`no-unused-vars`, `^_`). "일부러 안 쓰는 것"과 "실수로 남은 것"을 구분.
- **Prettier + Tailwind 클래스 자동 정렬** — 포맷과 클래스 순서를 손이 아니라 도구가 맞춘다.

여기서 한 번 데였다. 처음엔 이 룰을 **프로젝트 전체에 한 방에 적용**해서 올렸는데, 그 PR이 **당일 되돌려졌다.** 수백 개 파일이 포맷 때문에 한꺼번에 바뀌니 다른 사람들 작업과 충돌이 나고, 리뷰도 불가능했다. 규칙 자체가 틀린 게 아니라 **들이는 방식이 틀린** 거였다. 컨벤션은 "옳으니까 받아라"로 되는 게 아니라, 굴러가는 팀에 충돌 없이 스며들게 해야 한다는 걸 그때 배웠다.

## 동료와 함께 세운 강제 게이트

그래서 방식을 바꿨다. 룰을 얹는 걸 별도의 "리팩토링 베이스 브랜치"로 만들어, 파일 이동·이름 변경 같은 큰 변화를 한곳에 모으고 팀과 맞춘 뒤 병합했다. 이 베이스 작업은 동료 한 명이 맡아서 했는데, 무엇을 강제하고 무엇을 옮길지는 나와 상의하며 정했다. 여기서 두 가지가 자리 잡았다.

**git 훅으로 규칙을 강제.** 커밋할 때(`pre-commit`) 바뀐 파일만 골라 lint·포맷을 돌리고(`lint-staged`) 타입 체크(`tsc --noEmit`)를 통과해야 하고, 푸시할 때(`pre-push`) 테스트가 통과해야 한다. 규칙을 "지켜주세요"가 아니라 "안 지키면 커밋이 안 됨"으로 바꿨다.

```bash
# .husky/pre-commit
pnpx lint-staged
pnpm type-check
# .husky/pre-push
pnpm test --run
```

**폴더 규칙.** API 훅은 `hooks/api/` 아래로 모아(`use-source.ts`, `use-handouts.ts`…) 이름과 위치만 봐도 뭔지 알게 했다.

**파일 이름 규칙엔 사연이 있다.** 처음엔 파일명을 전부 케밥케이스로 통일했다 — 그렇게 하자는 의견을 따랐다. 그런데 리액트 컴포넌트 파일은 대부분의 개발자가 UpperPascalCase를 당연하게 여긴다. 이미 다들 머릿속에 `ConfirmDialog.tsx`를 기대하는데 `confirm-dialog.tsx`로 강제하니 오히려 매번 낯설었다. 그래서 공통 룰에선 **컴포넌트 파일을 UpperPascalCase로 되돌렸다**(`confirm-dialog.tsx` → `ConfirmDialog.tsx`). 규칙은 내가 옳다고 여기는 쪽보다, 이미 모두의 머릿속에 있는 암묵적 표준에 맞추는 게 마찰이 적었다.

빅뱅으로 혼자 밀어붙였다가 되돌려진 걸, 동료와 나눠 베이스 브랜치로 정리하니 이번엔 충돌 없이 병합됐다. 컨벤션은 혼자 옳은 것보다 같이 굴러가게 만드는 게 중요했다.

## 리팩토링 — 세운 규칙 위에서

규칙을 들인다는 건 곧 기존 코드를 그 규칙에 맞추는 일이었다. 베이스 브랜치에서 프로젝트 전반의 파일이 포맷·import 순서·이름·위치 기준으로 한 번 정리됐고, 그 위에서 내 담당 영역(문제 에디터·페이징)을 다시 짰다. 흩어진 매직 넘버를 상수로 모으고(`STEP` 상수화), 하드코딩한 `#000`을 테마 토큰으로 바꾸고, 로직을 컴포넌트에서 store로 분리하고, 재사용을 위해 설정 패널 구조를 다시 짰다. 끝에는 문제 에디터 페이징을 통째로 리빌딩하는 리팩토링까지 이어졌다(이 페이징 엔진이 나중에 독립 오픈소스 패키지로 떨어져 나간다).

## 결과

이 프로젝트는 여덟 명이 붙어 만드는 큰 코드베이스였고, 내 커밋 비중 자체는 크지 않다. 그래도 "규칙이 없던 상태"를 "규칙이 강제되는 상태"로 바꾼 출발점은 내 첫 커밋이었고, 그 뒤로 규칙은 팀에 그대로 남았다.

제일 확실한 증거는, 내가 그 프로젝트를 떠난 한참 뒤의 커밋들도 이 규칙을 따라 돌아간다는 거다. 공통 로직을 유틸로 추출하고, 커스텀 컴포넌트를 디자인 시스템 것으로 교체하고, 네이밍을 통일하는 리팩토링이 계속됐다. 내가 세운 import 정리 규칙은 지금 팀의 작업 규칙 문서에도 그대로 들어가 있다.

그리고 이 방식은 스튜디오 하나에서 끝나지 않았다. 시작할 때 걸렸던 게 "회사의 모든 프로젝트에 일관된 설정이 없다"였으니, 해결도 한 곳으로 끝나면 안 됐다. ESLint·Prettier를 git 훅으로 강제하고(커밋 전 lint·타입체크, 푸시 전 게이트), 컴포넌트는 UpperPascalCase로 두는 이 규칙 골격이 형제 프론트엔드 서비스들(콘텐츠 제작 도구, 디자인 시스템 등)에도 같은 모양으로 들어갔다. 서비스마다 세부 룰은 조금씩 다듬어졌지만 "규칙은 문서가 아니라 git 훅이 강제한다"는 뼈대는 같다. 그래서 서비스를 옮겨 다녀도 코드가 같은 얼굴로 읽히고, 리뷰가 다시 스타일 싸움으로 돌아가지 않는다.

## 회고

규칙 없는 프로젝트에 규칙을 들이는 일에서 제일 크게 배운 건, 룰의 내용보다 **들이는 방식**이었다.

처음엔 "이게 맞으니까 전부 적용"이라고 밀어붙였다가 당일 되돌려졌다. 옳은 규칙도 팀의 진행을 한 번에 덮치면 거부당한다. 결국 베이스 브랜치로 충돌을 모으고, 동료와 나눠서 병합하고, 사람 대신 git 훅이 강제하게 만들고 나서야 자리를 잡았다. 컨벤션은 문서로 선언하는 게 아니라 구조로 강제될 때만 지켜진다는 것, 그리고 그 구조는 혼자보다 같이 세워야 오래간다는 것. 이 두 개가 남았다.
