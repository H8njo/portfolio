끝 단어를 먼저 클릭하고 시작을 나중에 클릭했더니, 표시가 엉뚱한 자리에 찍혔다.

교육 콘텐츠 편집기의 본문분석 에디터였다. 영어 지문 위에 밑줄·생략표시·슬래시·메모 같은 걸 얹는 기능으로, 데코레이터가 한 10종쯤 됐다. 화면은 둘이었다. 왼쪽이 지문 미리보기, 오른쪽이 속성 편집 폼. 폼에서 색을 바꾸면 미리보기가 바로 따라오고, 미리보기에서 범위를 새로 잡으면 폼이 그 범위를 받는다. 이 둘을 한 몸처럼 묶는 게 일이었다.

마침 그때 기획자가 공석이라, 기획부터 디자인 확정, 개발까지 흐름 대부분을 내가 쥐고 갔다. 부담은 부담대로 있었는데, 화면 위 동작 하나를 어떻게 둘지 직접 정할 수 있어서 솔직히 꽤 재밌는 작업이었다.

## 선택은 클릭-클릭으로

처음엔 드래그로 범위를 긋게 하려고 했다. 근데 문자 단위로 정확히 긋는 건 생각보다 손이 떨리는 동작이라, 한 글자 더 잡히거나 덜 잡히는 실수가 잦을 게 뻔했다. 그래서 시작을 한 번 클릭, 끝을 한 번 클릭으로 두 점만 찍게 했다. 대신 끝을 먼저 찍는 사람도 있으니, 거꾸로 잡힌 범위를 코드가 알아서 바로잡아야 했다. (이 가정이 나중에 버그의 절반을 만든다.)

## 트레이드오프 — 범위를 어떤 형태로 들까

범위를 어떤 형태로 저장할지부터 갈렸다. 양 끝점 `[2,5]`만 들지, 인덱스를 다 펼쳐서 `[2,3,4,5]`로 들지.

나는 펼치는 쪽을 기본 규격으로 잡았다(끝점 한 쌍이 아니라 범위를 인덱스로 펼쳐 담는, Arrow의 range 표현과 비슷한 형태). 밑줄처럼 범위 전체를 칠하는 데코레이터는 펼친 배열이 그대로 쓰이고, 무엇보다 "이 범위가 정확히 어느 글자들인가"를 한 군데 의심 없이 들고 있을 수 있어서다. 의외로 이때 정한 이 범위 표현이, 나중에 이 에디터를 넘어 다른 서비스의 범위 규격으로도 그대로 쓰였다. 당장 이 기능만 보고 정한 건데, 범위라는 게 어디든 다시 나오더라.

문제는 끝점만 쓰는 데코레이터였다. 슬래시는 시작 뒤에 한 번 긋고, 생략 메모는 시작·끝 두 점만 본다. 여기에 펼친 배열을 그냥 넘기면, 받는 쪽이 `[0]`이랑 `[1]`만 읽으니까 `[2,3,4,5]`가 소리 없이 `[2,3]`으로 잘린다. 둘 다 타입이 `number[]`라 컴파일러도 안 잡아준다. 이런 게 제일 짜증난다. 그래서 끝점만 쓰는 놈이랑 펼친 배열 쓰는 놈을 타입부터 갈라놨다. 생긴 게 같다고 같은 게 아니니까.

## 접근

데코레이터는 평문 위에 따로 떠 있는 레이어가 아니라, 지문 흐름 안에 유기적으로 끼어들어야 했다. 밑줄은 글자 아래 선으로, 슬래시는 단어 사이에, 생략은 텍스트를 감싸고, 메모는 그 자리에 마커로 — 줄바꿈이 바뀌어도 글자를 따라 같이 흐르게. 그래서 데코레이터 위치를 절대좌표가 아니라 "몇 번째 글자"라는 인덱스에 묶었다. 화면이 줄어 줄바꿈이 달라져도, 글자 인덱스만 맞으면 표시는 알아서 제자리를 찾는다.

편집 흐름은 큐로 잡았다. 미리보기에서 범위를 고르면 큐에 쌓이고, 폼에서 종류랑 속성을 정하면 `applyRangeToQueueItem`이 실제 좌표로 바꿔 미리보기에 반영한다. 선택 → 편집 → 확정이 한 방향으로만 흐르게.

범위를 잡는 동안엔 손대면 안 되는 곳이 많았다. 다른 섹션, 다른 데코레이터, 폼의 일부 입력까지 — 잠가야 할 지점이 여기저기 흩어져 있었다. 그래서 락 상태를 전역 store로 빼서 관리했다. 컴포넌트마다 prop으로 내리기엔 그 상태를 봐야 하는 곳이 너무 많았다. 안 그러면 편집 중에 옆 문장을 클릭하고, 선택 두 개가 엉키고, 좌표 계산이 통째로 무너진다. (한 번 당하고 막았다.)

거꾸로 잡힌 범위는 저장 직전 한 군데서 잡았다. 끝을 먼저 클릭하면 `startIndex > endIndex`가 되는데, 그림 그리는 쪽 코드들이 죄다 "작은 인덱스가 먼저"라고 멋대로 믿고 있었다. 그래서 끝점 저장하는 그 줄에서 그냥 정렬해버렸다.

```ts
// applyRangeToQueueItem.ts
if (d.design === 'slash' || (d.design === 'memo' && d.type === 'omit')) {
  const { startIndex: s, endIndex: e } = input.primary
  return s <= e ? [s, e] : [e, s] // 무조건 [min, max]
}
```

여기 한 줄 정렬해두니까 슬래시도 메모도 그 아래 렌더 코드들도 클릭 순서를 더는 신경 안 써도 됐다.

좌표 틀어지는 경로가 하나 더 있었다. 폼 패널이 늘었다 줄거나 창이 작아지면 단어 줄바꿈이 바뀌는데, 데코레이터가 옛 줄바꿈 기준으로 어긋났다. `ResizeObserver`로 컨테이너만 보고 있었는데, 정작 줄바꿈을 바꾸는 건 그 부모 레이아웃이더라. 컨테이너는 가만있으니 트리거가 안 됐던 거다. 부모까지 같이 보고, 혹시 몰라 `window resize` fallback도 붙였다.

마지막은 잔상. 밑줄에 dim(opacity 40%) 트랜지션을 걸었더니 Safari에서만 absolute border가 흐릿하게 남았다. 이것만 한참 헤맸다. absolute border에 opacity 트랜지션을 걸면 합성 레이어가 안 생기는 게 원인이었고, `translateZ(0)` + `will-change: opacity`로 그 요소만 GPU로 올려 깔끔하게 다시 그리게 했다.

## 결과

범위를 클릭으로 잡으면 폼이 받고, 폼에서 고치면 미리보기가 바로 따라오는 양방향 에디터가 됐다. 데코레이터 10종, 문자 단위 선택, 끝을 먼저 찍어도 안 깨지고, 패널이 늘거나 창이 줄어 줄바꿈이 바뀌어도 표시 위치가 안 틀어진다. 정밀한 재현 버그 수치를 따로 남겨두진 않았지만, 좌표가 틀어지던 경로들(거꾸로 선택·리사이즈·Safari 잔상)은 하나씩 다 막혔다.

## 회고

돌아보면 좌표 동기화는 "어디 한 군데서 정규화하느냐" 싸움이었다. 끝점 정렬도, 부모 리사이즈 관찰도, 흩어진 가정을 한 곳으로 모으는 일이었으니까.

기획자가 없어서 떠안은 프로젝트였는데, 막상 인터랙션 하나하나를 직접 정해보니 — 드래그 대신 클릭을 고른 것도, 락을 store로 뺀 것도 내 판단이었다 — 오히려 제일 손에 잘 붙는 작업이었다. 화면 위 작은 동작 하나가 데이터 구조까지 끌고 내려가는 게 재밌었다.

범위 표현을 펼치는 형태로 잡아둔 게 다른 서비스에서까지 규격으로 쓰일 줄은 몰랐다. 어디든 다시 나오는 모양은 처음에 제대로 정해두는 게 값을 한다는 걸 그때 배웠다.

아쉬운 건, 데코레이터가 10종으로 늘어나는 동안 좌표 가정이 코드 곳곳에 퍼졌던 거다. 정렬도 락도 결국 "나중에" 한 곳으로 모은 거지, 처음부터 좌표를 다루는 레이어를 하나 두고 시작했으면 거꾸로 선택 버그 같은 건 아예 안 났을 것 같다. 다시 만든다면 거기부터 잡고 들어갈 거다.
