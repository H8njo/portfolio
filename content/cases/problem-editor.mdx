단어를 오른쪽에서 왼쪽으로 드래그했더니 메모가 엉뚱한 단어 사이에 찍혔다.

교육 콘텐츠 편집기의 본문분석 에디터였다. 영어 지문 위에 밑줄, 생략표시, 슬래시, 메모 같은 걸 얹는 기능. 한 10종쯤 됐다. 사용자가 문장을 문자 단위로 드래그하면 그 범위에 표시가 그려진다. 화면은 둘이었다. 왼쪽이 지문 미리보기, 오른쪽이 속성 편집 폼. 폼에서 색 바꾸면 미리보기가 바로 따라오고, 미리보기에서 다시 그으면 폼이 그 범위를 받는다. 이 둘을 한 몸처럼 묶는 게 일이었다.

선택한 인덱스 그대로 저장하면 되겠지 했는데, 당연히 아니었다.

## 트레이드오프 결정

좌표를 어떻게 들고 있을지부터 갈렸다. 선택한 단어 인덱스를 다 펼쳐서(`[2,3,4,5]`) 저장할지, 양 끝점(`[2,5]`)만 저장할지.

밑줄은 범위 전체를 칠하니까 배열을 다 들고 있는 게 편하다. 근데 슬래시랑 생략 메모는 양 끝점밖에 안 쓴다. 슬래시는 시작 뒤에 한 번 긋고, 생략 메모는 시작·끝 두 점만 본다. 여기에 전체 배열을 그냥 넣으면, 받는 쪽이 `[0]`이랑 `[1]`만 읽으니까 `[2,3,4,5]`가 소리 없이 `[2,3]`으로 잘린다. 둘 다 타입이 `number[]`라 컴파일러도 안 잡아준다. 이런 게 제일 짜증난다.

그래서 끝점만 쓰는 놈이랑 전체 배열 쓰는 놈을 타입부터 갈라놨다. 생긴 게 같다고 같은 게 아니니까.

## 접근

편집 흐름은 큐로 잡았다. 미리보기에서 범위를 고르면 큐에 쌓이고, 폼에서 종류랑 속성을 정하면 `applyRangeToQueueItem`이 실제 좌표로 바꿔 미리보기에 반영한다. 선택 → 편집 → 확정이 한 방향으로만 흐르게.

큐가 떠 있는 동안엔 다른 섹션을 못 건드리게 락을 걸었다. 안 그러면 편집 중에 옆 문장을 클릭하고, 선택 두 개가 엉키고, 좌표 계산이 통째로 무너진다. (한 번 당하고 막았다.)

거꾸로 드래그 버그는 저장 직전 한 군데서 잡았다. 우→좌로 끌면 `startIndex > endIndex`가 되는데, 그림 그리는 쪽 코드들이 죄다 "작은 인덱스가 먼저"라고 멋대로 믿고 있었다. 그래서 끝점 저장하는 그 줄에서 그냥 정렬해버렸다.

```ts
// applyRangeToQueueItem.ts
if (d.design === 'slash' || (d.design === 'memo' && d.type === 'omit')) {
  const { startIndex: s, endIndex: e } = input.primary
  return s <= e ? [s, e] : [e, s] // 무조건 [min, max]
}
```

여기 한 줄 정렬해두니까 슬래시도 메모도 그 아래 렌더 코드들도 드래그 방향을 더는 신경 안 써도 됐다.

좌표 틀어지는 경로가 하나 더 있었다. 폼 패널이 늘었다 줄거나 창이 작아지면 단어 줄바꿈이 바뀌는데, 데코레이터가 옛 줄바꿈 기준으로 어긋났다. `ResizeObserver`로 컨테이너만 보고 있었는데, 정작 줄바꿈을 바꾸는 건 그 부모 레이아웃이더라. 컨테이너는 가만있으니 트리거가 안 됐던 거다. 부모까지 같이 보고, 혹시 몰라 `window resize` fallback도 붙였다.

마지막은 잔상. 밑줄에 dim(opacity 40%) 트랜지션을 걸었더니 Safari에서만 absolute border가 흐릿하게 남았다. 이것만 한참 헤맸다. absolute border에 opacity 트랜지션을 걸면 합성 레이어가 안 생기는 게 원인이었고, `translateZ(0)` + `will-change: opacity`로 그 요소만 GPU로 올려 깔끔하게 다시 그리게 했다.

## 결과

밑줄 그으면 폼이 받고, 폼에서 고치면 미리보기가 바로 따라오는 양방향 에디터가 됐다. 데코레이터 10종, 문자 단위 선택, 거꾸로 끌어도 안 깨지고 패널 리사이즈에도 위치가 안 틀어진다. [확인: QA 티켓/재현 버그 수치]
