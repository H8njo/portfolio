끝 단어를 먼저 클릭하고 시작을 나중에 클릭했더니, 표시가 엉뚱한 자리에 찍혔다.

교육 콘텐츠 편집기의 본문분석 에디터였다. 영어 지문 위에 밑줄이나 슬래시, 메모, 괄호, 화살표 같은 걸 얹는 기능인데, 종류가 한 10종쯤 됐다. 화면은 둘이었다. 왼쪽이 지문 미리보기, 오른쪽이 속성 편집 폼. 폼에서 색을 바꾸면 미리보기가 바로 따라오고, 미리보기에서 범위를 새로 잡으면 폼이 그 범위를 받는다. 이 둘을 한 몸처럼 묶는 게 일이었다.

마침 그때 기획자가 공석이라, 기획부터 디자인 확정, 개발까지 흐름 대부분을 내가 쥐고 갔다. 부담은 부담대로 있었는데, 화면 위 동작 하나를 어떻게 둘지 직접 정할 수 있어서 솔직히 꽤 재밌는 작업이었다.

## 선택은 클릭-클릭으로

처음엔 드래그로 범위를 긋게 하려고 했다. 근데 문자 단위로 정확히 긋는 게 생각보다 손이 떨리는 동작이라, 한 글자 더 잡히거나 덜 잡히는 실수가 잦을 게 뻔했다. 그래서 시작을 한 번 클릭, 끝을 한 번 클릭으로 두 점만 찍게 했다. 대신 끝을 먼저 찍는 사람도 있으니까, 거꾸로 잡힌 범위는 코드가 알아서 바로잡아야 했다. (이 가정이 나중에 버그의 절반을 만든다.)

## 데코레이터는 독립적으로 붙는다

데코레이터는 지문의 단어에 붙는다. 정확히는 문장을 단어로 쪼갠 배열에서 "몇 번째 단어"라는 인덱스에 묶인다. 절대좌표가 아니라 단어 인덱스라서, 화면이 줄어 줄바꿈이 달라져도 글자를 따라 같이 흐른다. 한 문장 위에 종류가 다른 데코레이터가 여러 개 겹쳐도, 각자 자기 단어 범위만 알면 서로 간섭하지 않는다.

제일 신경 쓴 건 "새 종류가 생겨도 기존 걸 안 건드리는" 구조였다. 종류마다 — 편집 폼, 클릭 방식, 렌더 — 를 각자 독립된 한 덩어리로 구현하고, 공통 껍데기(카드 셸)랑 데이터 모델↔UI를 잇는 얇은 변환 한 겹만 공유했다. 새 데코레이터를 추가할 땐 그 덩어리 하나만 끼우면 기존 편집 흐름에 알아서 합류한다. 다른 종류를 열어볼 일이 없다.

클릭 방식부터가 종류마다 다르다. 일반 데코레이터는 시작·끝 두 번 클릭으로 범위 하나, 슬래시나 생략메모는 인접한 두 단어를 한 번에 페어로, 화살표는 두 그룹을 각각 두 번씩 총 네 번 클릭으로 잡는다. 동작은 다 다른데, 결국 전부 같은 "시작/끝" state 한 곳으로 수렴하게 했다. 미리보기에서 단어를 찍든, 폼에서 숫자로 직접 넣든 똑같이.

보이는 디테일이 의외로 일이었다. 박스 같은 도형은 한 줄에 안 들어가서 줄바꿈을 만나면, 첫 줄 끝에서 한 번 닫고 다음 줄 시작에서 다시 여는 "잘린 도형"으로 그려야 자연스러웠다. 밑줄이나 하이라이트는 반대로, 단어와 단어 사이 띄어쓰기(gap)까지 끊김 없이 이어지게 했다. 단어 단위로만 칠하면 공백마다 줄이 뚝뚝 끊겨 보이니까. 데코레이터가 "단어에 붙는다"고는 해도, 정작 화면에 보이는 건 글자와 공백을 가로지르는 하나의 연속된 모양이어야 했다.

## 범위를 어떤 형태로 들까

범위를 어떤 형태로 저장할지부터 갈렸다. 양 끝점 `[2,5]`만 들지, 인덱스를 다 펼쳐서 `[2,3,4,5]`로 들지.

나는 펼치는 쪽을 기본 규격으로 잡았다(끝점 한 쌍이 아니라 범위를 인덱스로 펼쳐 담는, Arrow의 range 표현과 비슷한 형태). 밑줄처럼 범위 전체를 칠하는 데코레이터는 펼친 배열이 그대로 쓰이고, 무엇보다 "이 범위가 정확히 어느 글자들인가"를 한 군데서 의심 없이 들고 있을 수 있어서다. 의외로 이때 정한 범위 표현이, 나중에 이 에디터를 넘어 다른 서비스의 범위 규격으로도 그대로 쓰였다. 당장 이 기능만 보고 정한 건데, 범위라는 게 어디든 다시 나오더라.

문제는 끝점만 쓰는 데코레이터였다. 슬래시는 시작 뒤에 한 번 긋고, 생략 메모는 시작·끝 두 점만 본다. 여기에 펼친 배열을 그냥 넘기면, 받는 쪽이 `[0]`이랑 `[1]`만 읽으니까 `[2,3,4,5]`가 소리 없이 `[2,3]`으로 잘려버린다. 둘 다 타입이 `number[]`라 컴파일러도 안 잡아주고. 이런 게 제일 짜증난다. 그래서 끝점만 쓰는 놈이랑 펼친 배열 쓰는 놈을 타입부터 갈라놨다. 생긴 게 같다고 같은 게 아니니까.

## state와 context로 묶기

미리보기와 편집 폼을 한 몸처럼 만든 건, 작은 store 몇 개가 맞물려 돌게 한 덕이다.

"유형 추가"를 누르면 아직 서버에 안 보낸 임시(dummy) 데코레이터가 draft store로 들어간다. 폼 패널이랑 큐 처리기만 이 draft를 머지해서 보고, 본문 미리보기 캐시(React Query)는 안 건드린다 — 편집 중인 가짜가 미리보기를 더럽히면 안 되니까. 범위를 정하는 단계는 큐 store가 cursor로 하나씩 소비하고, 펼쳐진 카드 하나는 또 다른 store가 들고 있어서 미리보기가 그 단어 범위에 포커스를 준다. 작은 store 셋이 한 편집 흐름에서 손을 맞잡는 셈이다.

범위를 잡는 동안엔 손대면 안 되는 곳이 많았다. 다른 섹션, 다른 데코레이터, 폼 입력 일부까지 — 잠가야 할 지점이 여기저기 흩어져 있어서, 락 상태도 store로 뺐다. 그리고 섹션이 "완료"로 바뀌면 그 안의 카드를 전부 dim 처리하고 클릭을 막아야 했는데, 카드마다 `disabled`를 prop으로 내리는 대신 context로 섹션을 감쌌다. 그 아래 카드들은 자기가 잠겼는지 context만 보고 알아서 처신한다. 안 그러면 편집 중에 옆 문장을 클릭하게 되고, 선택 두 개가 엉키고, 좌표 계산이 통째로 무너지거든. (한 번 당하고 막았다.)

## 좌표가 틀어지던 자리들

거꾸로 잡힌 범위는 저장 직전 한 군데서 잡았다. 끝을 먼저 클릭하면 `startIndex > endIndex`가 되는데, 그림 그리는 쪽 코드들이 죄다 "작은 인덱스가 먼저"라고 멋대로 믿고 있었거든. 그래서 끝점 저장하는 그 줄에서 그냥 정렬해버렸다.

```ts
// applyRangeToQueueItem.ts
if (d.design === 'slash' || (d.design === 'memo' && d.type === 'omit')) {
  const { startIndex: s, endIndex: e } = input.primary
  return s <= e ? [s, e] : [e, s] // 무조건 [min, max]
}
```

여기 한 줄 정렬해두니까 슬래시도 메모도 그 아래 렌더 코드들도 클릭 순서를 더는 신경 안 써도 됐다.

좌표 틀어지는 경로가 하나 더 있었다. 폼 패널이 늘었다 줄거나 창이 작아지면 단어 줄바꿈이 바뀌는데, 데코레이터가 옛 줄바꿈 기준으로 어긋나더라. `ResizeObserver`로 컨테이너만 보고 있었는데, 정작 줄바꿈을 바꾸는 건 그 부모 레이아웃이었다. 컨테이너는 가만있으니 트리거가 안 됐던 거다. 그래서 부모까지 같이 보고, 혹시 몰라 `window resize` fallback도 붙였다.

마지막은 잔상이었다. 밑줄에 dim(opacity 40%) 트랜지션을 걸었더니 Safari에서만 absolute border가 흐릿하게 남더라. 이것만 한참 헤맸다. absolute border에 opacity 트랜지션을 걸면 합성 레이어가 안 생기는 게 원인이었고, `translateZ(0)` + `will-change: opacity`로 그 요소만 GPU로 올려서 깔끔하게 다시 그리게 했다.

## 결과

범위를 클릭으로 잡으면 폼이 받고, 폼에서 고치면 미리보기가 바로 따라오는 양방향 에디터가 됐다. 데코레이터는 종류별로 독립 구현돼서 새 종류를 끼우기 쉽고, 끝을 먼저 찍어도 안 깨지고, 패널이 늘거나 창이 줄어 줄바꿈이 바뀌어도 표시 위치가 안 틀어진다. 정밀한 재현 버그 수치를 따로 남겨두진 않았지만, 좌표가 틀어지던 경로들(거꾸로 선택·리사이즈·Safari 잔상)은 하나씩 다 막아뒀다.

## 회고

제일 마음에 드는 건 데코레이터를 종류별로 떼어놓은 구조다. 종류가 10종이든 20종이든 서로를 모른 채 같은 단어 위에 얹히고, 새 종류는 덩어리 하나만 끼우면 끝난다. 처음에 "종류가 계속 늘 것 같다"는 감 하나로 이렇게 잡았는데, 실제로 종류가 늘 때마다 기존 걸 안 건드려도 돼서 그 판단이 두고두고 고마웠다.

미리보기랑 폼이 한 몸처럼 움직이는 것도, 알고 보면 작은 store와 context가 각자 한 가지씩만 맡아서 맞물린 결과다. 큰 전역 상태 하나에 다 몰아넣었으면 지금쯤 못 건드리는 코드가 됐을 거다.

기획자가 없어서 떠안은 프로젝트였는데, 인터랙션 하나하나를 직접 정해보니 — 드래그 대신 클릭을 고른 것도, 락을 store로 뺀 것도 내 판단이었다 — 오히려 제일 손에 잘 붙는 작업이었다. 화면 위 작은 동작 하나가 데이터 구조까지 끌고 내려가는 게 재밌었다.

아쉬운 건, 종류는 깔끔하게 떼어놨으면서 정작 좌표 가정은 코드 곳곳에 퍼뜨렸던 거다. 정렬도 리사이즈 보정도 결국 "나중에" 한 곳으로 모은 거지, 처음부터 좌표를 다루는 레이어를 하나 두고 시작했으면 거꾸로 선택 버그 같은 건 아예 안 났을 것 같다. 다시 만든다면 거기부터 잡고 들어가지 싶다.
