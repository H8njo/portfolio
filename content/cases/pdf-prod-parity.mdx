로컬에선 PDF가 완벽하게 나오는데, 프로덕션에서 생성하면 본문 스타일이 통째로 날아갔다. 같은 코드, 다른 결과.

구조부터. 이 파이프라인에서 PDF는 클라이언트가 만들지 않는다. 클라이언트는 화면에 렌더된 DOM과 적용 중인 스타일을 전부 긁어 **스타일이 인라인된 self-contained HTML 문자열**을 만들고, 그걸 서버로 보낸다. 서버의 Puppeteer가 그 HTML을 받아 PDF로 굽는다. 내 몫은 "PDF가 깨지지 않을 HTML을 만들어 넘기는 것"이었다.

그래서 버그는 죄다 그 경계에서 터졌다. 내가 만든 HTML이 내 화면에선 멀쩡한데 서버의 헤드리스 크로미움에선 다르게 보였다. 이게 제일 골 아프다.

## 같은 코드, 다른 결과

prod/local 불일치가 제일 끈질겼다. 원래 코드는 `<link rel="stylesheet">`의 `href`를 `fetch`해서 CSS를 긁어모았다.

```ts
const css = Array.from(document.head.querySelectorAll('link[rel="stylesheet"]'))
  .map((link) => fetch(link.href).then((r) => r.text()));
```

로컬에선 이게 잘 됐다. 한참 헤맸다. 결국 알아낸 건 dev와 prod가 CSS를 **다른 방식으로 전달한다**는 거였다. dev는 스타일을 `<style>` 태그로 인젝션하는데, prod 빌드는 별도 `.css` 파일을 `<link>`로 로드한다. 그러니 prod에서 `fetch(link.href)`는 CDN이냐 CORS냐 캐시냐에 따라 실패하거나 일부만 가져왔다. 내 로컬에선 죽어도 재현이 안 됐던 이유다. (이런 거 진짜 환장한다.)

그래서 `<link>`를 긁는 발상 자체를 버렸다. 대신 **`document.styleSheets`를 직접 순회**해서 브라우저가 실제로 적용 중인 규칙을 읽기로 했다.

```ts
for (const sheet of Array.from(document.styleSheets)) {
  try {
    Array.from(sheet.cssRules).forEach((rule) => {
      allStylesText += rule.cssText + '\n';
    });
  } catch (_e) {
    // cross-origin(SecurityError)일 때만 href fetch로 강등
    if (sheet.href) { /* fetch(sheet.href, { credentials: 'same-origin' }) */ }
  }
}
```

`document.styleSheets`는 `<link>`냐 `<style>`이냐 안 따지고 실제 적용 중인 `cssRules`를 그냥 준다. 환경에 안 휘둘린다. `fetch(href)`는 cross-origin 시트용 fallback으로만 남겨뒀다. 지금은 따로 모듈로 빼서, 브라우저 확장(`chrome-extension://`) CSS는 거르고 cross-origin이나 폰트 누락이 생기면 Sentry로 경고를 쏜다.

## print에서 사라지는 것들

화면엔 분명히 보이는데 PDF로 구우면 사라지는 것들이 있었다. 2단 컬럼 사이 구분선이 대표적이었다. 그 선은 `backgroundColor: 'black'` 막대였는데, 브라우저가 잉크를 아끼려고 print 때 배경색을 기본적으로 안 찍는다. 그래서 `border`로 다시 그렸다. border는 인쇄에 살아남으니까. 김에 전역에 `print-color-adjust: exact`도 박아, "꼭 찍혀야 하는 것"은 강제로 찍게 했다.

워터마크도 비슷한 싸움이었다. 서버 PDF 렌더에서 폰트 차이로 텍스트가 깨질 수 있어서, 워터마크를 텍스트 대신 `html-to-image`의 `toPng`로 만든 **이미지(data URL)** 로 박았다. `await document.fonts.ready` 뒤에 변환하고 `pixelRatio: 1`로 줌과 무관하게 크기를 고정했다. 그런데도 Windows에서 워터마크가 통째로 안 찍히는 케이스가 남아서, 결국 그 메시지 워터마크만 하드코딩 base64 PNG로 우회했다. 크로스 OS 렌더는 끝까지 안 깔끔했다. 솔직히 그 우회는 그냥 부채다.

## 매번 손으로 확인하기 싫어서

이 파이프라인을 고칠 때마다 손으로 PDF를 굽고 눈으로 비교하는 게 너무 싫었다. 그래서 **PDF Playground**라는 개발용 회귀 도구를 따로 만들었다. 실제 교재 데이터나 임의 JSON을 넣어 렌더 결과를 바로 미리보고, 그대로 PDF 생성까지 보낼 수 있는 거다. 조판 엔진을 고칠 때마다 "내가 뭘 깨뜨렸나"를 눈으로 빨리 확인하는 안전망. 이거 없었으면 회귀를 한참 못 잡았을 거다.

## 결과

- **prod/local**: `document.styleSheets` 직접 순회로 바꾼 뒤 환경에 따른 스타일 누락이 사라졌다. 잔여 누락은 Sentry로 가시화.
- **print**: 구분선·배경 누락 0건. `border` + `print-color-adjust: exact`로 화면과 PDF가 일치.
- **회귀**: PDF Playground로 조판 변경 때마다 결과를 즉시 검증.

## 회고

돌아보면 여기서 걸린 건 거의 다 "환경 차이"였다. 내 로컬 크로미움, prod 헤드리스 크로미움, Windows, print 모드. 같은 HTML도 어디서 그리냐에 따라 다르게 나온다. 그래서 "로컬은 되는데 프로덕션만 깨짐"이 뜨면, 이제 거의 반사적으로 "뭔가 환경별로 다르게 전달되고 있구나"부터 의심한다.

습관 몇 개도 따라붙었다. 페이지 스타일은 `<link>`를 fetch하지 말고 브라우저가 실제 쓰는 걸(`document.styleSheets`) 읽는다. print에 꼭 보여야 하는 선은 `background` 말고 `border`로 그린다. 안 풀린 것도 적어둔다 — Windows 워터마크 우회는 base64 상수라, 메시지가 바뀌면 같이 손봐야 하는 부채로 남아 있다.
