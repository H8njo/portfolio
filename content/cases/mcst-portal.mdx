Strapi에서 한 모델에 라이프사이클 훅은 딱 하나만 걸린다. 두 번째를 걸면 첫 번째가 조용히 사라진다.

공공기관 내부 업무포털이었다. 모든 컬렉션의 생성·수정·삭제를 감사 로그(`collection-history`)에 남겨야 했다. 보안관제 시스템이라 "누가 언제 뭘 바꿨나"가 기능보다 중요했으니까. 그런데 어떤 모델은 감사만 하면 끝이 아니었다. SOAR 알람 모델은 생성될 때 socket.io로 담당자한테 실시간 알림을 쏴야 했고, 자산 모델은 저장 전에 값을 가공해야 했다. 감사 훅이랑 도메인 훅이 같은 모델에 동시에 살아야 했다.

## 문제

문제는 Strapi 3.x가 모델 하나에 `lifecycles` 객체를 하나만 인정한다는 거였다.

감사 훅도 `afterCreate`를 쓰고, 도메인 훅도 `afterCreate`를 쓴다. 둘 다 `module.exports`에 박으면 나중 것이 앞 것을 통째로 덮어쓴다. 에러가 나는 것도 아니다. 그냥 감사 로그가 어느 날부터 안 남는다. 컬렉션 20개 남짓에 이 충돌이 깔려 있었다. 모델마다 감사 코드를 도메인 훅 안에 손으로 복붙해 끼워넣는 방법도 있었지만, 그러면 감사 정책이 20군데로 흩어진다. 한 곳을 고치면 19곳을 똑같이 고쳐야 하고, 한 곳을 빠뜨리면 거기만 추적이 끊긴다.

그리고 이 시스템엔 쌍둥이가 하나 더 있었다. 같은 포털의 외부 공개 버전. 산하 기관 담당자가 바깥 망에서 자기 권한만큼 열람·신청하는 통합포털을 따로 배포해야 했다. 백지에서 만든 게 아니라 내부 포털 프런트를 통째로 복제해 시작했다. components, hooks, features, api, pages를 전부 카피했다. 그래서 외부 포털 작업의 본질은 새 기능 추가가 아니라, 외부에 보이면 안 되는 걸 골라 빼는 가지치기였다.

## 트레이드오프 결정

훅은 합성으로, 포털은 복제로 갔다. 둘 다 "깔끔한 정답"은 아니었다.

훅 쪽은 선택이 둘이었다. 모델마다 감사+도메인을 손으로 합쳐 쓰느냐, 아니면 합쳐주는 헬퍼를 하나 만드느냐. 손으로 가면 당장은 명시적이지만 감사 정책이 20군데로 새고, 헬퍼로 가면 감사 정책 한 줄을 한 곳에서 관리하는 대신 합성 과정의 추상화를 떠안는다. 모델이 20개면 답은 정해져 있었다. 헬퍼.

포털 쪽이 더 무거운 결정이었다. 내부와 외부를 하나의 코드로 권한 플래그만 나눠 운영하느냐, 아니면 아예 복제해 두 레포로 가느냐. 정부 보안상 외부는 공개망, 내부는 폐쇄망에 분리 배포해야 했고 일정도 빠듯했다. 그래서 복제를 택했다. 대가는 분명했다. 같은 버스가 양쪽에 살면 같은 버그도 양쪽에서 따로 고쳐야 한다. 이건 나중에 진짜로 청구서가 날아왔다.

## 접근

`combineLifecycles`는 단순하다. 라이프사이클 객체들을 배열로 받아서, 같은 이벤트 키끼리 콜백을 모으고, 그 배열을 순차 호출하는 단일 함수로 다시 묶는다.

```js
// api/helper.js
const combineLifecycles = (lifecycleArray) => {
  const dividedCycles = {};
  const lifecycle = {};

  // 1) 같은 이벤트 키끼리 콜백을 배열로 수집
  lifecycleArray.forEach((item) => {
    for (const key in item) {
      (dividedCycles[key] ||= []).push(item[key]);
    }
  });

  // 2) 각 이벤트를 "배열 전체를 순차 호출"하는 단일 함수로 재조립
  for (const key in dividedCycles) {
    lifecycle[key] = (result, params, data) => {
      dividedCycles[key].forEach((fn) => fn(result, params, data));
    };
  }
  return lifecycle;
};
```

쓰는 쪽은 이렇게 된다. 공통 감사 훅과 도메인 훅을 배열에 나란히 넣으면 끝이다.

```js
// api/ext-alarm/models/ext-alarm.js
module.exports = {
  lifecycles: combineLifecycles([
    makeCollectionHistoryHooks('ext-alarm'),  // 공통 감사
    {
      async beforeCreate(data) { /* SOAR username→id 변환 */ },
      async afterCreate(result) { /* socket.io 담당자 알림 */ },
    },
  ]),
};
```

감사 정책은 `makeCollectionHistoryHooks` 한 함수에 모여 있고, 모델은 그걸 배열 첫 칸에 끼워넣기만 한다. board, 진단 결과, 자산 같은 20개 모델에 이 패턴으로 감사+도메인을 동시에 걸었다. 감사 로그 형식을 바꾸고 싶으면 헬퍼 한 곳만 손대면 20개가 같이 따라온다.

다만 이 헬퍼엔 함정이 있다. 시그니처를 `(result, params, data)`로 통일했는데, 정작 Strapi 훅은 이벤트마다 인자 모양이 다르다. `afterCreate`는 `(result, data)`인데 `afterUpdate`는 `(result, params, data)`다. 통일된 시그니처로 흘리다 보면 위치가 한 칸씩 어긋날 수 있다. 그래서 각 콜백 안에서 인자를 쓸 때 조심해야 했다. 게다가 `forEach`로 fire-and-forget이라 Promise를 await하지 않는다. 순서가 중요한 훅엔 못 쓴다. 알면서 둔 한계다. 감사+알림처럼 서로 독립적인 부수효과엔 충분했으니까.

### 외부 포털은 빼는 게 일이었다

복제한 외부 포털에서 한 작업을 커밋 메시지로 적어보면 거의 다 "제거"다. 생성·삭제·수정·검색·정렬·엑셀 import/export 제거. 모의훈련 통계 엑셀 다운로드 숨김. 취약점 통계 탭 제거. 일정 편집 제거. 자산 IP 테이블 설정 제거. 침해관리 페이지의 내부 ID 필드 제거. 외부는 기본이 열람이고, 민감한 통계와 식별자는 바깥 망에 나가면 안 됐다.

빼는 것만으론 부족했다. 데이터 자체를 좁혀야 했다. 외부(outer) 그룹 사용자는 자기 기관·본인 범위만 봐야 하니까, 내부에서 쓰던 쿼리를 외부 맥락에 맞게 분기했다. 권한 체계의 `isOuterGroup` 판정과 묶어서 where 조건으로 데이터 범위를 깎았다. 외부 쿼리는 내부와 별도 카피라, 같은 쿼리를 외부 버전으로 또 손봐야 했다.

특히 신경 쓴 건 식별자였다. GraphQL이 내부 DB id를 응답에 그대로 실어 보내면, 공개망에선 그게 그대로 단서가 된다. 진단 통계는 ORM으로 굴리면 N+1도 나고 내부 id가 줄줄 새서, knex raw 서브쿼리로 집계를 한 방에 묶고 응답 모양까지 직접 통제했다. 침해관리 식별자도 내부 id를 노출하는 대신 업무용 번호 체계(`inciNo`)로 갈아끼웠다.

## 결과

가장 비싼 청구서는 복제 그 자체였다.

boolean 값이 들어간 엑셀을 업로드하면 깨지는 버그가 있었다. 외부 포털에서 고쳤다. 그리고 며칠 뒤 내부 포털에서 같은 버그를 또 고쳤다. 두 커밋은 사실상 같은 수정의 두 카피다. 내부 id 노출 패치도 그랬다. 외부에서 한 번, 내부에서 한 번. 이런 게 제일 짜증난다. 생긴 게 같다고 같은 게 아니니까. (한 번 당하고, 그 뒤론 한쪽을 고칠 때마다 다른 쪽을 같이 열어보는 습관이 생겼다.)

외부 포털엔 내부에 없는 기능도 하나 붙였다. 공용 PC를 가정한 유휴 자동 로그아웃이다. `react-idle-timer`로 30분 무입력을 감지해 로그아웃하고 로그인 화면으로 보낸다. 기존 인증 가드(`AuthRoute`)에 타이머를 얹고, `crossTab` 옵션으로 탭 간 활동을 공유했다. 여러 탭을 띄워도 한 탭에서 일하고 있으면 다른 탭이 멋대로 로그아웃되지 않게. 이것도 지금 보면 거칠다. `setInterval`을 cleanup 없이 등록했고, 남은 시간을 `=== 0`으로 정확 매칭했다. 폴링 타이밍에 따라 0을 스쳐 지나가면 못 잡는다. `<= 0`이 안전했을 거다.

감사 훅 쪽 효과는 조용했다. 합성으로 20개 모델에 감사가 빠짐없이 깔리고 나선, 어느 모델에서 추적이 끊기는 사고가 안 났다. 잘 된 백엔드 인프라가 늘 그렇듯, 아무 일도 안 일어나는 게 결과였다.

내부 포털 본체에선 관제 담당자가 위젯을 드래그·리사이즈해 자기 화면을 짜는 대시보드도 만들었다. `react-grid-layout`으로 지표·캘린더·도메인 목록 위젯을 배치하고, 레이아웃을 사용자별로 저장(redux-persist)했다. 목록 위젯은 내부 CRUD 엔진을 읽기 전용으로 감싼 거라 새로 만든 게 거의 없었다. 이미 있는 걸 옵션만 꺼서 재사용한 셈이다. (감사 모델 수·페이지 수는 [확인])

## 회고

이 프로젝트는 "어디에 진실을 두느냐"를 두 번 다르게 답한 기록이다.

감사 정책은 헬퍼 한 곳에 모아서 옳았다. 한 곳을 고치면 20개가 따라오고, 빠뜨릴 곳이 없다. 합성이라는 추상화의 무게는 시그니처 불일치 같은 잔가시로 돌아왔지만, 감당할 만했다.

포털을 복제로 쪼갠 건 절반만 옳았다. 분리 배포라는 제약 안에선 합리적인 선택이었지만, 같은 버그를 양쪽에서 두 번 고치는 비용을 끝까지 냈다. 다시 한다면 [확인] — 공통 코어를 패키지로 빼고 내부/외부를 그 위의 얇은 껍데기로 둘 수 있었을지, 아니면 그 시점의 일정에선 역시 복제가 답이었을지. 적어도 "한쪽을 고치면 다른 쪽을 반드시 같이 본다"는 규칙은, 도구가 아니라 습관으로라도 진작 세웠어야 했다.
