Strapi에서 한 모델에 라이프사이클 훅은 딱 하나만 걸린다. 두 번째를 걸면 첫 번째가 조용히 사라진다.

공공기관 내부 업무포털이었다. 보안관제 시스템이라 "누가 언제 뭘 바꿨나"가 기능보다 중요했고, 그래서 모든 컬렉션의 생성·수정·삭제를 감사 로그(`collection-history`)에 남겨야 했다. 그런데 어떤 모델은 감사만 하면 끝이 아니었다. SOAR 알람 모델은 생성될 때 socket.io로 담당자한테 실시간 알림을 쏴야 했고, 자산 모델은 저장 전에 값을 가공해야 했다. 감사 훅이랑 도메인 훅이 같은 모델에 동시에 살아야 했다.

## 문제

문제는 Strapi 3.x가 모델 하나에 `lifecycles` 객체를 하나만 인정한다는 거였다.

감사 훅도 `afterCreate`를 쓰고, 도메인 훅도 `afterCreate`를 쓴다. 둘 다 `module.exports`에 박으면 나중 것이 앞 것을 통째로 덮어쓴다. 에러가 나는 것도 아니다. 그냥 감사 로그가 어느 날부터 안 남는다. 컬렉션 20개 남짓에 이 충돌이 깔려 있었다. 모델마다 감사 코드를 도메인 훅 안에 손으로 복붙해 끼워넣는 방법도 있었지만, 그러면 감사 정책이 20군데로 흩어진다. 한 곳을 고치면 19곳을 똑같이 고쳐야 하고, 한 곳을 빠뜨리면 거기만 추적이 끊긴다.

## 트레이드오프 결정

선택은 둘이었다. 모델마다 감사+도메인을 손으로 합쳐 쓰느냐, 아니면 합쳐주는 헬퍼를 하나 만드느냐. 손으로 가면 당장은 명시적이지만 감사 정책이 20군데로 샌다. 헬퍼로 가면 감사 정책 한 줄을 한 곳에서 관리하는 대신 합성 과정의 추상화를 떠안는다. 모델이 20개면 답은 정해져 있었다. 헬퍼.

## 접근

`combineLifecycles`는 단순하다. 라이프사이클 객체들을 배열로 받아서, 같은 이벤트 키끼리 콜백을 모으고, 그 배열을 순차 호출하는 단일 함수로 다시 묶는다.

```js
// api/helper.js
const combineLifecycles = (lifecycleArray) => {
  const dividedCycles = {};
  const lifecycle = {};

  // 1) 같은 이벤트 키끼리 콜백을 배열로 수집
  lifecycleArray.forEach((item) => {
    for (const key in item) {
      (dividedCycles[key] ||= []).push(item[key]);
    }
  });

  // 2) 각 이벤트를 "배열 전체를 순차 호출"하는 단일 함수로 재조립
  for (const key in dividedCycles) {
    lifecycle[key] = (result, params, data) => {
      dividedCycles[key].forEach((fn) => fn(result, params, data));
    };
  }
  return lifecycle;
};
```

쓰는 쪽은 이렇게 된다. 공통 감사 훅과 도메인 훅을 배열에 나란히 넣으면 끝이다.

```js
// api/ext-alarm/models/ext-alarm.js
module.exports = {
  lifecycles: combineLifecycles([
    makeCollectionHistoryHooks('ext-alarm'),  // 공통 감사
    {
      async beforeCreate(data) { /* SOAR username→id 변환 */ },
      async afterCreate(result) { /* socket.io 담당자 알림 */ },
    },
  ]),
};
```

감사 정책은 `makeCollectionHistoryHooks` 한 함수에 모여 있고, 모델은 그걸 배열 첫 칸에 끼워넣기만 한다. board, 진단 결과, 자산 같은 20개 모델에 이 패턴으로 감사+도메인을 동시에 걸었다. 감사 로그 형식을 바꾸고 싶으면 헬퍼 한 곳만 손대면 20개가 같이 따라온다.

다만 이 헬퍼엔 함정이 있다. 시그니처를 `(result, params, data)`로 통일했는데, 정작 Strapi 훅은 이벤트마다 인자 모양이 다르다. `afterCreate`는 `(result, data)`인데 `afterUpdate`는 `(result, params, data)`다. 통일된 시그니처로 흘리다 보면 위치가 한 칸씩 어긋날 수 있다. 그래서 각 콜백 안에서 인자를 쓸 때 조심해야 했다. 게다가 `forEach`로 그냥 호출만 하고 결과를 기다리지 않는다. 순서가 중요한 훅엔 못 쓴다. 알면서 둔 한계다. 감사+알림처럼 서로 독립적인 부수효과엔 충분했으니까.

## 결과

합성으로 20개 모델에 감사가 빠짐없이 깔리고 나선, 어느 모델에서 추적이 끊기는 사고가 안 났다. 잘 된 백엔드 인프라가 늘 그렇듯, 아무 일도 안 일어나는 게 결과였다. (감사 적용 모델 수·규모는 [확인])

이 포털엔 사연이 하나 더 있다. 같은 포털의 외부 공개 버전을, 백지가 아니라 내부 포털 프런트를 통째로 복제해 시작했다. 정부 보안상 외부는 공개망, 내부는 폐쇄망에 분리 배포해야 했고 일정도 빠듯했으니까. 그래서 외부 작업의 본질은 새 기능 추가가 아니라 외부에 보이면 안 되는 걸(민감한 통계, 내부 식별자 등) 골라 빼는 가지치기였다 — 빼는 게 일이었다. 대가는 분명했다. 같은 코드가 양쪽에 살면 같은 버그도 양쪽에서 따로 고쳐야 한다. boolean 엑셀 업로드 버그를 외부에서 고치고 며칠 뒤 내부에서 또 고쳤을 때, 이 청구서가 진짜로 날아왔다.

## 회고

감사 정책을 헬퍼 한 곳에 모은 건 옳았다. 한 곳을 고치면 20개가 따라오고, 빠뜨릴 곳이 없다. 합성이라는 추상화의 무게는 시그니처 불일치 같은 잔가시로 돌아왔지만, 감당할 만했다.

포털을 복제로 쪼갠 건 절반만 옳았다. 분리 배포라는 제약 안에선 합리적이었지만, 같은 버그를 양쪽에서 두 번 고치는 비용을 끝까지 냈다. 다시 한다면 [확인] — 공통 코어를 패키지로 빼고 내부/외부를 그 위의 얇은 껍데기로 둘 수 있었을지, 아니면 그 시점의 일정에선 역시 복제가 답이었을지.
