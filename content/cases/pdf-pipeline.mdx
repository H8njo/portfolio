교육 콘텐츠 편집 도구에서 만든 문서를 A4 PDF로 뽑는 일을 맡았다. 처음엔 "브라우저에서 print 누르면 되는 거 아닌가" 싶었다. 아니었다. 사실 한참 우습게 봤다.

구조부터. PDF는 클라이언트에서 만들지 않는다. 클라이언트는 화면에 렌더된 DOM과 적용 중인 스타일을 전부 긁어서 **스타일이 인라인된 self-contained HTML 문자열**을 만들고, 그걸 서버로 보낸다. 서버의 Puppeteer가 그 HTML을 받아 PDF로 굽고. 프론트엔드엔 PDF 라이브러리가 없다. 내 몫은 "PDF가 깨지지 않을 HTML을 만들어 넘기는 것", 그리고 그 결과를 검증하는 회귀 도구였다.

문제는 죄다 그 경계에서 터졌다. 내가 만든 HTML이 내 화면에선 멀쩡한데 서버의 헤드리스 크로미움에선 다르게 보였다. 이게 제일 골 아프다.

## 문제

크게 세 종류였다.

하나, **메모리 폭증.** 대용량 PDF에서 "전체 페이지 자동 영역 선택"을 누르면 메모리가 치솟고 탭이 그대로 얼었다. 각 페이지 이미지는 `scale = 3 × dpr`짜리 거대한 base64다. 그걸 전 페이지에 대해 한 번에 들고 있었으니, 페이지 수에 비례해서 그냥 무너졌다.

둘, **prod에서만 스타일이 빠졌다.** 로컬에선 PDF가 완벽하게 나오는데 프로덕션에서 생성하면 본문 스타일이 통째로 날아갔다. 같은 코드, 다른 결과. 솔직히 원인 찾는 데 며칠 까먹었다.

셋, **print에서 사라지는 선.** 화면엔 2단 컬럼 사이 구분선이 분명히 보이는데 PDF로 구우면 그 선만 없어졌다. 그것만.

## 트레이드오프 결정

메모리부터. 원인은 금방 보였다. `addFullPageRects`가 전체 페이지 이미지를 한 번에 `Promise.all`로 돌리고 있었다. 모든 페이지의 풀해상도 이미지랑 OpenCV 크롭 결과를 **동시에** 메모리에 올린 거다. 그러니 안 죽는 게 이상했다.

```ts
// before — 전체 페이지를 한 번에
const results = await Promise.all(
  pageImages.map((img) => splitRectsByPage(img, ...)),
);
```

그래서 5페이지씩 청크로 묶어 순차 처리하고, 각 페이지는 렌더한 직후 `page.cleanup()`으로 바로바로 버렸다. peak 메모리가 전체 N이 아니라 청크 크기에 묶이게.

```ts
const imgChunk = chunk(pageImages, CHUNK_LENGTH /* 5 */);
for (const c of imgChunk) {
  const results = await makeRect(c, pageNumberStart); // 렌더 → split → page.cleanup()
  pageNumberStart += c.length;
  setFullPageRectProgress(pageNumberStart); // 원형 진행률
  resultRects.push(...results);
}
```

여기서 한 번 욕심을 부렸다. 청크 재렌더 품질을 `0.2`까지 낮추면 메모리를 더 아낄 수 있겠다 싶어서. 했더니 저해상도라 OpenCV가 문제 경계를 제대로 못 잡았다. 메모리 다이얼이랑 정확도 다이얼이 따로 노는 줄 알았는데 아니었던 거다. 게다가 줄인 다음에 결과물을 봐야 알 수 있는 종류라 더 짜증났고. 결국 품질을 `1`로 도로 올렸다. 어차피 청크화만으로 탭은 안 죽었으니까.

## 접근

prod/local 불일치가 제일 끈질겼다. 원래 코드는 `<link rel="stylesheet">`의 `href`를 `fetch`해서 CSS를 긁어모았다.

```ts
const css = Array.from(document.head.querySelectorAll('link[rel="stylesheet"]'))
  .map((link) => fetch(link.href).then((r) => r.text()));
```

로컬에선 이게 잘 됐다. 한참 헤맸다. 결국 알아낸 건 dev랑 prod가 CSS를 **다른 방식으로 전달한다**는 거였다. dev는 스타일을 `<style>` 태그로 인젝션하는데 prod 빌드는 별도 `.css` 파일을 `<link>`로 로드한다. 그러니 prod에서 `fetch(link.href)`는 CDN이냐 CORS냐 캐시냐에 따라 실패하거나 일부만 가져왔다. 내 로컬에선 죽어도 재현이 안 됐던 이유다. (이런 거 진짜 환장한다.)

그래서 `<link>`를 긁는 발상 자체를 버렸다. 대신 **`document.styleSheets`를 직접 순회**해서 브라우저가 실제로 적용 중인 규칙을 읽기로.

```ts
for (const sheet of Array.from(document.styleSheets)) {
  try {
    Array.from(sheet.cssRules).forEach((rule) => {
      allStylesText += rule.cssText + '\n';
    });
  } catch (_e) {
    // cross-origin(SecurityError)일 때만 href fetch로 강등
    if (sheet.href) { /* fetch(sheet.href, { credentials: 'same-origin' }) */ }
  }
}
```

`document.styleSheets`는 `<link>`냐 `<style>`이냐 안 따지고 실제 적용 중인 `cssRules`를 그냥 준다. 환경에 안 휘둘리고. `fetch(href)`는 cross-origin 시트용 fallback으로만 남겨뒀다. 지금은 `extract-style-sheets-for-pdf.ts`로 따로 빼서, 브라우저 확장(`chrome-extension://`) CSS는 거르고 cross-origin이나 폰트 누락이 생기면 Sentry로 경고를 쏜다.

print 구분선도 결국 같은 부류였다. 그 선은 `backgroundColor: 'black'` 막대였는데, 브라우저가 잉크 아끼려고 print 때 배경색을 기본적으로 안 찍는다. 그래서 `border`로 다시 그렸다. border는 인쇄에 살아남으니까. 김에 전역에 `print-color-adjust: exact`도 박아서 "꼭 찍혀야 하는 것"은 강제로 찍게 했다.

워터마크도 비슷한 싸움. 서버 PDF 렌더에서 폰트 차이로 텍스트가 깨질 수 있어서, 워터마크를 텍스트 말고 `html-to-image`의 `toPng`로 만든 **이미지(data URL)** 로 박았다. `await document.fonts.ready` 뒤에 변환하고 `pixelRatio: 1`로 줌이랑 무관하게 크기 고정. 그런데도 Windows에서 워터마크가 통째로 안 찍히는 케이스가 남았다. 결국 메시지 워터마크만 하드코딩 base64 PNG로 우회했다. 크로스 OS 렌더는 끝까지 안 깔끔했다. 솔직히 그 우회는 그냥 부채다.

마지막으로 자동선택 건. 배경 pre-render 끝나기 전에 자동선택을 누르면 이미지 배열에 `undefined` 구멍이 생겨서 결과가 틀어졌다. 처음엔 "전부 로드되기 전엔 못 누르게" gate를 걸었다. 제일 쉬운 방어였는데 사용자를 기다리게 했다. 다시 보니 막을 게 아니었다. 캐시에 없는 페이지만 그 시점에 즉석 렌더하면 일찍 눌러도 그냥 된다.

```ts
let image = imgElement;            // 배경 캐시에 있으면 그대로
if (!imgElement) {                 // 구멍이면 지금 렌더
  const page = await pdf.getPage(pageNumberStart + pageIndex + 1);
  image = await getViewportImage(page, RENDER_IMAGE_QUALITY, true);
}
```

gate 막던 코드랑 관련 prop들을 다 지웠다. 막는 쪽보다 그냥 되게 만드는 쪽이 코드도 더 적게 들었다. 진작 이렇게 할 걸.

## 결과

- 메모리: 전 페이지 동시 처리 → 5페이지 청크 + 즉시 해제. peak가 페이지 수에 비례하던 게 청크 크기로 고정돼, 대용량 PDF에서 탭이 죽지 않게 됐다. (처리 시간 변화 [확인])
- prod/local: `document.styleSheets` 직접 순회로 바꾼 뒤 환경에 따른 스타일 누락이 사라졌다. 잔여 누락은 Sentry로 가시화.
- print: 구분선·배경 누락 0건. `border` + `print-color-adjust: exact`로 화면과 PDF가 일치.
- 자동선택: gate 제거로 사용자가 로딩을 기다리는 단계 자체가 없어짐.

그리고 이 파이프라인을 매번 손으로 검증하기가 너무 싫어서 **PDF Playground**라는 개발용 회귀 도구를 따로 만들었다. 실제 교재 데이터나 임의 JSON을 넣어서 렌더 결과를 바로 미리보고, 그대로 PDF 생성까지 보낼 수 있는 거다. 조판 엔진 고칠 때마다 "내가 뭘 깨뜨렸나"를 눈으로 빨리 확인하는 안전망. 이거 없었으면 회귀 한참 못 잡았을 거다.

## 회고

돌아보면 여기서 걸린 것들은 거의 다 "환경 차이"였다. 내 로컬 크로미움, prod 헤드리스 크로미움, Windows, print 모드. 같은 HTML도 어디서 그리냐에 따라 다르게 나온다. 그래서 "로컬은 되는데 프로덕션만 깨짐"이 뜨면 이제 거의 반사적으로 "아, 뭔가 환경별로 다르게 전달되고 있구나" 부터 의심한다.

습관 몇 개도 그냥 따라붙었다. 페이지 스타일 모을 땐 `<link>`를 fetch하는 게 아니라 브라우저가 실제 쓰는 걸(`document.styleSheets`) 읽는 거. print에 꼭 보여야 하는 선은 `background` 말고 `border`로 그리는 거. 일괄 처리는 청크로 묶어서 peak 잡는 거. 그리고 메모리 줄였으면 그 자리에서 결과물부터 다시 보는 거 — 안 그러면 OpenCV가 조용히 엉뚱한 경계를 잡고 있다. 이건 한 번 당하고 배웠다.

안 풀린 것도 그냥 적어둔다. Windows 워터마크 우회는 base64 상수라서, 메시지 바뀌면 같이 손봐야 한다. 부채로 남아 있다.
